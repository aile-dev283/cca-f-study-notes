# 出題プロセス（deliver）— バンクから出題・採点・レポート

このファイルは「**問題を解く**」経路、または**進行中ランの再開**で読む。入力は既存バンク `banks/<name>/`。
**ここでは問題を生成しない**（生成は `generate.md`）。バンクの `quiz-questions.json` を読んで出題し、`answer-key.json` で採点するだけなので**即座に始められる**。

SKILL.md の「共通の不変条件」（特に**カンニング防止**＝出題前に `answer-key.json` を読まない・チャットに正解を出さない、**毎ターン自己定位**、**1ターン1アクション**）は**この経路でも全て守る**。

**内部キーをユーザに見せない（表示ルール）**: `drill`/`mock`、`en-q-ja-explain`、`D1`〜`D5`、`S1`〜`S6` などはファイル保存用の内部識別子。
チャットやレポートに出すときは必ず**日本語の説明的な表記**に置き換える。例: `mock`→「まとめて一括採点」、`drill`→「1問ずつ解説」、
`en-q-ja-explain`→「英語問題＋日本語解説」。「mockモードで始めます」のような内部キーの露出は禁止（ドメイン名はタグとして `D1 Agentic` のように併記するのは可）。

## ステップ D0: 進行中ランの再開（最初に行う）

`sessions/*/progress.json` を探し、`phase` が `report`/`done` 未満の進行中ランがあれば、**設定質問をせずにそのランを継続**する。
最新の `progress.json` と `answers.json` を読み、`phase` と `next_index`／`answered` から「次にとるべき**唯一の**アクション」を決め、それだけを実行する（ステップ D3／D4 のうち該当する一手）。
進行中ランが無ければ D1 へ。

## ステップ D1: 出題設定（AskUserQuestion 1回・最大4問）

まず `banks/*/manifest.json` を読み、`status: "ready"` のバンク一覧（名前・問題数・focus・ドメイン配分）を把握する。
**ready のバンクが1つも無ければ**、出題できない旨を伝え「まず『問題を作成する』でバンクを作ってください」と案内して終える。

AskUserQuestion で次の4問を確認する（**言語はここで選ぶ**のが本プロセスの肝。バンクは英語 canonical なので、好みの言語で提示できる）。

1. **どのバンクを解くか** — ready バンクを選択肢に出す（label にバンク名、description に「{count}問・{focus}・{ドメイン配分}」）。
2. **言語** — 「英語問題＋日本語解説（推奨）」（内部キー `en-q-ja-explain`）／「英語のみ」（`en`）／「日本語のみ」（`ja`）
   - 推奨は **en-q-ja-explain**。本番は英語なので問題文・選択肢は英語に慣れつつ、解説は日本語で深く理解できる。
   - この選択は**このランだけ**に適用され、バンクは書き換えない。同じバンクを次回別の言語で解いてもよい。
3. **何問解くか** — 「全問」／「部分（例 20問）」。部分のときはバンクから出題対象を選ぶ（出題順に従って先頭から N 問）。
4. **フィードバック** — 「1問ずつじっくり（理解重視・**やや時間がかかる**）」（内部キー `drill`）／「まとめて解いて一括採点（**短時間**・本番に近い）」（内部キー `mock`）
   - 選択肢提示時に**所要時間の差を簡潔に添える**。「模試」という重い表現は前面に出さない。
   - `drill` は1問ごとに即座に正誤＋解説。`mock` は4問ずつ出題し全問回収後に一括採点。

**出題順**は既定で「バンクの順番どおり」。ただしバンクの focus が weakness の場合、またはユーザが「弱点優先で」と希望した場合は、
`cca-f/reports/weakness-log.json` を参照して弱点 TS の問題を前に並べる（5問目を増やさず、既定＋希望で対応）。

回答後、`date +%Y-%m-%d-%H%M` で `sessions/<run>/` を作り（`mkdir -p`）、`progress.json` を初期化する
（`question-schema.json` の `session_progress` 定義に従う。`bank`・`question_ids`（選んだ id 列・出題順）・`language`（**上で選んだ値**）・`mode`・`total`・`answered:0`・`next_index:1`・`phase:"asking"`）。
以降、各アクションのたびに `progress.json` を更新する。

## ステップ D2: 毎ターンの自己定位（脱線防止・最重要）

このプロセスは AskUserQuestion で駆動するため、実質「**1問＝1ターン**」の長いマルチターンになる。
長い対話では初期指示が薄れ、モデルが**一度にやり過ぎたり脱線**しやすい。これを防ぐため、進捗を `progress.json` に持ち、**各ターンの最初に必ず**次を行う:

1. 最新ランの `progress.json` と `answers.json` を読む。
2. `phase` と `next_index`／`answered` から「次にとるべき**唯一の**アクション」を決める。
3. その唯一のアクションだけを実行する。**モード変更・複数問の一括提示・別作業・要約や脱線は一切しない。**
4. 進行が分からなくなったら勝手に再構成せず、現在地を一文でユーザに確認する。

## ステップ D3: 出題（増分ステートマシン・1ターン＝1アクション）

`quiz-questions.json` から `question_ids` の順に出題する。AskUserQuestion で single-select・4択で提示する。
`question` に設問（シナリオ文脈＋設問）、各選択肢は `label` を短く（例 `"A: プログラム的な前提ゲート"`）、`description` に**選択肢の全文**、`header` に短いドメインタグ（例 `"D1 Agentic"`）。

**言語ごとの提示（`progress.json` の `language` に従う。バンクは英語 canonical）**:
- `en` / `en-q-ja-explain`: 設問・選択肢は**バンクの英語をそのまま**提示する（翻訳しない）。本番が英語のため。
- `ja`: 設問・選択肢を**日本語に翻訳して提示**する（意味を変えず自然な日本語に）。技術用語・コード識別子・固有名詞は原語のまま残す。
- 翻訳はバンクを書き換えず提示時のみ行う（`quiz-questions.json` は英語のまま保つ）。

**解説の言語（drill の即時解説・mock の一括採点・D4 レポートすべて共通）**: `answer-key.json` の `rationale`/`option_notes` は英語 canonical。これを次のように出す:
- `en`: 解説は**英語のまま**。
- `en-q-ja-explain` / `ja`: 解説は**日本語**にする（英語の rationale を日本語へ訳して説明する）。

回答を受け取るたび即 `answers.json` に追記し、`answered`／`next_index`／`phase` を更新する。**全問回答を促す**（無回答＝誤答）。

- **演習モード（drill）の1問サイクル**:
  1. `answer-key.json` を読み、`next_index` の**1問だけ**採点する。
  2. その1問の正誤＋解説（正解の理由＋他3つがなぜアンチパターンか）を出す。
  3. `answers.json`／`progress.json` を更新する。
  4. 残問があれば、**ただちに次の1問だけ**を AskUserQuestion で提示してターンを終える（**複数問を並べない**）。
  5. 全問終了でステップ D4 へ。
  - **絶対規則**: 演習モードのターンは「**次の1問の AskUserQuestion**」または「**ステップ D4 のレポート**」以外で終わってはならない。残問があるのに解説だけ出して止まる・他作業を始める・残りを一括提示するのは**すべて禁止**。
- **模試モード（mock）**: **4問ずつ** AskUserQuestion で提示（`questions[]` に最大4問）。全問回収するまで解説を出さず、各バッチ回収のたび `progress.json` を更新し、全回収でステップ D4 の一括採点へ進む。

## ステップ D4: 採点・分析・レポート

全回答を回収したら、バンクの `answer-key.json` を読んで突合採点し、`report-template.md` に従ってレポートを作る。
**各問に設問・選択肢の全文を必ず含める**（後から振り返れるように）。設問・選択肢の言語は出題時の提示言語に合わせる
（`en`/`en-q-ja-explain` は英語のまま、`ja` は日本語訳）。ただし本番慣れのため、`ja` でも**英語原文を併記**してよい。解説の言語は上記「解説の言語」ルールに従う。

- **素点**（正答数/総数, %）
- **ドメイン別正答率**（ヒートマップ）＋**サブドメイン（タスクステートメント）別**の正誤
- **スケールド推定**: 素点をドメイン重みで加重した目安。**必ず「目安。720＝72%ではない」と明記**。素点 85〜90% を安全圏の目安として併記。
- **各問の解説**: ①英文の設問全文 ②英文の選択肢 A〜D 全文 ③あなたの回答と正解 ④正解の理由 ⑤他3つがなぜアンチパターンか ⑥出典。
- **弱点ランキング**: 正答率の低いドメイン/サブドメイン順。
- **復習先**: 弱点ごとに `blueprint.yaml` の `review_links`（リポジトリ内パス＋該当 Skilljar フェーズ）を提示。
- **次回の推奨（追加生成への橋渡し）**: 弱点ドメイン／サブドメインを示し、**次に作るべきバンクを推奨だけ提示**する（例「次は `/cca-f-trainer` の『問題を作成する』で "弱点ターゲット 10問" または "D2 集中 10問" バンクを作るとよい」）。
  ここで**自動生成はしない**（出題パスに重い生成を混ぜず高速を保つため）。ユーザが改めて作成プロセスを起動する。

レポートは `cca-f/reports/<YYYY-MM-DD>-session.md` に保存（同日複数回は `-2`,`-3` のサフィックス）。要点をチャットに提示する。
`progress.json` の `phase` を `done` にする。最後に**弱点ログを更新**する（次項）。

## ステップ D5: セッション横断の弱点トラッキング（間隔反復）

`cca-f/reports/weakness-log.json` に各問の結果を**追記蓄積**する。これにより弱点ターゲットの作成・弱点優先の出題順が回を重ねるほど賢くなる。

- 1問ごとに次を記録: `date`・`session_dir`・`question_id`・`domain`・`task_statement`・`scenario`・`difficulty`・`correct`(bool)・`user_choice`。
- ファイルが無ければ新規作成。`question-schema.json` の `weakness_log` 定義に従う。
- **間隔反復スコア**（弱点優先の出題順・弱点ターゲット生成が利用）: 各 task_statement について
  「直近で間違えた」「誤答回数が多い」「正答率が低い」「長く出題していない」ものを優先スコア化する。
  具体例: `priority = 2*(誤答率) + 1*(最後に誤答してからの経過セッション数の少なさ) + 0.5*(未出題期間)`。直近で連続正解した TS は優先度を下げる。
- weakness-log とレポートは個人の学習記録のため `cca-f/reports/.gitignore` でコミット対象外（ディレクトリは `.gitkeep` で保持）。
