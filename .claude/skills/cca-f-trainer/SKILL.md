# CCA-F Trainer

CCA-F 試験対策の問題演習トレーナー。**自作問題の最大リスクは「正解キーが間違っていること」**なので、
このスキルは「コーパスに根拠を持つ問題だけを、生成時のセルフチェックを通してから出題する」ことを最優先に設計されている。

## このスキルが守る公式仕様（崩してはいけない前提）

これらは公式 Exam Guide（`cca-f/CCA-F_Certification_Exam_Guide.md`）の確定事項。問題生成・採点はこれに従う。

| 項目 | 確定事項 | 実装 |
|---|---|---|
| 回答形式 | 各問は**正解1つ＋ディストラクタ3つの単一正解（single-best-answer）** | デフォルトは単一選択。複数選択は「本番には無い練習バリアント」と明示した上で任意 |
| 出題 | シナリオベース。6本中4本がランダム選出され、**全問がシナリオに紐づく** | 6シナリオ原型を内蔵し、問題を必ずシナリオに包む |
| ディストラクタ | 明らかな誤答ではなく本気で迷う設計。**2択が技術的に成立するが片方が設計的に優れている** | 生成時に「もっともらしいアンチパターン」を強制 |
| スコア | 100〜1000のスケールドスコア。**720は素点72%ではない**（難問が重み付け） | 素点とドメイン別正答率を出し、スケールド推定は「目安」と明記。**素点85〜90%を安全圏に** |
| 無回答 | 無回答は誤答扱い | 全問回答を促す |

**ドメイン重み:** D1 Agentic 27% / D3 Claude Code 20% / D4 Prompt 20% / D2 MCP 18% / D5 Context 15%。

## カンニング防止の鉄則（最重要・最初に読む）

このスキルの会話とファイルはユーザに見える。**出題前に正解・解説をユーザが読めてはいけない。**そのため:

1. **2ファイル分割**: 出題前に書くのは `quiz-questions.json`（**問題文と選択肢のみ。正解・解説・出典・is_correct を一切含めない**）。
   正解キー・解説・出典・各選択肢の正誤は `answer-key.json`（**平文・別ファイル**）に分離して保存する。
   分離の目的は「**出題時に読むファイルに答えを同居させない**」こと。
2. **答えは出題前のチャットに出さない**: `answer-key.json` は採点時にだけ読む。生成中の推論でも「正解は〜」と断定的に書かない。
3. **正解位置をランダム化**: 正解を常に A に置かない。各問で A〜D のどれかにばらつかせ、`quiz-questions.json` 全体の正解分布が概ね均等か（特定キーに偏っていないか）を生成後に自己チェックする。
4. 出題前のチャットに正解・根拠・「正解は〜」といった文言を出さない。生成中の推論でも答えを断定的に書かない。

## ターン開始時の自己定位（脱線防止・最重要）

このスキルは AskUserQuestion で駆動するため、実質「**1問＝1ターン**」の長いマルチターンのループになる。
長い対話では初期指示が要約・圧縮で薄れ、モデルが**一度にやり過ぎたり脱線したり**しやすい（参照:
`cca-f/_originals/official-blog/07-effective-harnesses-and-agent-skills.md`／`04-context-engineering-and-management.md`）。
これを防ぐため、進捗を**外部ファイル `progress.json` に持ち、毎ターン読み直して現在地を再固定**する。

**セッション進行中（session ディレクトリが存在し、まだステップ4のレポートを書いていない）あいだ、各ターンの最初に必ず次を行う:**

1. 最新の session ディレクトリの `progress.json` と `answers.json` を読む。
2. `phase` と `next_index`／`answered` から「次にとるべき**唯一の**アクション」を決める。
3. その唯一のアクションだけを実行する。**モード変更・複数問の一括提示・リポジトリの別作業・要約や脱線は一切しない。**
4. 進行が分からなくなったら勝手に再構成せず、現在地を一文でユーザに確認する。

## 実行フロー（4ステップ）

AskUserQuestion は1回の呼び出しで終ターンになる。そのため「提示 → 次ターンで回収」の対話になる。

### ステップ1: 設定（AskUserQuestion 1回・最大4問）

`date +%F`／`date +%Y-%m-%d-%H%M` で日時を取得し、`sessions/<YYYY-MM-DD-HHMM>/` を作る（`mkdir -p`）。
AskUserQuestion を1回呼び、次の4問で設定を確認する。回答が揃ったら、その session dir に
`progress.json` を初期化する（`question-schema.json` の `session_progress` 定義に従う。
`phase:"setup"`・`mode`・`total`・`answered:0`・`next_index:1` 等）。以降、各アクションのたびに更新する。

1. **言語** — 「英語問題＋日本語解説（推奨）」/「英語のみ」/「日本語のみ」
   - 推奨デフォルトは**英語問題＋日本語解説**。本番は英語なので問題文・選択肢は英語に慣れつつ、解説は日本語で深く理解できる。
2. **問題数** — 5 / 10 / 20 / 60（フル模試）
3. **出題内容モード** — 「ドメイン横断ランダム」/「特定ドメイン集中」/「弱点復習（前回までの成績参照）」
   - 「弱点復習」→ `cca-f/reports/weakness-log.json` を読み、間隔反復で出題を弱点へ寄せる（後述）。無ければ横断にフォールバック。
   - 「特定ドメイン集中」→ 次ターンでどのドメインかを追加で1問聞く。
4. **フィードバックモード** — 「1問ずつじっくり（理解重視・**やや時間がかかる**）」/「まとめて解いて一括採点（**短時間**・本番に近い）」
   - 選択肢提示時に**所要時間の差を簡潔に添える**こと（例: 前者の description に「1問ごとに解説を読むので時間がかかる」、後者に「まとめて解くので短時間で終わる」）。「模試」という重い表現は前面に出さない。
   - 「1問ずつじっくり」（内部キー `drill`）は公式 practice 同様、**1問ごとに即座に正誤＋解説**を出す。
   - 「まとめて解いて一括採点」（内部キー `mock`）は**4問ずつ**出題し、全問回収後に一括採点（本番シミュレーション）。任意でユーザが希望すれば所要時間も記録する。

問題数に応じてドメイン配分を決める。横断モードは**公式重みに比例**（例: 20問なら D1×5, D3×4, D4×4, D2×4, D5×3。端数は D1 に寄せる）。
**各ドメイン内の task_statement は固定せず、出題履歴（後述）を見て被りの少ない TS を巡回選択する**（毎回同じ「定番」TS に収束させない）。
難易度は **L3（トレードオフ判断）を主軸**に、L2（応用）を従にする。

### ステップ2: 生成（コーパスにグラウンディング）

`blueprint.yaml`・`scenarios.yaml`・`antipattern-playbook.md`・`question-schema.json` を読み込み、問題を生成する。

**まず出題履歴を読み込む（重複防止・最重要）**: 生成前に `cca-f/reports/weakness-log.json`（あれば）と
`sessions/*/quiz-questions.json`（直近数セッション）を読み、**既出の `(task_statement × scenario)` と stem の要旨**を一覧化する。
これを「既出セット」として、今回の問題が過去と**被らないように生成する**。履歴が無ければ全 task_statement から均等に選ぶ。

各問は**必ず次を満たすこと**。満たせない問題は破棄して作り直す。

- **既出と被らせない（多様化）**: 同じ `(task_statement × scenario)` の組を直近セッションから再利用しない。
  やむなく同じ task_statement を使う場合でも、**シナリオ・具体的な制約や数値・どのアンチパターンを罠にするか・問う角度を必ず変え**、
  過去とほぼ同一の stem を絶対に再生産しない。横断モードでは blueprint の task_statement プール（約30）を**巡回**し、
  「最も出題回数が少ない／長く出していない」TS を優先採用してドメイン内の偏りを解消する（公式ドメイン重みは維持）。
- **シナリオに包む**: `scenarios.yaml` の原型を選び、その文脈で問う。trivia（用語暗記）禁止。必ず**シナリオ＋制約＋設計判断**の粒度。
  各 task_statement は複数のシナリオ（S1〜S6）で枠付けできる。毎回同じ TS↔シナリオの結びつきに固定しない。
- **根拠を紐付ける**: 正解には必ずコーパス内の出典（ファイルパス＋タスクステートメント番号や見出し）を `citations` に記録。
  根拠を示せない問題は作らない。第一の根拠は Exam Guide のタスクステートメント、補強は公式ブログ・docs。
- **ディストラクタを「もっともらしいアンチパターン」にする**: `antipattern-playbook.md` から素材を引く。
  少なくとも1つは「技術的には成立するが設計的に劣る」選択肢にする。頻出トラップ4種（単一スーパーエージェント／文脈自動継承の誤解／
  CI で `-p` 無し対話実行／高リスクで人間監督を減らす）は「魅力的だが誤り」のディストラクタとして特に有効。
- **正解位置をランダム化**: 上の「鉄則3」に従い、正解を A〜D にばらつかせる。
- **タグ付け**: `domain`（D1〜D5）・`task_statement`（例 "1.4"）・`scenario`（S1〜S6）・`difficulty`（L2/L3）。

**生成後のセルフチェック（各問・必須）**: 自作問題の最大リスクは正解キー誤り。別エージェントは使わず、**この場で自己点検**する。
各問について、**自分が書いた `rationale` をいったん伏せ**、コーパス（第一根拠は Exam Guide のタスクステートメント、補強は
`cca-f/official-blog/`・`docs/`・`cca-f/skilljar-courses/`）から最良の選択肢を**ゼロから再導出**し、`correct` と一致するか確認する。
あわせて ①一意に最良か ②ディストラクタがもっともらしい誤りか ③出典が正解を支持するか ④trivia でないか を点検し、
**1つでも満たさなければ修正または作り直す**。これは同一コンテキスト内の点検なので別インスタンスより検出力は劣るが、サブエージェント起動が無く高速。

生成物を**2ファイルに分けて保存**する（鉄則1）:

- `sessions/<dir>/quiz-questions.json` — `question-schema.json` の **public 部分のみ**（session メタ＋各問の id/domain/task_statement/scenario/difficulty/stem/options[key,text,label]）。
  **`is_correct`／`correct`／`rationale`／`citations` を絶対に書かない。**答えは必ず `answer-key.json` のみに置く
  （過去に `quiz.json` へ正解を平文埋め込みしたカンニング防止違反の再発を防ぐ）。
- `sessions/<dir>/answer-key.json` — 各問の `correct`／`rationale`／`citations`／各 option の `is_correct`/`tempting`/`why_wrong` を入れた平文 JSON。
  Write で保存してよい。**出題前のチャットには中身を出さない**（採点時にだけ読む）。`quiz-questions.json` には絶対に混ぜない。

### ステップ3: 出題（増分ステートマシン・1ターン＝1アクション）

検証を通った問題を AskUserQuestion で single-select・4択で提示する。`question` に設問（シナリオ文脈＋設問）、
各選択肢は `label` を短く（例 `"A: プログラム的な前提ゲート"`）、`description` に**選択肢の全文**、`header` に短いドメインタグ（例 `"D1 Agentic"`）。

進捗は `progress.json` に持ち、**各アクションのたびに更新**する（agentic memory／progress-file パターン）。
回答を受け取るたび即 `sessions/<dir>/answers.json` に追記し、`answered`／`next_index`／`phase` を更新する。
**全問回答を促す**（無回答＝誤答）。**1ターンでは「次の唯一のアクション」だけを行う。**

- **演習モード（drill）の1問サイクル**:
  1. `answer-key.json` を読み、`next_index` の**1問だけ**採点する。
  2. その1問の正誤＋解説（正解の理由＋他3つがなぜアンチパターンか）を出す。
  3. `answers.json`／`progress.json` を更新する。
  4. 残問があれば、**ただちに次の1問だけ**を AskUserQuestion で提示してターンを終える（**複数問を並べない**）。
  5. 全問終了でステップ4へ。
  - **絶対規則**: 演習モードのターンは「**次の1問の AskUserQuestion**」または「**ステップ4のレポート**」以外で終わってはならない。
    残問があるのに解説だけ出して止まる・他作業を始める・残りを一括提示するのは**すべて禁止**。
- **模試モード（mock）**: **4問ずつ** AskUserQuestion で提示（`questions[]` に最大4問）。全問回収するまで解説を出さず、
  各バッチ回収のたび `progress.json` を更新し、全回収でステップ4の一括採点へ進む。

### ステップ4: 採点・分析・レポート

全回答を回収したら `answer-key.json` を読んで突合採点し、`report-template.md` に従ってレポートを作る。**各問に英文の設問・選択肢の全文を必ず含める**（後から振り返れるように）。

- **素点**（正答数/総数, %）
- **ドメイン別正答率**（ヒートマップ）＋**サブドメイン（タスクステートメント）別**の正誤
- **スケールド推定**: 素点をドメイン重みで加重した目安。**必ず「目安。720＝72%ではない」と明記**。素点 85〜90% を安全圏の目安として併記。
- **各問の解説**: ①英文の設問全文 ②英文の選択肢 A〜D 全文 ③あなたの回答と正解 ④正解の理由 ⑤他3つがなぜアンチパターンか ⑥出典。
- **弱点ランキング**: 正答率の低いドメイン/サブドメイン順。
- **復習先**: 弱点ごとに `blueprint.yaml` の `review_links`（リポジトリ内パス＋該当 Skilljar フェーズ）を提示。
- **次回推奨モード**: 例「次回は D2 集中10問」。

レポートは `cca-f/reports/<YYYY-MM-DD>-session.md` に保存（同日複数回は `-2`,`-3` のサフィックス）。要点をチャットに提示する。
最後に**弱点ログを更新**する（次項）。

### セッション横断の弱点トラッキング（間隔反復）

`cca-f/reports/weakness-log.json` に各問の結果を**追記蓄積**する。これにより「弱点復習」モードが回を重ねるほど賢くなる。

- 1問ごとに次を記録: `date`・`task_statement`・`domain`・`scenario`・`difficulty`・`correct`(bool)・`question_id`。
- ファイルが無ければ新規作成。`question-schema.json` の `weakness_log` 定義に従う。
- **弱点復習モードの出題選定**（間隔反復）: 各タスクステートメントについて
  「直近で間違えた」「誤答回数が多い」「正答率が低い」「長く出題していない」ものを優先スコア化し、上位の TS から出題する。
  直近セッションで連続正解した TS は優先度を下げる。具体スコア例: `priority = 2*(誤答率) + 1*(最後に誤答してからの経過セッション数の少なさ) + 0.5*(未出題期間)`。
- weakness-log とレポートは個人の学習記録のため `cca-f/reports/.gitignore` でコミット対象外（ディレクトリは `.gitkeep` で保持）。

## 参照ファイル

- `blueprint.yaml` — 5ドメイン×タスクステートメントの網羅チェックリスト＋ドメイン別の復習先リンク。**生成前に必読**。
- `scenarios.yaml` — 6シナリオ原型。問題を包む文脈として必ず使う。
- `antipattern-playbook.md` — ディストラクタ素材（頻出トラップ＋各タスクのアンチパターン）。**生成時に必読**。
- `question-schema.json` — 問題JSONのスキーマ（public 問題ファイル／answer-key／weakness-log の各定義）。
- `report-template.md` — レポート雛形。ステップ4で使う。
- `sessions/` — セッション作業領域（gitignore 済み）。各 session dir に `progress.json`（現在地）・`quiz-questions.json`・`answer-key.json`・`answers.json` を置く。

## 品質に関する不変条件（必ず守る）

1. **根拠なき問題は出さない** — `citations` が空、またはセルフチェックで出典不支持と判定した問題は破棄。
2. **trivia を出さない** — 暗記でなく設計判断を問う。シナリオに包めない問題は破棄。
3. **正解は生成時にセルフチェックする** — 自分の生成理由で正解を正当化しない。`rationale` を伏せてコーパスから正解を再導出し、一致を確認する。
4. **正解を事前に漏らさない** — 出題前に `correct`/`rationale` を `quiz-questions.json` にもチャットにも出さない。答えは別ファイル `answer-key.json` に分離する。
5. **正解位置をランダム化** — 正解を A 固定にしない。生成後に分布を自己チェック。
6. **スケールド推定は目安** — 720=72% ではないことを毎回明記する。
7. **進行中は脱線しない** — セッション進行中の各ターンは `progress.json` の次アクションだけを実行する。
   モード変更・演習モードでの一括提示・無関係な作業をしない。
8. **状態は外部ファイルに持つ** — 進捗は記憶に頼らず `progress.json` に書き、ターン開始時に必ず読み直す。
9. **毎回同じ問題を出さない** — 生成前に出題履歴（`weakness-log.json`・直近 `sessions/*/quiz-questions.json`）を読み、
   既出の `(task_statement × scenario)` や同一 stem を再利用しない。task_statement プールを巡回し、シナリオ・角度・制約を変えて多様化する。
