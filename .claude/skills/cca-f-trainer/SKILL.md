# CCA-F Trainer

CCA-F 試験対策の問題演習トレーナー。**自作問題の最大リスクは「正解キーが間違っていること」**なので、
このスキルは「コーパスに根拠を持つ問題だけを、生成時のセルフチェックを通してから出題する」ことを最優先に設計されている。

## このスキルは「作成」と「出題」の2プロセスに分かれている

問題生成は重い（コーパス読込＋全問セルフチェック）。これを毎回の出題のたびに走らせないため、処理を2つに分けた。
**呼び出すスキルは1つ（`cca-f-trainer`）だけ**。下のディスパッチが状況に応じて適切なプロセスへ振り分けるので、ユーザはどちらを使うか意識しなくてよい。

- **作成プロセス（generate）** — コーパスから問題を生成し、**永続バンク** `banks/<name>/`（コミット対象）に保存する。重いが、一度作れば何度でも再利用できる。手順は `procedures/generate.md`。
- **出題プロセス（deliver）** — 既存バンクを読んで出題・採点し、弱点レポートを出す。生成しないので**即座に始まる**。手順は `procedures/deliver.md`。

典型フロー: ①作成で60問の模試バンクを一度作る → ②出題でそれを解く → ③レポートの推奨に従い、弱点ターゲットの追加バンクを作成 → ②出題…。

## 起動時のディスパッチ（最初に必ず行う）

1. **進行中ランの自動再開を優先**: `sessions/*/progress.json` を確認し、`phase` が `report`/`done` 未満の進行中ランがあれば、
   **設定質問をせず** `procedures/deliver.md` を読み、その「ステップ D0/D2」に従って**そのランを継続**する。これは脱線防止のため最優先。
2. 進行中ランが無ければ、`banks/*/manifest.json` の有無を確認しつつ、AskUserQuestion を1回呼んで**何をするか**を聞く:
   - **「問題を解く（作成済みバンクからすぐ始まる）」** → `procedures/deliver.md` を読んで実行。
   - **「問題を作成する（バンクを作る・時間がかかる）」** → `procedures/generate.md` を読んで実行。
   - `status:"ready"` のバンクが1つも無い場合は、「まず作成が必要」と一言添えて作成へ誘導する。
3. どちらの経路でも、**設定の詳細はそのプロセスのファイルが持つ**。SKILL.md ではこれ以上設定を聞かない。

> このファイルは常時ロードされる薄い入口。重い手順は `procedures/*.md` に置き、その経路に入ったときだけ読む。
> 「解くだけ」のときに生成手順をコンテキストに載せないための進行ディスクロージャ設計。

## 共通の不変条件（両プロセスで必ず守る）

これらは作成・出題のどちらでも崩してはいけない前提。各プロセスファイルはこれを前提に書かれている。

### 公式仕様（Exam Guide `cca-f/CCA-F_Certification_Exam_Guide.md` の確定事項）

| 項目 | 確定事項 | 実装 |
|---|---|---|
| 回答形式 | 各問は**正解1つ＋ディストラクタ3つの単一正解（single-best-answer）** | デフォルトは単一選択。複数選択は「本番には無い練習バリアント」と明示した上で任意 |
| 出題 | シナリオベース。6本中4本がランダム選出され、**全問がシナリオに紐づく** | 6シナリオ原型を内蔵し、問題を必ずシナリオに包む |
| ディストラクタ | 明らかな誤答ではなく本気で迷う設計。**2択が技術的に成立するが片方が設計的に優れている** | 「もっともらしいアンチパターン」を強制し、**2方向の誤反射**（ソフト指示に逃げる／コンポーネントを足す）を狙う |
| スコア | 100〜1000のスケールドスコア。**720は素点72%ではない**（難問が重み付け） | 素点とドメイン別正答率を出し、スケールド推定は「目安」と明記。**素点85〜90%を安全圏に** |
| 無回答 | 無回答は誤答扱い | 全問回答を促す |

**ドメイン重み:** D1 Agentic 27% / D3 Claude Code 20% / D4 Prompt 20% / D2 MCP 18% / D5 Context 15%。

### カンニング防止の鉄則（最重要）

このスキルの会話とファイルはユーザに見える。**出題前に正解・解説をユーザが読めてはいけない。**そのため:

1. **2ファイル分割**: バンク内でも、`quiz-questions.json` は**問題文と選択肢のみ（正解・解説・出典・is_correct を一切含めない）**。
   正解キー・解説・出典・各選択肢の正誤は `answer-key.json`（**平文・別ファイル**）に分離する。目的は「**出題時に読むファイルに答えを同居させない**」こと。
2. **答えは出題前のチャットに出さない**: `answer-key.json` は採点時にだけ読む。生成中の推論でも「正解は〜」と断定的に書かない。
3. **正解位置をランダム化**: 正解を常に A に置かない。各問で A〜D にばらつかせ、バンク全体の正解分布が概ね均等かを生成後に自己チェックする。

### 品質に関する不変条件

1. **根拠なき問題は出さない** — `citations` が空、またはセルフチェックで出典不支持と判定した問題は破棄。
2. **trivia を出さない** — 暗記でなく設計判断を問う。シナリオに包めない問題は破棄。
   問題設計の背骨は **「壊れた本番系を最も早いレイヤーで最小修正する」**。正解は原則 **Constrain, don't add／Prompts suggest, systems enforce／Fix the earliest layer** の側に立ち、ディストラクタは「ソフト指示に逃げる／コンポーネントを足す」誤反射を突く（詳細は `procedures/generate.md`・`antipattern-playbook.md`）。
3. **正解は生成時にセルフチェックする** — 自分の生成理由で正解を正当化しない。`rationale` を伏せてコーパスから正解を再導出し、一致を確認する。
4. **正解を事前に漏らさない** — 出題前に `correct`/`rationale` を `quiz-questions.json` にもチャットにも出さない。答えは別ファイル `answer-key.json` に分離する。
5. **正解位置をランダム化** — 正解を A 固定にしない。生成後に分布を自己チェック。
6. **スケールド推定は目安** — 720=72% ではないことを毎回明記する。
7. **進行中は脱線しない** — 出題ランの各ターンは `progress.json` の次アクションだけを実行する。モード変更・一括提示・無関係な作業をしない。
8. **状態は外部ファイルに持つ** — ランの進捗は `progress.json`、バンクの生成進捗は `manifest.json` に書き、ターン開始時に必ず読み直す。
9. **毎回同じ問題を出さない** — 生成前に出題履歴（`weakness-log.json`・既存 `banks/*/quiz-questions.json`）を読み、既出の `(task_statement × scenario)` や同一 stem を再利用しない。

## ファイル構成

- `procedures/generate.md` — **作成プロセス**の手順。「問題を作成する」経路で読む。
- `procedures/deliver.md` — **出題プロセス**の手順。「問題を解く」経路・進行中ラン再開で読む。
- `blueprint.yaml` — 5ドメイン×タスクステートメントの網羅チェックリスト＋ドメイン別の復習先リンク。**作成時に必読**。
- `scenarios.yaml` — 6シナリオ原型。問題を包む文脈として必ず使う。**作成時に必読**。
- `antipattern-playbook.md` — ディストラクタ素材（頻出トラップ＋各タスクのアンチパターン）。**作成時に必読**。
- `question-schema.json` — JSON スキーマ（`bank_manifest`／`quiz_questions_public`／`answer_key`／`session_progress`／`weakness_log`）。
- `report-template.md` — レポート雛形。出題プロセスのレポートで使う。
- `banks/` — **永続バンク（コミット対象）**。各バンク `banks/<name>/` に `manifest.json`・`quiz-questions.json`・`answer-key.json`。
- `sessions/` — **出題ランの作業領域（gitignore 済み）**。各ラン `sessions/<YYYY-MM-DD-HHMM>/` に `progress.json`（現在地・解いているバンクを参照）・`answers.json`。
- 出力: レポート `cca-f/reports/<YYYY-MM-DD>-session.md`、弱点ログ `cca-f/reports/weakness-log.json`（どちらも gitignore 済み）。
