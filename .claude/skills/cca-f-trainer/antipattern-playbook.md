# アンチパターン・プレイブック（ディストラクタ素材集）

良いディストラクタは「明らかな誤答」ではなく「**本気で迷う**」もの。理想は **2択が技術的に成立するが片方が
設計的に優れている**状態。各ディストラクタには「なぜ魅力的か（tempting）」と「なぜ劣るか（why_wrong）」を
言語化できること——これがそのまま解説の `rationale` になる。

出典は `cca-f/CCA-F_Certification_Exam_Guide.md` の各タスクの "Avoiding anti-patterns" と、正解の裏返し。

---

## 1. 合格者が口を揃える「頻出トラップ」4種（最優先のディストラクタ素材）

これらは「魅力的だが誤り」の典型。設問の文脈に合えば積極的に1つのディストラクタとして仕込む。

### T1. 単一の「スーパーエージェント」で複数エージェントを代替する
- **tempting**: 1体に全ツール・全役割を持たせれば調整コストが消える、シンプルに見える。
- **why_wrong**: 役割分離・最小権限が崩れ、専門外ツールの誤用（統合エージェントが Web 検索する等）と
  ツール選択信頼性の低下（18個 vs 4-5個）を招く。コーディネーター-サブエージェントが正道（D1.2 / D2.3）。

### T2. サブエージェントは親の文脈を自動継承すると誤解する
- **tempting**: コーディネーターが知っていることはサブエージェントも知っているはず、と直感的に思える。
- **why_wrong**: サブエージェントは**隔離された文脈**で動き、親の会話履歴を継承しない。前段の発見は
  プロンプトに**明示的に注入**しなければならない（D1.2 / D1.3）。「自動共有される」系の選択肢は誤り。

### T3. CI/CD で適切なフラグを使わず Claude を対話実行する
- **tempting**: ローカルと同じコマンドをそのまま CI で叩けば動くはず。
- **why_wrong**: 対話モードは入力待ちでハングする。非対話は `-p`/`--print`、構造化出力は
  `--output-format json` + `--json-schema`（D3.6）。`CLAUDE_HEADLESS` や `--batch` は**存在しない**フラグ＝ディストラクタ。

### T4. 高リスク判断で人間の監督を減らす
- **tempting**: 自動化率・解決率を上げるために人間を外す方が「進歩的」に見える。
- **why_wrong**: 金銭・本人確認・方針例外など高リスクでは決定論的ゲートと human-in-the-loop が必要（D1.4 / D5.2 / D5.5）。
  「自己申告確信度／sentiment でエスカレーション」「閾値以上の返金も自動化」系はほぼ誤り。

## 2. 「もっともらしいが劣る」常套手段（汎用ディストラクタ型）

> **誤答の2方向（設計反射のズレ）**: 実試験の不正解は知識不足ではなく**反射の方向ズレ**から生まれる。
> 大きく2方向ある——**(a) 決定論が要る所でソフト指示／プロンプト依存に逃げる**（"Prompts suggest" の罠）、
> **(b) 最小修正で済む所にコンポーネント／上位モデル／コンテキスト窓を足す**（over-engineering, "Constrain, don't add" の罠）。
> 良い問題は1問にこの両方向を仕込むことが多い。正解は常に **Constrain, don't add／Prompts suggest, systems enforce／Fix the earliest layer** の側に立つ。

- **プロンプトで強制すれば十分**: 決定論的遵守が必要な場面（本人確認→返金）でプロンプト指示のみ。
  確率的遵守は非ゼロの失敗率（→ 正解はプログラム的前提ゲート/フック D1.4 / D1.5）。
- **few-shot を足す / 説明を増やす**: 問題が「ツール説明の不足」なのに few-shot で糊塗する（トークン増だけ）。
  逆に「曖昧ケースの判断不足」なのに説明文だけ直す、も誤り。**根本原因に当てた最小・高レバレッジな手**が正解。
- **より大きいモデル/コンテキスト窓に変える**: 注意の希薄化（14ファイル単一パス）はコンテキスト窓では直らない。
  分割パスが正解（D4.6）。「上位モデルに変える」は安易なディストラクタになりやすい。
- **ルーティング分類器/別 MLを足す**: プロンプト最適化を試す前に ML インフラを足す over-engineering（D5.2 / escalation）。
- **多数決/3回実行で2回出たものだけ**: 間欠的にしか出ないバグを抑制してしまう（D4.6 の誤答型）。
- **存在しない機能**: `CLAUDE_HEADLESS`、`--batch`、`.claude/config.json` の commands 配列など。1つだけ混ぜると有効。
  - 注意: ここで誤りなのは **Claude Code CLI のフラグ `--batch` が存在しない**こと。これは下の「Batch **API**」の機能とは別物。混同しない。

> **⚠️ Batch API の事実の罠（正解根拠にしてはいけない）**: 現行 Anthropic Docs では **Batch API は tool use・server tools・
> multi-turn 会話に対応している**。よって「Batch は multi-turn／tool calling ができないから不適」を**正解の根拠にしてはならない**（古い誤情報）。
> Batch の正しい弁別軸は **緊急性・同期性・処理規模だけ**。すなわち「非緊急・非同期・大量処理（夜間レポート等）には向く／
> 即時の対話制御・human-in-the-loop・リアルタイム（ブロッキングな pre-merge 等）には向かない」という軸で問う。
> Batch を扱う問題では、正解も誤答もこの軸で組み、機能の有無（tool use 可否）に依拠しないこと。

## 3. ドメイン別アンチパターン（タスクステートメント対応）

### D1 Agentic
- 自然言語シグナル / アシスタントのテキスト有無でループ終了を判定する（正: `stop_reason`）。
- 反復回数の上限を**主たる**停止機構にする。
- 常にフルパイプラインを通す（正: クエリ複雑性に応じ動的に委任先を選ぶ）。
- サブエージェント通信をコーディネーター経由にしない（観測性・一貫したエラー処理を失う）。
- 並列化すべき Task 呼び出しをターンを分けて逐次発行する（正: 1レスポンスで複数 Task）。

### D2 Tool / MCP
- 似た説明文の重複ツールを放置（analyze_content vs analyze_document）。
- 一律の "Operation failed"（正: errorCategory / isRetryable / 人間可読な説明）。
- transient と business エラーを区別しない、retriable:false を返さない。
- 1エージェントに過剰なツール（18個）。専門外ツールを与えて誤用を招く。
- 共有すべき MCP を `~/.claude.json`（個人）に、個人実験を `.mcp.json`（共有）に置く取り違え。
- 標準統合（Jira 等）で既存コミュニティ MCP を使わず自前実装する。
- 内容検索に Glob、パス検索に Grep（逆）。

### D3 Claude Code
- チーム共有設定を `~/.claude/`（user 個人＝非共有）に置く。
- CI で `-p` 無し対話実行（T3）。存在しないフラグ（`--batch` 等）。
- 横断ルールを各ディレクトリ CLAUDE.md に分散（正: `.claude/rules/` の paths グロブ）。
- スキル（手動/任意ロード）を「自動・決定論的」適用が必要な場面に充てる（正: rules）。
- 大規模・アーキ判断を直接実行で始める。「複雑になったらプランへ切替」も誤り（複雑さは既に明示されている）。
- CLAUDE.md（常時ロードの普遍規約）と skill（オンデマンド）を取り違える。

### D4 Prompt / Structured Output
- 「be conservative」「高確信度のみ」で精度向上を図る（正: 具体的カテゴリ基準）。
- prose の説明だけで一貫性を出そうとする（正: 2-4個の入出力 few-shot）。
- 情報が無いかもしれないフィールドを required にして捏造を誘発（正: nullable）。
- tool_use の strict スキーマ／structured outputs が**意味エラー**（合計不一致・誤った値・事実の捏造等）も防ぐと誤解する。
  保証されるのは**構造（スキーマ準拠）だけ**で、**意味的正しさは別レイヤーの検証（semantic validation）が必要**。「スキーマを通った＝正しい」は典型的な誤答。
- 情報が源文書に無いのに retry を繰り返す（format/構造エラーには有効、不在には無効）。再試行は「どのフィールドがどう不正か」を具体的に返すと有効、「エラーだからやり直して」の曖昧リトライは無効。
- ブロッキングな pre-merge を Batches API（最大24h・SLA無し）に載せる。※不適な理由は**レイテンシ／同期性**であって「Batch が tool use/multi-turn 不可」ではない（上の事実の罠を参照）。
- 生成と同一セッションで自己レビュー（正: 独立インスタンス）。

### D5 Context / Reliability
- 進捗的要約で数値・日付・顧客の明示的期待を曖昧化（正: case facts ブロックに転記）。
- lost-in-the-middle を無視して重要所見を中間に置く（正: 先頭/末尾＋見出し）。
- 冗長ツール出力（40+フィールド）をそのまま蓄積（正: 関連フィールドのみトリミング）。
- self-reported confidence / sentiment でエスカレーション（不安定な代理指標）。
- 複数顧客一致をヒューリスティックで選ぶ（正: 追加識別子を要求）。
- サブエージェントのエラーを空結果=成功で握り潰す / 単一失敗で全体終了（正: 構造化エラー文脈で局所回復）。
- 集計精度（97%）で局所不良を隠す（正: 文書型・フィールド別に検証、層化抽出）。
- 矛盾する統計から根拠なく1つ選ぶ（正: 出所注釈で両論併記、temporal は日付付与）。
