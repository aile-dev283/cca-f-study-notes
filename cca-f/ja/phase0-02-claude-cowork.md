Cowork の紹介

このビデオでは、Claude Cowork と、それが担うために作られた仕事の種類を紹介します。Cowork がどのように機能するかに注目してください——より賢いチャットボットとしてではなく、Claude と協働するまったく別の方法として。

**重要なポイント**

Cowork は、単なる答えのためではなく、仕事のために作られた Claude です。ローカルのマシン上、クラウドアプリ、ブラウザのいずれであっても、あなたが働く場所に寄り添います。
Cowork は委任することがテーマです。Chat は、考えを声に出して整理したり、下書きをしたり、質問したりする場として依然として使えます。Cowork が新たに加えるのは、成果（アウトカム）を説明すれば、Claude が計画・実行し、その全体を成果物として届けてくれる能力です。
Cowork は、複数ステップにわたる、長時間かかる仕事のために作られています。複数のツールにまたがり、完了までに時間がかかり、最終的に実際の成果物に至るタスク——Cowork はそうした一連の流れ全体をエンドツーエンドで処理できるよう設計されています。
あなたが主導権を握り続けます。Claude は開始前に計画を見せ、デフォルトでは重要な操作（送信・削除・共有）を行う前に確認を求め、いつでもあなたが軌道を修正できるようにします。
Cowork の実体とは何か

Cowork は、デスクトップアプリ内の Claude のモードで、その中で Claude があなたと並んでタスクに取り組みます。コンピュータ上のフォルダを指定し、仕事が存在するアプリ（Gmail、Slack、Google Drive、カレンダー）を接続し、やってほしいことを説明します。Claude は計画を立て、ステップを順に進め、必要なツールとファイルを使い、実際の成果物をあなたのフォルダに保存します。

最も重要なメンタルモデル: Cowork は、単にチャットすることではなく、委任することがテーマです。Chat は依然として非常に強力です——問題を考え抜き、下書きと推敲を行い、ブレインストーミングをし、質問をし、Claude を壁打ち相手にする場です。Cowork はそれらのいずれも置き換えません。Cowork が新たに加えるのは、一連の仕事全体——コンテキスト収集、分析、ファイル作成、ツール利用——を Claude に渡し、完了した状態で返ってくるという能力です。

これは新しい Cowork ユーザーが乗り越えるべき最大の発想の転換であり、これを本当に身につけることが不可欠です。多くの人の最初の本能は——Chat から来ているため——質問を入力し、返ってきたものを見て、追加の質問を入力し、反復する、というものです。それは Cowork でも機能しますが、本来あなた自身がやるであろう仕事（普段なら他人に尋ねる仕事ではなく）のために使ったときに、最も大きな効果を得られます。

しかし両者の違いはこれよりもさらに深いところにあります。仕組みの面で、Chat はターンごとのやり取りです。ツールから情報を引き出すことはできますが、ステップは自分でつなぎ合わせます。Chat は、区切られた範囲のやり取りと素早い思考のために専用設計されています。一方 Cowork は、一連の仕事全体を持続させます: 多数のツールから同時に情報を引き出し、複数ステップの計画を実行し、完成した出力を返します。これが、Chat へのアップロードでは完全には埋められない差です。

Chat 対 Cowork 対 Code

Cowork が何であるかが分かったところで、デスクトップアプリで Claude と協働する他の 2 つの方法と、それがどう並び立つのかを少し見てみましょう。

3 つのモードは、それぞれ異なる形の仕事のために作られています。Chat はターンごとの対話です——素早いやり取り、ブレインストーミング、探索的な思考、一度に 1 つのツール。Cowork はこれまで読んできたとおりのものです: Claude が多数のツールを横断して手を伸ばし、時間をかけて仕事を持続させ、完成した出力を生み出すワークスペース。Code は、コードベースの内部で動作し、ターミナルと git のアクセスを備えた、完全な開発環境です——ソフトウェアを書いて出荷する開発者のために作られています。

コネクタの追加

コネクタは、Claude が、あなたの仕事がすでに行われているアプリ——メール、カレンダー、チームのメッセージングツール、CRM、クラウドストレージ——に手を伸ばすための仕組みです。Customize エリアで一度設定すれば、あとはタスクごとに必要なコネクタをオン/オフで切り替えられます。

多くの人が最初に設定するコネクタ:

メールとカレンダー（M365 経由の Outlook、または Gmail）— ミーティングからコンテキストを引き出したり、フォローアップを下書きしたり、過去のスレッドを見つけたりするため
メッセージング（Slack、または M365 経由の Teams）— チャネル履歴を検索し、チームが言ったことを統合するため
クラウドストレージ（Microsoft 365 経由の SharePoint や OneDrive、Google Drive、Box）— ローカルマシンに存在しない文書にアクセスするため
CRM とプロジェクトツール — Notion、Salesforce、HubSpot、Asana、Linear など。チームが使っているもの、実データが存在する場所に応じて

コネクタがオンになれば、プロンプトの中で自然に参照できます。「ローンチについてチームが Slack で言ったことを確認して」や「先四半期の顧客フォローアップメールを見つけて」——Claude はどこを見ればよいかを把握しています。

Claude Cowork に向いたタスクとは何か

すべてのタスクが Cowork に向いているわけではなく、向いているタスクを見分けられるようになると、より多くの効果を得られます。3 つのパターンが、Cowork のために作られた仕事の大半をカバーします: 複数のステップを要するタスク、実在のファイルからコンテキストを活用するタスク、すでに使っているツールにまたがるタスク。

パターン 1: タスクが複数ステップである。タスクがいくつものステップを要するなら、Cowork に向いています。1 つのプロンプトで、Claude はコンテキストを収集し、情報源を比較し、追加の調査を行い、要約を下書きし、成果物を整形します。Cowork なら、その一連の流れ全体を Claude に渡せます。

例:

1 週間分の顧客フィードバックメールをテーマ別に振り分け、各テーマに代表的な引用を添える
3 つのレポートと 1 つのスプレッドシートから数値を引き出し、それらを 1 つのインタラクティブなダッシュボードに組み立てる

パターン 2: タスクがコンピュータ上の実在のファイルに存在する。Cowork は、タスクの目的が実際の成果物——Word 文書、スプレッドシート、デッキ、PDF——を必要な形式で保存して出力することであるとき、最適です。入力もまた実在のものです: Claude はフォルダ内の既存ファイルを読み、編集し、変更を保存して戻します。Chat は新しい文書を作成できますが、Cowork はあなたがすでに持っているファイルに対して作業します。

例:

標準テンプレートとミーティングメモから作られた、整形済みの顧客提案書
生のスプレッドシートから組み立てられ、チャートまで備えた月次メトリクスレポート

パターン 3: タスクが複数のツールにまたがる。仕事は、あなたがすでに作業しているシステム——Gmail、Slack、M365 スイート、カレンダー、CRM——に触れます。Claude はあなたのコネクタ（Claude をアプリにつなぐ統合機能）を通じてそれらにアクセスし、コンテキストを収集し、操作します。Chat もツールに接続できます——違いは、Cowork はそれらを横断して計画を立て、一連の流れ全体を 1 つの委任として実行する点です。あなたが 1 プロンプトずつステップをつなぎ合わせるのではなく。

例:

ミーティング後に、カレンダー招待・出席者リスト・ミーティングメモに基づいて Outlook でフォローアップメールを下書きする
ローンチについてチームが言ったすべてを Slack で検索し、更新情報に統合する

Cowork は仕事をステップに分解し、実行しながら各ステップを見せてくれます。作業を見守ってもよいし、終わったら戻ってきてもかまいません。

3 つのパターンすべてに共通する筋道は同じです: かつては自分の頭の中でステップ・ファイル・ツールをやりくりすることを意味していた仕事を、いまや 1 つのタスクとして引き渡せます。その形を見分けたら、Cowork に渡しましょう。

スケジュールされたタスク: 一定の周期で Claude に仕事をさせる

スケジュールされたタスク（Scheduled tasks）を使うと、タスクを一度設定すれば、Claude に一定の周期で実行させられます。

スケジュールされたタスクを設定する方法は 2 つあります。ゼロから始める方法: /schedule と入力し、タスクと周期を説明すると、Claude があなたが確認して承認するためのプロンプトを下書きします。あるいは——より自然な方法——Cowork で一度そのタスクを実行し、出力が正しく見えることを確認してから、/schedule と入力してその同じ手順を繰り返し実行されるものに変えます。希望する周期（毎時、毎日、平日、または手動）を Claude に伝え、プロンプトを確認して承認します。

スケジュールされたタスクは、コンピュータがオンで Claude が起動しているときにのみ実行されます。ノートパソコンが閉じている、スリープ中、またはスケジュール時刻にオフだった場合、Claude は戻ってきた時点で速やかにタスクを拾い、遅延したことを知らせます。

スケジュールする価値のあるアイデアの例:

金曜午後のレビュー: 今週何が出荷されたか、Slack や Teams で下された主要な意思決定の振り返り、来週のタスクのプレビュー
毎月 1 日にスプレッドシートから引き出される月次メトリクスのまとめ
カレンダー上のミーティングのための朝のブリーフィング。フォルダ内の過去のミーティングメモやメールスレッドを参照して、必要なコンテキストを浮かび上がらせる
Dispatch: スマートフォンから Claude にタスクを送る

ここまでで、デスクで開始するタスクと、Claude がスケジュールで開始するタスクを取り上げました。Dispatch は 3 つ目のケースをカバーします: コンピュータからまったく離れている間に開始するタスク。Dispatch を使うと、Claude モバイルアプリから Cowork タスクを開始できます。タスクをスマートフォンから割り当て、仕事はデスクトップで実行されます——同じファイル、同じコネクタ、同じ権限で。

このビデオでは、委任から成果物まで、1 つの実在する仕事を通して進めます: プロンプト、Claude が尋ねる確認質問、ユーザーが途中で進路を変える瞬間、そして最後の完成した結果。そのリズムに注目してください——チャットボットにプロンプトを与えるというより、これからタスクに没頭して何かを完成させて戻ってくる賢い同僚に手短に指示を出すような感覚です。

**重要なポイント**

Claude は着手する前に確認します。たいていの単純でないタスクは、1 つか 2 つの確認質問から始まります。質問は、仕事が始まる前に Claude がコンテキストのギャップを埋める手段であって、邪魔になる摩擦ではありません。
タスク途中で軌道を修正できます。Claude が誤った方向に進んでいるなら、割り込んでください。終わるのを待ってやり直す必要はありません。
完成した成果物が成果物であり、チャットが成果物ではありません。Claude はコンピュータ上のファイルに変更を加えます。最後にあなたがすべき仕事は、同僚の仕事をレビューするように、それをレビューすることです。
最初のタスクを委任する

前のレッスンでタスクを選びました。いよいよそれがあなたの手から離れます。Cowork を開き、タスクのコンテキストが存在するフォルダを指定し、タスクに関連するコンテキストを持つコネクタをオンにし、プロンプトを書きます。

良い Cowork プロンプトは 3 つのことを行います:

成果物を明示する。「1 ページのブリーフ」「QBR 用のスライド 1 枚」「候補ごとに注記を添えた順位付きリスト」。形式と長さについて具体的に示すと、再生成を 1 回省けます。

**学習目標**

Chat では、メモリは自動的に構築されます——オンにすれば、Claude が会話から自動的に学びます。Cowork は異なる働きをします。タスクからタスクへ引き継がれるコンテキストは、主にあなたが設定するコンテキストです: すべての Cowork セッションに適用されるグローバル指示と、その中での会話から Claude が自動的にメモリを構築するプロジェクト。多くの人はグローバル指示から始め、繰り返しの仕事が浮上するにつれてプロジェクトを追加します。

グローバル指示: すべてのセッションに適用されるブリーフ

グローバル指示（Global instructions）は常設のブリーフです。Settings で一度書けば、Claude はすべてのセッション——すべてのチャット、すべてのスケジュールされたタスク、すべての Dispatch——でそれを参照します。

設定するには:

1. Claude デスクトップアプリを開き、Settings → Cowork に移動します。
2. Global instructions の横の Edit をクリックします。
3. ブリーフを書いて保存します。

何を入れるか: あなたが誰で何をしているか; あなたが使う略語や頭字語（Claude が「the QBR deck」とは何かをいちいち尋ねなくて済むように）; そして出力をどう届けてほしいか（形式、長さ、トーンなど）。以下のインタラクティブは、異なる役割に基づくいくつかの実例を示します。

プロジェクト: 1 つの仕事の流れのための範囲を絞ったワークスペース

グローバル指示はあなたをカバーします。プロジェクトはあなたが取り組んでいるものをカバーします。

プロジェクトは、特定の仕事の流れ——顧客、繰り返しの成果物、ローンチ——に結びついたワークスペースです。プロジェクトの中には、あなたが設定できる 3 つのものと、Claude があなたのために構築する 1 つのものがあります:

Instructions — グローバル指示と同様ですが、このプロジェクトに範囲を絞ったもの。（「このプロジェクトは毎週のトランスフォーメーションチームミーティング用です。目的は、組織全体からデータを集め、毎週それをスライドデッキに凝縮することです。」）
Scheduled tasks — このプロジェクトに属する、設定済みの繰り返し実行: 金曜のステータス収集、ミーティング前のブリーフ、月末のまとめ。これらはプロジェクト内の会話から設定し、毎回プロジェクトのコンテキストとともに実行されます。
Context — Claude が作業の起点とすべき 1 つ以上のフォルダまたはリンク。プロジェクト内のすべての会話がそれらにアクセスできます。
Memory — プロジェクト内で行った会話から Claude が学ぶこと。作業するにつれて時間とともに積み上がります。あなたが書くものではありません。

最後のものが、プロジェクトがもたらす違いです。プロジェクトの外では、各セッションはグローバル指示を除いてまっさらな状態で始まります。プロジェクトの中では、すべての会話が Claude の知識に加わります——その結果、次のタスクは、顧客の状況、先週の意思決定、まだ未解決のことをすでに手にした状態で始まります。

### レッスン 07: Skills: Teach Claude Cowork your way

**URL:** <https://anthropic.skilljar.com/introduction-to-claude-cowork/444170>  

Open in Claude
⁠

所要時間の目安: 12 分

**学習目標**

**このレッスンを終えると、次のことができるようになります:**

スキルを定義し、Claude がそれをどう使うかを説明できる
スキルが含みうる 4 つの構成要素を認識できる
あなた自身の繰り返しプロセスの 1 つからスキルを構築できる
スキルの実体とは何か

スキルは再利用可能なプレイブック——ファイルとリソースのフォルダ——であり、特定の種類の仕事を、あなたが望むやり方で行う方法を Claude に教えます。スキルに合致するタスクを始めると、Claude はそのプレイブックを読み込み、それに従います。

スキルは、必要なときにタスク中で自動的に使われます。名前で呼び出す必要はありません。Claude は、タスクがインストール済みのスキルに合致することに気づき、自動的に読み込みます。明示的に指定すること（「the board memo drafting skill を使って」）もできます。

スキルの中身

スキルは、長い指示文以上のものです。スキルが含みうる 4 種類のファイル——そしてそれらがどう連携するか——こそが、あなたのチームがやるように Claude が実行できるほど、実プロセスをうまくエンコードする方法です:

Instructions（SKILL.md ファイル）。そのスキルが何をするか、いつ使うか、どうやるかを Claude に伝えるブリーフ。新しい同僚向けのランブックを書くように書きましょう——彼らが仕事をこなせるほど具体的に。
Assets。ロゴ、ブランドテンプレート、スライドマスター、フォント。スキルが本物らしい出力を生み出すために使う素材。
References。良い出力の例、スタイルガイド、条項ライブラリ、新しいチームメイトに目標水準として手渡すような過去の仕事。References は、この種の仕事における「良い」とは何かを Claude が学ぶ手段です。
Scripts。毎回同じように行われるべきプロセスの部分のために Claude が実行できる小さなコード——差異計算、構造化された比較、チャートの整形ツール、文書の再整形ツール。

スキルはこれらの任意の組み合わせを使えます。あるスキルは指示だけの SKILL.md ファイルで、それで一部のプロセスにはまったく問題ありません。別のスキルは SKILL.md にブランドアセットフォルダを加えたもの。別のスキルは 4 つすべてを持ちます。組み合わせは仕事に従います: 含むべきものを含め、それ以上は含めない。

以下はスキルの 3 つの例です。それぞれをクリックして、その用途と構成の感覚をつかんでください。

INSIDE A SKILL

スキルはフォルダです。中身は、プロセスによって変わります。3 つの例を切り替えてください:

meeting-recap
board-memo
variance-analysis
meeting-recap/
SKILL.md
meeting-recap フォルダの SKILL.md をクリックして、それが何をするかを見てください。

これこそが、スキルがチームの働き方を成文化するのにとても役立つ理由です。Cowork はあなたの代わりに動ける同僚であり——スキルは、あるべきやり方で仕事をさせる手段です。Instructions が何をするかを伝え、Assets が素材を与え、References が良いものとは何かを示し、Scripts が繰り返し可能な部分を毎回同じように実行させます。

Claude とスキルを構築する

スキルを構築する最速の方法は、Claude と一緒に行うことです。

Cowork で新しい会話を始めて、次のように言ってください:

I want to build a skill for [the recurring process you're tired of re-explaining]. Walk me through what you need to know.

Claude はいくつか質問します: スキルが何をすべきか、いつトリガーすべきか、良い出力とは何か、スキルに情報を与えるためにどのリソースを使うべきか。できるだけ具体的に答えましょう——実際の仕事の例、実際のテンプレート、実際の過去の出力を示してください。出力は、SKILL.md と、スキルが必要とするアセット・リファレンス・スクリプトを含む、インストール可能な状態のスキルフォルダです。

インストールされると、そのスキルは Customize で見つけられます。スキルに変更を加えたい場合は、修正点を Claude に伝えてスキルを更新するよう頼むだけです。「2 ステージ滑った 10 万ドル超のディールにフラグを立てるステップを追加して——それは常に重要だから。」Claude はその場でスキルを更新します。

スキルは、プロジェクト内の会話を含め、どの会話の中でも同じように働きます。だから、差異分析用に構築したスキルは、差異分析がタスクであるときには——デフォルトの Cowork セッションで作業していようと、特定のファイナンスプロジェクトの中であろうと——いつでも現れます。

レッスンの振り返り

**重要なポイント**

Claude は文書の中に存在します。Claude は Word、Excel、PowerPoint、Outlook の中にアドインとして現れ、あなたが開いているファイルとスペースで作業します。
1 つの会話が、アプリ間でコンテキストを運べます。Excel で分析を構築し、それを PowerPoint に渡してスライドを作る。Word のメモを、下書きする Outlook の返信の元として使う。

各アプリの中で Claude が何をするか

Excel。データを分析し、数式を書き、#REF! エラーや循環参照をデバッグし、モデルを壊さずにシナリオテストを実行し、テンプレートからシートを構築し、特定のセルに引用を戻す。最も強力な一手: 「Q3 シートから実績を引き出し、同じワークブック内の Q3 計画と比較し、各明細項目の隣の F 列に差異コメントを書いて。」

PowerPoint。スライドマスター、フォント、配色を読み、それに合うスライドを構築する。ネイティブで編集可能なチャート（貼り付けた画像ではなく）を生成する。選択中のスライドに対して操作する。最も強力な一手: 「さっき Excel で行った分析を、私たちのデッキテンプレートを使って QBR 用の 3 枚スライドのデッキにして。」

Word。その場で下書き・修正・再整形する。コメントと変更履歴を扱う。接続された情報源からコンテキストを引き出して下書きを裏付ける。最も強力な一手: 「このメモの本文と、付録で参照されている元データに基づいて、エグゼクティブサマリーを下書きして。」

Outlook。仕事の他の部分からのコンテキストを使って受信メールを振り分ける。過去のスレッド、カレンダーのコンテキスト、最近の意思決定を反映した返信を下書きする。

1 つの会話でアプリ間を仕事が移動する

アプリ間の移動こそ、M365 と Claude の組み合わせが、単一のアプリで作業するのとは違うと感じ始める場所です。1 つの文書で作業するだけでなく——そのコンテキストを次の文書へ運んでいるのです。

Outlook → Word。顧客ブリーフが受信トレイに届きます。「ブリーフを Word で開いて、私たちの事務所テンプレートからメモを始めて。」Word が起動し、添付ファイルが開かれ、メールスレッドがすでにサイドバーに読み込まれています。Claude は、送信者が何を求めたかをすでに把握しています。

Outlook に戻る。「木曜までにチームと 30 分を見つけて。」招待が出席者とともに下書きされ、あなたが Send をクリックするのを待ちます。

役立つルール:

仕事が多数の情報源から引き出し、成果物に至るときは Cowork に手を伸ばす。20 個の元ファイルからブリーフを構築する、Salesforce と 3 つの Slack チャネルからレポートをまとめる、スケジュールでワークフローを実行する。
Office ファイルそのものの中で作業しているとき——その場で編集し、あるアプリから次のアプリへコンテキストを運ぶとき——は M365 内の Claude に手を伸ばす。

ほとんどの実際の仕事は両方を使います。Claude がデッキの最初の下書きを構築し、あなたが PowerPoint で開いてスライド内の Claude と推敲する。Claude がデータをモデルに引き出し、あなたが Excel で開いて、見つけた外れ値を調べるよう Claude に頼む。2 つの面（サーフェス）はきれいに引き継ぎ合います——そしてどちらの場合も、ファイルはそのファイルです。

今すぐ試す

Cowork で安全に作業するうえでのあなたの役割

レッスン 2 から、Claude は常に削除の前に確認し——そしてデフォルトの権限モードでは、送信や共有の前にも確認することを、すでに知っています。それが最低限の基準です。このレッスンは、あなたがその上に加えるもの——ファイル・アプリ・コネクタにアクセスできる自律的なツールが、意図しないことを行わないようにする先回りの手——についてです。

ミスが重要なものに届かないように設定する

最もレバレッジの大きい一手は、Claude を指定するフォルダです。それが、Claude が読み・書き・（あなたの確認のうえで）削除できる範囲の境界です。

何でも入れる場所ではなく、専用の作業フォルダを使う。Claude を Documents、Downloads、Desktop に向けるのは、新しい同僚に持っているすべてのファイルをあさらせるのと同じことです。仕事用のフォルダを作りましょう。必要なものを移動（またはコピー）して入れます。Claude をそこに向けます。
始める前に、かけがえのないものはバックアップする。あるファイルが重要で、新しいコピーを再生成できないなら——古い顧客成果物、再発行できない契約、失ったら困るもの——Cowork が届かない場所にコピーがあることを確かめてください。クラウドバックアップ、別のフォルダ、接続されていないドライブ。Claude は確認なしに削除しませんが、誤った確認をクリックして通過させるコストは、そのファイルのコストです。
新しいワークフローはまずコピーでテストする。たとえば毎週金曜に実行されるスケジュールタスクを構築しているなら、最初の実行はデータのコピーに対して行います。きちんと振る舞うのを見届けたら、本番のフォルダに向けます。
誤った操作の余地を残さないプロンプトを書く

どう頼むかは、どのフォルダを指定するかと同じくらい重要です。

破壊的な動詞については具体的に。「セクションを cut して」は「表示から取り除く」とも「ファイルから削除する」とも読めます。「ファイルを update して」は「書き直す」とも「追加する」とも取れます。誤った読み方が取り返しのつかないものなら、意図することを明示しましょう: 「ドラフトからセクションを取り除いて、でもファイルは残して。」「新しい付録を追加して。既存のセクションは書き直さないで。」
プロンプトで境界を明示する。「このフォルダの中で最も最近更新された 3 ファイルだけ。」「Q3 にクローズした契約だけ。」「誰にもメッセージを送らないで——下書きだけ。」これは Cowork がやっていることを狭め、逸脱を見つけるための明確な基準を与えます。
最初はスケジュールタスクを下書き用に使う。スケジュールタスクは、あなたが見ていない間に実行されます。そのタスクが必要なとおりに動くと確信できるまでは、あなたの代わりに送信するのではなく、レビュー用に下書きするよう指示しましょう。
その瞬間に: 残りを捕まえる 3 つのチェック

計画が立てられたら、それを一度読む。Claude がタスクを始めると、進捗タブに何をしようとしているかを示します。ざっと目を通しましょう。考えてください: 計画は理にかなっているか? ステップは正しい順序か? 正しい情報源を使っているか? 必要に応じて軌道を修正します。

予期しないパターンに注意する。すべてのコマンドを検証する必要はありません。しかし、Claude が言及していないファイルやサイトに触れていたり、頼んだ範囲を超えてスコープが広がっていたりするなら、タスクを停止してください。「何かおかしい気がする」は本物の信号です——これに注意を払いましょう。

確認プロンプトは慎重に承認する。送信・投稿・共有するものについては「Ask before acting（操作前に確認）」のままにしてください——そして確認プロンプトが実際に現れたら、それを読んでください。ほとんどのミスは、安全装置が失敗したから起こるのではありません。意図した操作とは少し違う確認をクリックして通過させたから起こるのです。ダイアログが存在するのは、その操作が重要だからです——そのように扱いましょう。

Cowork が適切なツールでないとき

短いリスト:

監査証跡を必要とする規制対象のワークフロー。Cowork の活動は、監査ログ、Compliance API、データエクスポートには記録されません。
賢く素早い同僚に、監督なしで任せたくないこと。リーガル文書を取引相手に送る、公開アナウンスを投稿する、顧客向けの変更をプッシュする。Claude は準備できます。出荷するのはあなたです。
IT チームが明示的に承認した境界の外にある、極めて機微な個人データ。
さらに深く

Use Claude Cowork safely は、Cowork で Claude と作業するときに何をすべきか——そして何をすべきでないか——を学ぶための価値あるリソースです。組み込みのガードレールと、Claude があなたの代わりに動く間にあなたが引き続き負う責任を理解できるよう、さらに考慮を要する残りの事柄をカバーしています。

なぜこれが重要か

スキルを構築したり、それらをプラグインにまとめたりするとき、あなたは本質的に、他の人が使う小さなプロダクトを作っています。そして、同僚に手渡すもの——テンプレート、スプレッドシートモデル、チェックリスト——と同じように、机を離れる前に試運転する価値があります。

自分で構築したスキルを使うときは、どんな問題や失敗も回避する方法を知っています。何を尋ねればよいか、どのファイルを与えればよいか、答えがどう見えるはずかを正確に知っています。チームメイトにはそれが一切ありません。彼らはリクエストを少し違う言い回しで述べたり、わずかに違う入力を渡したり、エッジケース——スキルが想定した範囲のすぐ外にあるリクエストのような、珍しいが実際に起こる状況——にぶつかったりするかもしれません。スキルがつまずきがちなのがそこで、使っている人はなぜかが分かりません。

evals——evaluations の略——でスキルをテストすることは、他の誰かが気づく前にそうしたつまずきを捕まえる方法です。この言葉に怖じ気づかないでください。eval は単なる試運転です: 現実的なリクエストを入れ、出てくるものを見て、何を直すべきかを Claude に伝えます。コードもテストスクリプトも不要——あなたの名前を冠して出せるほど結果が十分良いかどうかについての、あなたの判断だけです。

eval システムの仕組み

skill-creator——Claude の組み込みのスキル作成ヘルパー——でスキルを構築すると、プロセスの一部として evals を案内してくれます。実際にそれがどう見えるかは次のとおりです。

skill-creator は、誰かがあなたのスキルで使いそうな現実的なプロンプトを 2 つ以上考え出します。各プロンプトについて、1 組の出力を生成します:

1 つは Claude があなたのスキルを使ったもの
1 つは Claude が同じプロンプトにあなたのスキルなしで答えたもの

2 つ目が比較の基準点です。あなたのスキルが実際にどんな違いを生んでいるかを並べて見られるようにそこにあります——単に「この出力は問題ないか」ではなく、「この出力は、Claude が単独でやったであろうものより良いか」を見るためです。

各組をレビューし、レビューページ上で、プレーンな英語でフィードバックを与えます。各組を読みながら、あなたが本当に答えているのは 2 つの問いです:

スキル版が、私が使う方か? もしそうなら、素晴らしい——スキルがそれをやり続けるよう、何が良かったかを書き留めましょう。
そうでないなら、何が欠けているか、ずれているか? 具体的に。「トーンがフォーマルすぎる」や「エグゼクティブサマリーを飛ばした」は、Claude が手を打てる材料になります。「これはちょっと違う」では何もなりません。

フィードバックを送信すると、Claude はあなたが言ったことに基づいてスキルを修正します。

スキルを反復する

あなたのフィードバックが修正そのものです。送信すると、Claude はスキルを更新し——指示を書き直し、例を調整し、求めるものを引き締め——同じプロンプトをもう一度実行して、変更が定着したかを見られます。

一度に 1 つだけ変える。最初のラウンドで、スキルが冗長すぎてセクションが欠けていると分かったら、より重要な方を選び、それを直し、再実行し、それから別のレビューに戻ります。何が実際に効いたかが分かるようになります。修正後の出力にまだ満足できないなら、もう一度実行してください——これはループであって、一度きりの関門ではありません。ほとんどのスキルは 1 〜 2 ラウンドで仕上がります。スキルを——自分にも、チームメイトにも——出荷する基準は、完璧な evals ではありません。気にかけるケースがベースラインより有意に良く通過すること、そしてまだ対応できていないケースを明示したこと、です。

そして、最初のパスで出力がすでに素晴らしく見えたら? それで完了です。Evals は飛び越えるための関門ではありません——儀式ではなく、確信が必要なときのためにあります。

### レッスン 13: Share what you build with your team

**URL:** <https://anthropic.skilljar.com/introduction-to-claude-cowork/485950>  

Open in Claude

所要時間の目安: 7 分

**学習目標**

**このレッスンを終えると、次のことができるようになります:**

プラグインが Enterprise 組織全体にどう配布されるかを説明できる
共有プラグインを時間とともに健全に保つための良い習慣をいくつか適用できる
チーム全体にワークフローを拡大する

この時点で、あなたのチームには、その地位を勝ち取ったいくつかのスキルがあります。それらは誰かの個人的な働き方として始まり、evals を通り抜け、複数の人のユースケースで持ちこたえています。

それらをチーム全体に効率的に拡大したいときは、それらをプラグインにまとめます——やり方はレッスン 8 で扱いました。このレッスンは次のステップについてです: そのプラグインを必要とするすべての人に届けること。

組織内でプラグインを配布する

より大きな会社の中では、プラグインを共有する推奨される方法は、組織のプライベートマーケットプレイス——管理者が管理する、会社承認済みプラグインのカタログ——を通じてです。

実際には、配布は引き渡しです。あなたは、組織のマーケットプレイスを所有する人——チームリード、イネーブルメントやオペレーションの担当者、または IT かもしれません——にプラグインを持っていき、彼らがそれを公開します。公開するとき、彼らは他のみんなにとってそれがどう届くかを選びます:

Available — 会社の Directory に表示され、欲しい人はインストールできる。
Installed by default — 人々が Cowork を開いたときにすでにそこにある。オフにできる。
Required — インストールされ、オンのままになる。誰もが同じやり方で実行する必要があるコンプライアンスチェックのようなものに有用。

チームメイトの立場からは、プラグインは単に、公開された Anthropic のものと並んで、あなたの会社から来ていると表示された状態で Directory に現れます。彼らはそれを使い、（必須でない限り）オフにできますが、編集はできません——更新は、それを保守する人から流れてきます。

これがあなたにとってどう見えるかは、管理者が何を設定したかによります。マーケットプレイスを稼働させ、引き渡す明確なオーナーがいる組織もあれば、まだ有効にしていない組織もあります。以下のインタラクティブで、あなたの状況と適切な次のステップを見つけてください。

DISTRIBUTION PATHS

Cowork でプラグインを共有しに行くとき、見えるものは組織がどう設定したかによります。合致するものを選んでください:

道はすでに舗装されています。あなたの仕事は設定ではなく、引き渡しです。

YOUR NEXT STEP

プラグインをマーケットプレイスのオーナー——チームリード、イネーブルメント、または IT——に持っていきましょう。誰向けか、どう届くべきかを伝えます。彼らは公開時に以下のインストールレベルの 1 つを選びます。

Available
欲しい人のために Directory に。ほとんどのチームプラグインに適切。
Installed by default
チームメイトが Cowork を開いたときにすでにオン。オフに切り替えられる。
Required
オンで、オンのまま。コンプライアンスや、同じやり方で実行しなければならない仕事に限定。
Hidden
マーケットプレイスにはあるが Directory には表示されない——ステージングや限定的な展開のため。

公開されると、チームメイトは、あなたの会社から来ていると表示された状態で Directory にそれを見ます。あなたの更新は自動的に彼らに流れます。

どの道が当てはまるにせよ、ゴールは同じです: 1 台のノートパソコンで始まったスキルが、チームメイトがインストールして実行できるものになること。

組織で見えるものを、それに合う配布パス——そしてマーケットプレイスがまだ稼働していなければ誰に話すか——に合致させましょう。

維持する価値のある習慣

共有プラグインがひっそりと陳腐化するのを防ぐ、短い実践のセット:

1 人のオーナー。すべての共有プラグインには、変更をレビューし、編集後に evals を実行し、いつ更新または引退させるかを決める、指名された人がいます。
公開のたびに evals。eval ループを関門として扱いましょう——変更後に、気にかけるケースが持ちこたえないなら、みんなにプッシュしないでください。
スキルとプラグインに具体的な名前を付ける。「meeting-prep」は、組織のプラグインディレクトリ全体にある他の 3 つの meeting-prep スキルと衝突するかもしれません。「sales-customer-renewal-prep」なら衝突しません。
レビューのリズムを設定する。四半期ごとは、何がインストールされ、何が実際に使われ、何が陳腐化したかを見る、妥当な出発点です。誰も実行しないものは引退させ、人々が改善の機会を見つけたところには修正を加えます。
