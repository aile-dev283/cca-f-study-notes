## カリキュラム

### レッスン 01: サブエージェントとは何か

**URL:** <https://anthropic.skilljar.com/introduction-to-subagents/450698>  

サブエージェントとは、Claude Code がタスクを委任できる専門化されたアシスタントである。集中して働くヘルパーだと考えるとよい。それぞれが独自の会話コンテキストウィンドウで実行され、作業を行い、メインスレッドへ要約を返す。途中で行われたすべてのファイル読み取り、検索、ツール呼び出しといった中間ステップは隔離され、メインの会話を散らかさない。

なぜサブエージェントが重要なのか

Claude Code と会話するたびに、メインのコンテキストウィンドウには情報が追加されていく。すべてのツール呼び出し、すべてのファイル読み取り、すべての検索結果がそこに保存される。この領域には上限があり、いっぱいになると Claude は会話の初期部分を追跡しづらくなる。

サブエージェントは、別のコンテキストウィンドウを起動することでこの問題を解決する。サブエージェントは次の 2 つを受け取る。

設定ファイルに書かれたカスタムシステムプロンプト。サブエージェントの役割と振る舞いを定義する。
親エージェントがユーザーの依頼内容をもとに書いたタスク説明。

その後、サブエージェントは単独で作業する。ファイルを読み、検索を実行し、コードを編集する。必要なことは何でも行う。作業が終わると、メイン会話に戻るのは要約だけである。サブエージェント内の会話全体はその後破棄される。

つまり、メインコンテキストはきれいなまま保たれる。答えを得るために通った道のりのノイズなしに、必要な回答だけを受け取れる。ただしトレードオフとして、サブエージェントがどのように結論に到達したかの可視性は失われる。

実践的な例

見慣れないコードベースを調査していて、返金処理を扱っているサービスを知りたいとする。サブエージェントを使わない場合、Claude は 15 個のファイルを読み、複数の検索を実行し、複数の関数呼び出しをたどるかもしれない。必要だったのは 1 つの事実だけなのに、そのすべてがコンテキストウィンドウを埋めてしまう。

サブエージェントを使うと体験はずっとすっきりする。質問をすると、Explore サブエージェントが起動し、独自のコンテキスト内で調査を行い、焦点の定まった回答を返す。

メインのコンテキストウィンドウに記録されるのは、質問と要約だけである。途中で読まれた 15 個のファイルは記録されない。

組み込みサブエージェント

Claude Code には、すぐに使える組み込みサブエージェントがいくつか同梱されている。

General purpose subagent -- 探索とアクションの両方を必要とする複数ステップのタスク向け。
Explore -- コードベースを高速に検索・ナビゲーションするためのもの。
Plan -- plan mode で使われ、計画を提示する前にコードベースを調査・分析する。

カスタムサブエージェント

組み込みオプションに加えて、カスタムシステムプロンプトとツールアクセスを持つ独自のサブエージェントを作成できる。これにより、ワークフローに合わせた専門エージェントを定義できる。コードレビュアー、テスト作成者、ドキュメント生成者など、必要なものを作れる。

重要なポイント

サブエージェントには主に 3 つの利点がある。

作業を焦点の定まった単位に分割し、各サブエージェントが特定のタスクに集中できる。
中間作業をすべて隔離することで、メインのコンテキストウィンドウをきれいに保てる。
必要な情報だけを簡潔な要約として持ち帰る。

組み込みサブエージェントを使う場合でも、独自に作成する場合でも、サブエージェントは長い Claude Code セッションをより有効に使うための実践的な方法である。メインコンテキスト内のノイズが少ないほど、より長く、より効果的に作業できる。

### レッスン 02: サブエージェントを作成する

**URL:** <https://anthropic.skilljar.com/introduction-to-subagents/450699>  

Claude Code には組み込みサブエージェントがあるが、自分で作成することもできる。カスタムサブエージェントは、コードレビュー、テスト作成、ドキュメント確認のような特定タスクに特化する。これらは YAML frontmatter を持つ Markdown ファイルとして定義され、Claude に対して、そのサブエージェントをいつ使うべきか、サブエージェントがどのように振る舞うべきかを伝える。

サブエージェントの作成

サブエージェントを作成する最も簡単な方法は `/agents` スラッシュコマンドを使うことだ。これにより、サブエージェントを管理するためのメインインターフェイスが開く。そこから `Create new agent` を選択する。

まず、サブエージェントのスコープを選ぶよう求められる。

Project-level -- 現在のプロジェクト内でのみ利用可能。
User-level -- 自分のマシン上のすべてのプロジェクトで共有される。

次に、作成方法を選べる。設定を手動で書くこともできるが、推奨される方法は Claude に生成させることだ。サブエージェントに何をしてほしいかを説明するだけで、Claude は入力に基づいて名前、説明、システムプロンプトを生成する。

ツールをカスタマイズする

作成時には、サブエージェントがアクセスできるツールをカスタマイズできる。ツールカテゴリには次のものが含まれる。

読み取り専用ツール。
編集ツール。
実行ツール。
MCP ツール。
その他のツール。

そのサブエージェントが実際に何を必要とするかを考える。コードレビュアーはおそらく編集ツールを必要としない。コードを読んで分析すべきであり、変更すべきではない。ただし、未コミットの変更をより簡単に特定できるよう、実行ツールは有効にしておきたい場合がある。

モデルと色を選ぶ

ツールを設定した後、そのサブエージェントを動かす Claude モデルを選択する。モデルピッカーには 4 つの選択肢がある。

Haiku -- 高速で軽量なタスクに最適。
Sonnet -- 速度と深さの中間をとるよい選択肢。
Opus -- 複雑な分析に最適。
Inherit -- メイン会話で実行されているモデルをそのまま使う。

最後に色を選ぶ。これは UI に表示され、どのサブエージェントがアクティブかをすばやく見分けられるようにする。小さな工夫だが、複数のサブエージェントを実行しているときに役立つ。

設定ファイル

作成が完了すると、サブエージェント設定ファイルはプロジェクト内に保存される。通常は `.claude/agents/your-agent-name.md` である。典型的なサブエージェント設定は次のようになる。

---

name: code-quality-reviewer
description: Use this agent when you need to review recently written or modified code for quality, security, and best practice compliance.
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch
model: sonnet
color: purple
---

あなたは品質保証、セキュリティのベストプラクティス、プロジェクト標準への準拠を専門とする熟練のコードレビュアーです。あなたの役割は、最近書かれた、または変更されたコードを徹底的に調べ、信頼性、セキュリティ、保守性、性能に影響し得る問題を特定することです。

各フィールドを分解して見ていく。

name -- サブエージェントの一意な識別子。Claude に直接依頼するとき、またはメッセージ内で `@agent code-quality-reviewer` と入力するときに、この名前で参照する。
description -- Claude がそのサブエージェントを使うかどうかを判断するタイミングを制御する。これは 1 行でなければならない。改行が必要な場合はエスケープされた改行文字 `\n` を使う。ここに会話例を含めることで、Claude がどのような場合に委任が適切かを理解しやすくできる。
tools -- サブエージェントがアクセスできるツールを列挙する。生成時に選択した内容と一致するが、このリストはいつでも編集できる。
model -- 使用する Claude モデルを指定する。`sonnet`、`opus`、`haiku`、`inherit` のいずれか。
color -- サブエージェントを識別するための UI 色。

システムプロンプト

Markdown ファイル本文、つまり YAML frontmatter より下のすべてがシステムプロンプトである。ここでサブエージェントに指示を与える。何に注目すべきか、どのように分析すべきか、どのように結果をメインエージェントへ報告すべきかを指定する。

よく書かれたシステムプロンプトは、有用なサブエージェントと要点を外すサブエージェントの差を生む。サブエージェントが何を探すべきか、出力をどのように構成すべきかを具体的に書く。

Claude に自動でサブエージェントを使わせる

明示的に頼まなくても Claude にタスクをサブエージェントへ委任させたい場合は、description フィールドに `"proactively"` という語を含める。たとえば次のように書く。

description: Proactively suggest running this agent after major code changes...

また、description に会話例を追加して、サブエージェントを使うべき具体的なシナリオを Claude に理解させることもできる。例が具体的であるほど、Claude はいつ委任すべきかをよりよく判断できる。

サブエージェントをテストする

サブエージェントを作成した後、いくつかコード変更を行い、Claude にレビューを依頼してテストする。

期待したときにサブエージェントが使われない場合は、description を見直す。より具体的な例やトリガーシナリオを追加すると、Claude がいつサブエージェントへ作業を委任すべきかを理解しやすくなる。

### レッスン 03: 効果的なサブエージェントを設計する

**URL:** <https://anthropic.skilljar.com/introduction-to-subagents/450700>  

サブエージェントの作り方が分かったので、次は実際に効果を出すためのパターンを見る。設定の悪いサブエージェントは迷走し、長く走りすぎ、メインエージェントが使えない出力を返す。修正の要点は 4 つある。よい description を書くこと、出力形式を定義すること、障害を報告すること、ツールアクセスを制限することである。

サブエージェント設定データがどのように使われるか

メインコンテキストウィンドウのエージェントへメッセージを送ると、利用可能なすべてのサブエージェントの name と description がシステムプロンプトに含まれる。メインエージェントはこれを使って、どのサブエージェントをいつ起動するかを決める。サブエージェントが自動的に起動されるタイミングをより細かく制御したい場合、調整すべきなのは name と description である。

description には 2 つ目の役割もある。メインエージェントがサブエージェントを起動するとき、タスクを開始するための入力プロンプトを書く。その入力プロンプトを書く際のガイダンスとして description を使う。つまり description は、サブエージェントがいつ実行されるかを制御するだけでなく、サブエージェントに何をするよう伝えるかの形も決める。

入力プロンプトを形作る description を書く

コードレビュー用サブエージェントを考える。一般的な description だけだと、メインエージェントは `"use get diff to find the current changes"` のような入力プロンプトを書くかもしれない。これは曖昧である。サブエージェントは、どのファイルが重要かを自分で判断しなければならない。

description に `"You must tell the agent precisely which files you want it to review"` のような文を含めると、メインエージェントはレビュー対象の実ファイルを列挙した、はるかに具体的な入力プロンプトを書くようになる。

同じ技法は、さまざまな種類のサブエージェントに使える。たとえば Web 検索サブエージェントの description に `"return sources that can be cited"` を追加すると、メインエージェントはタスク委任時にその指示を含めるようになる。

出力形式を定義する

サブエージェントに加えられる最も重要な改善は、システムプロンプト内で出力形式を定義することである。これには 2 つの効果がある。

自然な停止地点を作る。サブエージェントは、その形式の各セクションを埋め終わったときに作業完了だと分かる。
サブエージェントが長く走りすぎるのを防ぐ。定義された出力がないと、サブエージェントはどこまで調査すれば十分かを判断しにくく、不必要に長く実行されがちである。

コードレビュー用サブエージェントの構造化出力形式の例は次のとおりである。

レビューは構造化された形式で提供する。

1. Summary: レビューした内容と全体評価の短い概要。
2. Critical Issues: ただちに修正しなければならないセキュリティ脆弱性、データ整合性リスク、またはロジックエラー。
3. Major Issues: 品質上の問題、アーキテクチャとの不整合、または重要な性能上の懸念。
4. Minor Issues: スタイルの不整合、ドキュメント不足、または軽微な最適化。
5. Recommendations: 改善提案、リファクタリング機会、または適用すべきベストプラクティス。
6. Approval Status: そのコードが merge/deploy 可能か、変更が必要かを明確に述べる。

この形式により、サブエージェントは取り組むべき明確なチェックリストを得る。すべてのセクションが埋まると、サブエージェントは停止してよいと分かる。

障害を報告する

サブエージェントが作業中に回避策を見つけた場合、たとえば依存関係の問題を解決した、あるコマンドには特定のフラグが必要だと分かった、などの詳細は、返す要約に含める必要がある。そうしないと、メインスレッドは同じ解決策を自分で再発見しなければならず、時間とトークンを無駄にする。

表面化させたい情報には次のようなものがある。

セットアップ上の問題や環境固有の癖。
タスク中に見つけた回避策。
特別なフラグや設定が必要だったコマンド。
問題を引き起こした依存関係や import。

この情報を得る方法は、出力形式の中で明示的に求めることだ。出力テンプレートに `"Obstacles Encountered"` セクションを追加すると、この情報を信頼性高く表面化できる。

7. Obstacles Encountered: レビュー過程で遭遇した障害を報告する。これには、セットアップ上の問題、発見した回避策、環境固有の癖を含められる。特別なフラグや設定が必要だったコマンドを報告する。問題を引き起こした依存関係や import を報告する。

ツールアクセスを制限する

すべてのサブエージェントがすべてのツールへアクセスする必要はない。サブエージェントが実際に何をする必要があるかを考え、その仕事に必要なツールだけを与える。これには 2 つの効果がある。意図しない副作用を防ぎ、複数のサブエージェントがある場合に各サブエージェントの役割をより明確にする。

一般的なサブエージェント種別に対するツールアクセスの考え方は次のとおり。

Research / read-only subagent -- Glob、Grep、Read だけが必要。誤ってファイルを変更することがない。
Code reviewer -- 何が変わったかを見るために `git diff` を実行する Bash アクセスは必要だが、それでも Edit や Write は不要。
Styling / code modification agent -- ここでは Edit と Write アクセスを与える。サブエージェントの仕事が実際にコードを変更することだからである。

### レッスン 04: サブエージェントを効果的に使う

**URL:** <https://anthropic.skilljar.com/introduction-to-subagents/450701>  

サブエージェントの作成方法と、うまく設計する方法は分かった。では、実際にどんな場合に役立ち、どんな場合に邪魔になるのか。その違いは 1 つに集約される。中間作業がメインスレッドにとって重要かどうかである。

サブエージェントが力を発揮する場面

サブエージェントが最も有効なのは、探索と実行が分離されている場合である。タスクの各ステップが前のステップで発見した内容に依存するなら、その作業はメインスレッド内に置くべきである。しかし、単に答えが必要で、その答えに至る道筋を気にしないなら、委任するとよい。

サブエージェントは次のようなタスクで優れている。

必要なのは結果であり、それがどう見つかったかの逐一の説明ではない。
探索作業がメインスレッドのコンテキストを散らかしてしまう。
タスクが新鮮な視点やカスタムシステムプロンプトから恩恵を受ける。

調査タスク

調査は典型的なサブエージェントのユースケースである。見慣れないコードベースで認証がどのように動いているかを調べる場合を考える。メインスレッドが知りたいのは JWT がどこで検証されているかであり、途中で検索されたすべてのファイルを見る必要はない。

調査サブエージェントは、何十ものファイルを読み、関数呼び出しをたどり、さまざまなコードパスを探索できる。その探索はすべてサブエージェントのコンテキスト内にとどまる。メインスレッドが受け取るのは、次のようなきれいな要約である。

JWT validation happens in middleware/auth.js line 42,
called from the Express router in route/api.js

重い調査はサブエージェントが担った。メインスレッドは、前へ進むために必要なものだけを受け取る。

コードレビュー

Claude は、コードが他者によって書かれたものとして提示されると、より効果的にレビューできる。メインスレッドで何ターンもかけて機能を作った場合、その同じスレッドにレビューを依頼すると、フィードバックが弱くなりがちである。Claude 自身が作成に関わっていたため、新鮮な目で見るのが難しい。

レビュアーサブエージェントは、変更を別のコンテキストで見る。`git diff` を実行し、変更されたファイルを読み、コードがどのように書かれたかの履歴なしに、専門化されたレビュー基準を適用する。この分離により、プロジェクト固有のレビュー基準をサブエージェントのシステムプロンプトに埋め込めるため、チーム全体で一貫したレビュー基準を確保できる。

カスタムシステムプロンプト

Claude Code のデフォルトのシステムプロンプトは、簡潔でコード中心の応答を重視している。これはコーディングにはとてもよく機能するが、すべてに適しているわけではない。

カスタムシステムプロンプトによって、サブエージェントがメインスレッドより本当に優れたものになる 2 つのケースを示す。

Copywriting subagent -- トーン、読者、スタイルに関する指示を与える。Claude Code のデフォルトプロンプトは簡潔な技術文書寄りになりやすく、ランディングページやメールキャンペーンにはあまり向かない。コピーライティングサブエージェントには、声や構成についてまったく異なる指示を与えられる。
Styling subagent -- デザインシステムファイルを指し示す。サブエージェントが実行されると、それらのファイルが自動的にコンテキストへ読み込まれるため、CSS を書き始める前から、色変数、余白の規約、コンポーネントパターンを知っている状態になる。

サブエージェントが害になる場面

サブエージェントを起動するオーバーヘッド、つまり作業の可視性を失い、発見を要約へ圧縮するコストは、サブエージェントがメインスレッドにはできないことをする場合にのみ意味がある。注意すべき一般的なアンチパターンは 3 つある。

専門家を名乗るだけのもの

専門性を主張するサブエージェントは、ほとんど役に立たない。`"you are a Python expert"` や `"you are a Kubernetes specialist"` のようなプロンプトは価値を加えない。Claude はすでにその知識を持っているからである。いわゆる専門家サブエージェントにできて、メインスレッドが直接できないことはない。

逐次パイプライン

逐次的なサブエージェントパイプラインは問題を生む。バグを再現するエージェント、デバッグするエージェント、修正するエージェントの 3 つからなるフローを考える。パイプラインは、タスクが本当に独立している場合には機能する。しかし各ステップが前のステップの発見に依存する場合には失敗する。そしてバグ修正はほぼ常にそうである。エージェント間のハンドオフで情報が失われる。

テストランナー

テストランナーサブエージェントは、必要な情報を隠してしまいがちである。テストが失敗したときは、問題を診断するために完全な出力が必要になる。`"tests failed"` とだけ返すサブエージェントでは、直接出力なら見えていた詳細を得るために、追加のデバッグスクリプトを作る羽目になる。テストでは、テストランナーパターンがすべての構成の中でより悪い性能を示すことが分かっている。

判断ルール

サブエージェントを使うべきか判断するときは、1 つだけ質問する。中間作業は重要か。

答えが no なら、つまり最終結果だけが必要なら、サブエージェントへ委任する。答えが yes なら、つまり途中で何が起きているかを見て反応する必要があるなら、メインスレッドに置く。

サブエージェントを使うべきもの。

調査と探索。
コードレビュー。
カスタムシステムプロンプトを必要とするタスク。

サブエージェントを避けるべきもの。

実際の能力を追加しない「専門家」ペルソナ。
各ステップが直前のステップに依存する複数ステップのパイプライン。
デバッグのために完全な出力が必要なテスト実行。
