### レッスン 01: What are skills?（スキルとは何か）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434525>  

学べること

推定所要時間: 15 分

**このレッスンを終えると、次のことができるようになります:**

Claude Code のスキルとは何か、どのように動作するかを定義する
スキルがどこに置かれるか（個人用ディレクトリ vs. プロジェクト用ディレクトリ）を説明する
スキル、CLAUDE.md、スラッシュコマンドの違いを区別する
スキルが適切なカスタマイズツールとなるシナリオを見極める
スキルとは何か？

（3 分）

このビデオではスキルを紹介します。スキルとは、特定のタスクを自動的に処理する方法を Claude Code に教える再利用可能な Markdown ファイルです。PR をレビューしたり、コミットメッセージを書いたりするよう Claude に頼むたびに指示を繰り返す代わりに、スキルを一度書いておけば、そのタスクが発生するたびに Claude が適用してくれます。このビデオでは、スキルとは何か、どこに置かれるか、他の Claude Code カスタマイズオプションとどう比較されるかを扱います。

**主な要点**

スキルは命令を集めたフォルダであり、Claude Code がそれを発見してタスクをより正確に処理するために使用できます。各スキルは、フロントマターに name と description を持つ SKILL.md ファイルに存在します
Claude は description を使ってスキルをリクエストに照合します。何かを Claude に頼むと、Claude はあなたのリクエストを利用可能なスキルの description と比較し、一致するものを起動します
個人用スキルは ~/.claude/skills に置かれ、すべてのプロジェクトであなたについて回ります。プロジェクト用スキルはリポジトリ内の .claude/skills に置かれ、それをクローンする誰とでも共有されます
スキルはオンデマンドで読み込まれます。CLAUDE.md（すべての会話に読み込まれる）やスラッシュコマンド（明示的な呼び出しが必要）とは異なり、スキルは Claude が状況を認識すると自動的に起動します
同じことを Claude に繰り返し説明している自分に気づいたら、それは書かれるのを待っているスキルです

チームのコーディング規約を Claude に説明するたびに、あなたは同じことを繰り返しています。PR レビューのたびに、フィードバックをどう構成してほしいかを説明し直します。コミットメッセージのたびに、好みのフォーマットを Claude に思い出させます。スキルはこれを解決します。

スキルは、何かのやり方を Claude に一度だけ教える Markdown ファイルです。Claude はその後、それが関連するときはいつでもその知識を自動的に適用します。

スキルとは何か

スキルは命令とリソースを集めたフォルダであり、Claude Code がそれを発見してタスクをより正確に処理するために使用できます。各スキルは、フロントマターに name と description を持つ SKILL.md ファイルに存在します。

description は、Claude がそのスキルを使うかどうかを判断する手がかりです。PR をレビューするよう Claude に頼むと、Claude はあなたのリクエストを利用可能なスキルの description に照合し、関連するものを見つけます。Claude はあなたのリクエストを読み、利用可能なすべてのスキルの description と比較し、一致するものを起動します。

スキルのフロントマターは次のようになります:

---

name: pr-review
description: Reviews pull requests for code quality. Use when reviewing PRs or checking code changes
---

フロントマターの下には、実際の命令を書きます。レビューのチェックリスト、フォーマットの好み、あるいはそのタスクのために Claude が知る必要のあるものは何でもです。

スキルはどこに置かれるか

誰が必要とするかに応じて、スキルを異なる場所に保存できます:

個人用スキルは ~/.claude/skills（ホームディレクトリ）に置かれます。これらはすべてのプロジェクトであなたについて回ります。コミットメッセージのスタイル、ドキュメントのフォーマット、コードの説明のされ方の好みなどです。
プロジェクト用スキルはリポジトリのルートディレクトリ内の .claude/skills に置かれます。リポジトリをクローンする誰もがこれらのスキルを自動的に取得します。ここにはチームの標準が存在します。たとえば、会社のブランドガイドライン、Web デザイン用の好みのフォントや色などです。

Windows では、個人用スキルは C:/Users/<your-user>/.claude/skills に存在します。

プロジェクト用スキルはコードとともにバージョン管理にコミットされるため、チーム全体で共有されます。

スキル vs. CLAUDE.md vs. スラッシュコマンド

Claude Code には動作をカスタマイズする方法がいくつかあります。スキルが独特なのは、自動的でタスク固有である点です。比較は次のとおりです:

CLAUDE.md ファイルはすべての会話に読み込まれます。Claude に常に TypeScript の strict モードを使ってほしいなら、それは CLAUDE.md に入れます。
スキルはリクエストに一致したときにオンデマンドで読み込まれます。Claude は最初は name と description のみを読み込むため、コンテキストウィンドウ全体を埋め尽くすことはありません。PR レビューのチェックリストは、デバッグ中はコンテキストに入っている必要はなく、実際にレビューを頼んだときに読み込まれます。
スラッシュコマンドは明示的に入力する必要があります。スキルはそうではありません。Claude は状況を認識したときにそれらを適用します。

Claude がスキルをあなたのリクエストに一致させると、ターミナルで読み込まれるのが見えます:

スキルをいつ使うか

スキルは、特定のタスクに適用される専門知識に最も適しています:

チームが従うコードレビューの標準
好みのコミットメッセージのフォーマット
組織のブランドガイドライン
特定の種類のドキュメント用のドキュメントテンプレート
特定のフレームワーク用のデバッグチェックリスト

経験則はシンプルです。同じことを Claude に繰り返し説明している自分に気づいたら、それは書かれるのを待っているスキルです。

### レッスン 02: Creating your first skill（最初のスキルを作る）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434527>  

学べること

推定所要時間: 20 分

**このレッスンを終えると、次のことができるようになります:**

適切なフロントマター構造を持つスキルをゼロから作成する
スキルが Claude Code で正しく読み込まれることをテストして検証する
Claude Code が受信したリクエストを利用可能なスキルにどう照合するかを説明する
スキルの優先順位の階層（Enterprise、Personal、Project、Plugins）を説明する
最初のスキルを作る

（4 分）

このビデオでは、スキルをゼロから構築する流れを示します。すべてのプロジェクトで動作する個人用の PR 説明スキルです。SKILL.md ファイルの構成方法、テストの仕方、Claude Code がスキルをどう発見してリクエストに照合するかを正確に確認できます。また、名前が衝突したときにどのスキルが勝つかを決める優先順位の階層も扱います。

**主な要点**

スキルは、フロントマターにメタデータ（name、description）を持ち、その下に命令を持つ SKILL.md ファイルを含むディレクトリです
Claude は起動時にスキルの name と description のみを読み込み、その後セマンティックマッチングを使って受信リクエストをそれらの description に照合します
Claude がスキルの全内容をコンテキストに読み込む前に、確認のプロンプトが表示されます
名前衝突時の優先順位: Enterprise → Personal → Project → Plugins
スキルを更新するには、その SKILL.md を編集します。削除するには、そのディレクトリを削除します。変更を反映させるには常に Claude Code を再起動してください

スキルをゼロから作成する流れを追い、その後で Claude Code が実際にどのようにスキルを読み込んで照合しているかを舞台裏で見てみましょう。

スキルを作成する

PR 説明を一貫したフォーマットで書く方法を Claude に教える個人用スキルを構築します。個人用スキルなので、ホームディレクトリに存在し、すべてのプロジェクトで動作します。

まず、skills フォルダ内にスキル用のディレクトリを作成します。ディレクトリ名はスキル名と一致させるべきです:

mkdir -p ~/.claude/skills/pr-description

次に、そのディレクトリ内に SKILL.md ファイルを作成します。ファイルはフロントマターのダッシュで区切られた 2 つの部分から成ります:

---

name: pr-description
description: Writes pull request descriptions. Use when creating a PR, writing a PR, or when the user asks to summarize changes for a pull request
---

When writing a PR description:

1. Run `git diff main...HEAD` to see all changes on this branch
2. Write a description following this format:

## What

One sentence explaining what this PR does.

## Why

Brief context on why this change is needed

## Changes

- Bullet points of specific changes made
- Group related changes together
- Mention any files deleted or renamed

name はスキルを識別します。description は Claude にいつ使うべきかを伝えます。これが照合の基準です。2 番目のダッシュのセット以降のすべてが、スキルが起動したときに Claude が従う命令です。

スキルをテストする

Claude Code は起動時にスキルを読み込むため、作成後はセッションを再起動してください。利用可能なスキルのリストを確認することで、それが利用可能であることを検証できます。

あなたのスキルがリストに表示されるはずです。テストするには、ブランチでいくつか変更を加え、「write a PR description for my changes」のように言ってみてください。Claude は PR 説明スキルを使っていることを示し、diff を確認し、あなたのテンプレートに従って説明を書きます。毎回同じフォーマットです。

スキルのマッチングの仕組み

Claude Code は起動すると、スキルを 4 つの場所でスキャンしますが、name と description のみを読み込み、全内容は読み込みません。これは重要な詳細です。

リクエストを送信すると、Claude はあなたのメッセージを利用可能なすべてのスキルの description と比較します。たとえば、「explain what this function does」は、意図が重なるため「explain code with visual diagrams」と説明されたスキルに一致します。

一致が見つかると、Claude はスキルを読み込むことの確認を求めます。この確認ステップによって、Claude がどんなコンテキストを取り込んでいるかをあなたが把握し続けられます。確認すると、Claude は完全な SKILL.md ファイルを読み、その命令に従います。

スキルの優先順位

個人用スキルのひとつと同じ名前のスキルを持つリポジトリをクローンした場合、どちらが勝つでしょうか？明確な優先順位があります:

Enterprise — 管理された設定、最高優先度
Personal — ホームディレクトリ（~/.claude/skills）
Project — リポジトリ内の .claude/skills ディレクトリ
Plugins — インストールされたプラグイン、最低優先度

これにより、組織はエンタープライズスキルを通じて標準を強制しつつ、個々のカスタマイズも許容できます。会社にエンタープライズの「code-review」スキルがあり、あなたが同じ名前の個人用「code-review」スキルを作成した場合、エンタープライズ版が優先されます。

衝突を避けるには、説明的な名前を使ってください。単に「review」ではなく、「frontend-review」や「backend-review」のようなものを使いましょう。

スキルの更新と削除

スキルを更新するには、その SKILL.md ファイルを編集します。削除するには、そのディレクトリを削除します。変更後はそれを反映させるために Claude Code を再起動してください。

レッスンの振り返り
日々のワークフローの中で、今すぐスキルに変えられるタスクはひとつ何でしょうか？その description はどのようになるでしょうか？
優先順位の階層はあなたのチームにどう影響するでしょうか

### レッスン 03: Configuration and multi-file skills（設定とマルチファイルスキル）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434526>  

学べること

推定所要時間: 20 分

**このレッスンを終えると、次のことができるようになります:**

allowed-tools や model を含む高度なスキルメタデータフィールドを設定する
適切なリクエストで確実にトリガーされる効果的なスキル description を書く
スキルがアクティブなときに Claude ができることを制限するために allowed-tools を使う
プログレッシブディスクロージャーとマルチファイル構造を使って複雑なスキルを整理する
設定とマルチファイルスキル

（4 分）

このビデオは、スキルをより強力にする高度なテクニックを扱います。メタデータフィールドの完全なセット、確実にトリガーされる description の書き方、セキュリティに敏感なワークフローのためのツールアクセス制限、そしてプログレッシブディスクロージャーを使って大きなスキルを複数ファイルにわたって整理する方法です。複雑なユースケースをサポートしつつ、スキルを効率的に保つ方法を学びます。

**主な要点**

name と description は必須です。allowed-tools と model はオプションですが強力な追加です
良い description は 2 つの質問に答えます: そのスキルは何をするか？ Claude はいつそれを使うべきか？
allowed-tools は、スキルがアクティブなときに Claude が使えるツールを制限します。読み取り専用やセキュリティに敏感なワークフローに役立ちます
プログレッシブディスクロージャー: SKILL.md を 500 行未満に保ち、Claude が必要なときだけ読む補助ファイル（references、scripts、assets）にリンクします
スクリプトは内容をコンテキストに読み込むことなく実行されます。出力のみがトークンを消費するため、コンテキストを効率的に保ちます

基本的なスキルは name と description だけで動作しますが、Claude Code でスキルをはるかに効果的にできる高度なテクニックがいくつかあります。主要なフィールド、description のベストプラクティス、ツールの制限、そして大きなスキルの構造化方法を順に見ていきましょう。

スキルのメタデータフィールド

エージェントスキルのオープン標準は、SKILL.md のフロントマターでいくつかのフィールドをサポートします。2 つは必須で、残りはオプションです:

name（必須）— スキルを識別します。小文字、数字、ハイフンのみを使います。最大 64 文字。ディレクトリ名と一致させるべきです。
description（必須）— Claude にいつスキルを使うべきかを伝えます。最大 1,024 文字。Claude がマッチングに使うため、これが最も重要なフィールドです。
allowed-tools（オプション）— スキルがアクティブなときに Claude が使えるツールを制限します。
model（オプション）— スキルに使う Claude モデルを指定します。
効果的な description を書く

指示は明確にしてください。誰かに「あなたの仕事はドキュメントを手伝うことです」と言われても、何をすべきか分かりません。Claude も同じように考えます。

良い description は 2 つの質問に答えます:

そのスキルは何をするか？
Claude はいつそれを使うべきか？

期待したときにスキルがトリガーされない場合は、実際にリクエストを言い表す方法に一致するキーワードを追加してみてください。description は Claude がスキルが関連するかどうかを判断するために使うものなので、言葉づかいが重要です。

allowed-tools でツールを制限する

ファイルを変更せず、読み取りだけができるスキルが欲しいことがあります。これはセキュリティに敏感なワークフロー、読み取り専用のタスク、あるいはガードレールが欲しいあらゆる状況で役立ちます。

この例では、allowed-tools フィールドが Read、Grep、Glob、Bash に設定されています。このスキルがアクティブなとき、Claude は許可を求めずにそれらのツールのみを使えます。編集も書き込みもありません。

---

name: codebase-onboarding
description: Helps new developers understand the system works.
allowed-tools: Read, Grep, Glob, Bash
model: sonnet
---

allowed-tools を完全に省略すると、スキルは何も制限しません。Claude は通常の権限モデルを使います。

プログレッシブディスクロージャー

スキルは Claude のコンテキストウィンドウを会話と共有します。Claude がスキルを起動すると、その SKILL.md の内容をコンテキストに読み込みます。しかし、スキルが依存する参考資料、例、ユーティリティスクリプトが必要なことがあります。

すべてを 1 つの 2,000 行のファイルに詰め込むと、2 つの問題があります。コンテキストウィンドウの領域を大量に占有し、メンテナンスが楽しくありません。

プログレッシブディスクロージャーがこれを解決します。必須の命令を SKILL.md に保ち、詳細な参考資料は Claude が必要なときだけ読む別ファイルに置きます。

オープン標準は、スキルディレクトリを次のように整理することを提案します:

scripts/ — 実行可能なコード
references/ — 追加のドキュメント
assets/ — 画像、テンプレート、その他のデータファイル

そして SKILL.md の中で、いつ読み込むかについての明確な指示とともに補助ファイルへリンクします:

この例では、Claude は誰かがシステム設計について尋ねたときだけ architecture-guide.md を読みます。コンポーネントをどこに追加するかを尋ねている場合は、そのファイルは決して読み込まれません。これは、ドキュメント全体ではなく目次をコンテキストウィンドウに持つようなものです。

良い経験則: SKILL.md を 500 行未満に保ちます。それを超えている場合は、内容を別の参考ファイルに分割すべきかどうかを検討してください。

### レッスン 04: Skills vs. other Claude Code features（スキル vs. 他の Claude Code 機能）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434528>  

学べること

推定所要時間: 15 分

**このレッスンを終えると、次のことができるようになります:**

スキルを CLAUDE.md、サブエージェント、フック、MCP サーバーと比較する
与えられたユースケースに適した Claude Code のカスタマイズ機能を選ぶ
複数の機能を効果的に組み合わせる補完的なセットアップを設計する
スキル vs. 他の Claude Code 機能

（3 分）

Claude Code はいくつかのカスタマイズオプションを提供しており、間違ったものを選ぶと不必要な複雑さにつながることがあります。このビデオは、スキルを CLAUDE.md、サブエージェント、フック、MCP サーバーと比べていつ使うべきかを分解します。各オプションの主な違いと、典型的な開発セットアップでそれらがどう補完し合うかを学びます。

**主な要点**

CLAUDE.md はすべての会話に読み込まれ、常時オンのプロジェクト標準に最適です。スキルはオンデマンドで読み込まれ、タスク固有の専門知識に最適です
サブエージェントは隔離された実行コンテキストで動作します。委譲する作業に使ってください。スキルは現在の会話に知識を追加します
フックはイベント駆動です（ファイル保存やツール呼び出しで発火）。スキルはリクエスト駆動です（あなたが尋ねている内容に基づいて起動）
MCP サーバーは外部ツールと統合を提供します。スキルとは全く別のカテゴリです
各機能はそれぞれの専門分野を扱います。すべてを 1 つのアプローチに押し込むのではなく、それらを組み合わせてください

Claude Code はいくつかのカスタマイズオプションを提供します: スキル、CLAUDE.md、サブエージェント、フック、MCP サーバーです。これらは異なる問題を解決し、いつどれを使うかを知ることで間違ったものを構築せずに済みます。それらを分解してみましょう。

CLAUDE.md vs スキル

CLAUDE.md は常にすべての会話に読み込まれます。Claude にプロジェクトで TypeScript の strict モードを使ってほしいなら、CLAUDE.md ファイルに入れてください。

スキルはオンデマンドで読み込まれます。Claude がリクエストをスキルに一致させると、そのスキルの命令が会話に加わります。PR レビューのチェックリストは、新しいコードを書いているときにコンテキストに入っている必要はなく、レビューを頼んだときに起動します。

CLAUDE.md を使う場合:

常に適用されるプロジェクト全体の標準
「データベーススキーマを決して変更しない」のような制約
フレームワークの好みとコーディングスタイル

スキルを使う場合:

タスク固有の専門知識
時々だけ関連する知識
すべての会話を散らかしてしまう詳細な手順
スキル vs サブエージェント

スキルは現在の会話に知識を追加します。スキルが起動すると、その命令が既存のコンテキストに加わります。

サブエージェントは別のコンテキストで動作します。タスクを受け取り、独立して作業し、結果を返します。メインの会話から隔離されています。

サブエージェントを使う場合:

タスクを別の実行コンテキストに委譲したい
メインの会話とは異なるツールアクセスが必要
委譲した作業とメインのコンテキストの間に隔離が欲しい

スキルを使う場合:

現在のタスクのために Claude の知識を強化したい
その専門知識が会話を通じて適用される
スキル vs フック

フックはイベントで発火します。フックは Claude がファイルを保存するたびにリンターを実行したり、特定のツール呼び出しの前に入力を検証したりするかもしれません。これらはイベント駆動です。

スキルはリクエスト駆動です。あなたが尋ねている内容に基づいて起動します。

フックを使う場合:

すべてのファイル保存時に実行すべき操作
特定のツール呼び出しの前の検証
Claude のアクションの自動化された副作用

スキルを使う場合:

Claude がリクエストをどう処理するかを左右する知識
Claude の推論に影響するガイドライン
すべてをまとめる

典型的なセットアップには次のものが含まれるかもしれません:

CLAUDE.md — 常時オンのプロジェクト標準
スキル — オンデマンドで読み込まれるタスク固有の専門知識
フック — イベントによってトリガーされる自動化された操作
サブエージェント — 委譲された作業のための隔離された実行コンテキスト
MCP サーバー — 外部ツールと統合

それぞれがそれぞれの専門分野を扱います。別のオプションのほうが適しているときにすべてをスキルに押し込まないでください。また、一度に複数を使うこともできます。スキルは自動的なタスク固有の専門知識を提供し、CLAUDE.md は常時オンの指示用、サブエージェントは隔離されたコンテキストで動作し、フックはイベントで発火し、MCP は外部ツールを提供します。

トピックが関連するときに Claude が自動的に適用すべき知識がある場合はスキルを使い、包括的なカスタマイズのために他の機能と組み合わせてください。

レッスンの振り返り
現在の CLAUDE.md ファイルを見てください。スキルとして（関連するときだけ読み込まれる形で）使ったほうが良いものはありますか？
チームの開発ワークフローを考えてください。Claude Code 機能（スキル、フック、サブエージェント、MCP）のどの組み合わせが、最もよくある悩みの種に対処するでしょうか？
次は何か

次のレッスンでは、スキルをチームや組織と共有する方法を学びます。リポジトリへのコミットから、プラグインによる配布、管理された設定を通じた組織全体への展開までです。

### レッスン 05: Sharing skills（スキルを共有する）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434529>  

学べること

推定所要時間: 20 分

**このレッスンを終えると、次のことができるようになります:**

スキルを Git リポジトリにコミットしてチームと共有する
プラグインとマーケットプレイスを通じてプロジェクト間でスキルを配布する
エンタープライズの管理された設定を使ってスキルを組織全体に展開する
特定のスキルを使うようカスタムサブエージェントを設定する
スキルを共有する

（4 分）

スキルはチームや組織で共有されると、はるかに価値が高まります。このビデオは、3 つの主要な配布方法（リポジトリへのコミット、プラグイン、エンタープライズの管理された設定）を扱い、スキルを使うようカスタムサブエージェントを設定する方法を説明します。どのアプローチがどのシナリオに合うか、そして重要な落とし穴（サブエージェントはスキルを自動的に継承しない）への対処法を学びます。

**主な要点**

.claude/skills 内のプロジェクトスキルは Git を通じて自動的に共有されます。リポジトリをクローンする誰もがそれらを取得します
プラグインを使うと、マーケットプレイスを通じてリポジトリ間でスキルを配布し、より広いコミュニティでの利用が可能になります
エンタープライズの管理された設定はスキルを最高優先度で組織全体に展開し、必須の標準やコンプライアンスに理想的です
サブエージェントはあなたのスキルを自動的には見ません。カスタムエージェントのフロントマターの skills フィールドにスキルを明示的にリストする必要があります
組み込みエージェント（Explorer、Plan、Verify）はスキルに全くアクセスできません。.claude/agents で定義されたカスタムサブエージェントのみがアクセスできます

スキルは共有されると、はるかに価値が高まります。あなただけが使う PR レビュースキルは役に立ちますが、同じスキルをチーム全体で共有すれば、コードレビューを標準化し、組織全体で一貫した体験を生み出します。スキルを配布するさまざまな方法を見てみましょう。

スキルをリポジトリにコミットする

最もシンプルな共有方法は、スキルをリポジトリに直接コミットすることです。それらを .claude/skills に置けば、リポジトリをクローンする誰もがそれらのスキルを自動的に取得します。追加のインストールは不要です。

更新をプッシュすると、全員が次の pull で取得します。このアプローチは次に適しています:

チームのコーディング標準
プロジェクト固有のワークフロー
コードベースの構造を参照するスキル

.claude ディレクトリには、エージェント、フック、スキル、設定が含まれます。すべて通常の Git ワークフローを通じてバージョン管理され、チームと共有されます。

プラグインを通じてスキルを配布する

プラグインは、チームやプロジェクト間で共有されるよう設計されたカスタム機能で Claude Code を拡張する方法です。プラグインプロジェクトで、.claude ディレクトリと似たファイル構造に従う skills ディレクトリを作成します。各スキルは、内部に SKILL.md ファイルを持つ独自のフォルダを取得します。

プラグインをマーケットプレイスに配布すると、他のユーザーがそれを発見し、自分の Claude Code にインストールできます。

このアプローチは、スキルがあまりプロジェクト固有でなく、直近のチームを超えてコミュニティのメンバーに役立つ場合に最適です。

管理された設定を通じたエンタープライズ展開

管理者は管理された設定を通じてスキルを組織全体に展開できます。エンタープライズスキルは最高優先度を持ちます。同じ名前の個人、プロジェクト、プラグインのスキルを上書きします。

管理された設定ファイルは、プラグインをどこからインストールできるかを制御する strictKnownMarketplaces のような機能をサポートします:

"strictKnownMarketplaces": [
{
"source": "github",
"repo": "acme-corp/approved-plugins"
},
{
"source": "npm",
"package": "@acme-corp/compliance-plugins"
}
]

これは、必須の標準、セキュリティ要件、コンプライアンスワークフロー、そして組織全体で一貫している必要のあるコーディング慣行に適した選択肢です。ここでのキーワードは「必須（must）」です。

スキルとサブエージェント

人々を驚かせることがあります。サブエージェントはあなたのスキルを自動的には見ません。タスクをサブエージェントに委譲すると、それは新しいクリーンなコンテキストで始まります。

理解すべき重要な区別があります:

組み込みエージェント（Explorer、Plan、Verify など）はスキルに全くアクセスできません
あなたが定義するカスタムサブエージェントはスキルを使えますが、明示的にリストした場合のみです
スキルはサブエージェントが起動するときに読み込まれます。メインの会話のようにオンデマンドではありません

スキルを持つカスタムサブエージェントを作成するには、.claude/agents にエージェントの Markdown ファイルを追加します。Claude Code の /agents コマンドを使って対話的に作成できます:

生成されるエージェントファイルには、どのスキルを読み込むかをリストする skills フィールドが含まれます。フロントマターは次のようになります:

---

name: frontend-security-accessibility-reviewer
description: "Use this agent when you need to review frontend code for accessibility..."
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch, Skill...
model: sonnet
color: blue
skills: accessibility-audit, performance-check
---

このサブエージェントに委譲すると、両方のスキルが読み込まれた状態になり、すべてのレビューにそれらを適用します。まずスキルが .claude/skills ディレクトリに存在することを確認し、それから新しいサブエージェントを作成するか、既存のエージェントの Markdown ファイルに skills フィールドを追加してください。

このパターンは次の場合に非常にうまく機能します:

特定の専門知識を持つ隔離されたタスク委譲が欲しい

### レッスン 06: Troubleshooting skills（スキルのトラブルシューティング）

**URL:** <https://anthropic.skilljar.com/introduction-to-agent-skills/434530>  

学べること

推定所要時間: 15 分

**このレッスンを終えると、次のことができるようになります:**

スキルバリデーターを使って、デバッグ前に構造上の問題を捉える
よくあるスキルのトリガーと読み込みの問題を診断して修正する
エンタープライズ、個人、プロジェクト、プラグインのスキル間の優先順位の衝突を解決する
依存関係の欠如、権限、パスの問題を含むランタイムエラーをデバッグする
スキルのトラブルシューティング

（4 分）

スキルが期待どおりに動作しないとき、問題はたいてい予測可能ないくつかのカテゴリーに分類されます。このビデオは、それぞれを順に扱います。トリガーされないスキルから、優先順位の衝突、ランタイムの失敗までです。そして、体系的なトラブルシューティングのアプローチを示します。また、スキルバリデーターツールについて、そして読み込みの問題を診断するための claude --debug の使い方も学びます。

**主な要点**

まずスキルバリデーターツールから始めてください。他のことをデバッグする時間を費やす前に、構造上の問題を捉えます
スキルがトリガーされない場合、原因はほぼ常に description です。実際にリクエストを言い表す方法に一致するトリガーフレーズを追加してください
スキルが読み込まれない場合、SKILL.md が名前付きディレクトリ内（skills のルートではない）にあること、そしてファイル名が正確に SKILL.md であることを確認してください
間違ったスキルが使われる場合、description が似すぎています。より明確に区別してください
ランタイムエラーには、依存関係、ファイル権限（chmod +x）、パス区切り文字（どこでもスラッシュを使う）を確認してください

スキルが動作しないとき、問題はたいていいくつかのカテゴリーのひとつに分類されます。スキルがトリガーされない、読み込まれない、衝突がある、ランタイムで失敗する、です。良いニュースは、ほとんどの修正がかなり単純であることです。

スキルバリデーターを使う

最初に試すべきは、エージェントスキルの verifier コマンドです。インストール手順はオペレーティングシステムによって異なりますが、uv を使うのが最も簡単に素早くセットアップする方法です。

インストールしたら、スキルディレクトリに移動するか、どこからでもコマンドを実行します。バリデーターは、他のことをデバッグする時間を費やす前に構造上の問題を捉えます。

スキルがトリガーされない

スキルは存在し、検証も通るのに、期待したときに Claude がそれを使いません。原因はほぼ常に description です。

Claude はセマンティックマッチングを使うため、あなたのリクエストが description の意味と重なる必要があります。重なりが十分でなければ、一致しません。やるべきことは次のとおりです:

実際にリクエストを言い表している方法に対して description を確認する
ユーザーが実際に言うであろうトリガーフレーズを追加する
「help me profile this」「why is this slow?」「make this faster」のようなバリエーションでテストする
いずれかのバリエーションがトリガーに失敗したら、それらのキーワードを description に追加する
スキルが読み込まれない

「what skills are available」と Claude に尋ねたときにスキルが表示されない場合は、これらの構造上の要件を確認してください:

SKILL.md ファイルは名前付きディレクトリ内になければなりません。skills のルートではありません
ファイル名は正確に SKILL.md でなければなりません。「SKILL」はすべて大文字、「md」は小文字です

claude --debug を実行して読み込みエラーを確認してください。あなたのスキル名に言及するメッセージを探します。これだけで問題に直接つながることもあります。

間違ったスキルが使われる

Claude が間違ったスキルを使ったり、スキル間で混乱しているように見える場合、おそらく description が似すぎています。明確に区別してください。できるだけ具体的にすることは、Claude がスキルをいつ使うか判断する助けになるだけでなく、似た響きの他のスキルとの衝突も防ぎます。

スキルの優先順位の衝突

個人用スキルが無視されている場合、エンタープライズまたはより高優先度のスキルが同じ名前を持っているかもしれません。

たとえば、エンタープライズの「code-review」スキルがあり、あなたも個人用の「code-review」スキルを持っている場合、エンタープライズのものが毎回勝ちます。あなたの選択肢は:

スキルをより明確に区別できる名前に変更する（通常はこちらのほうが簡単な道です）
エンタープライズスキルについて管理者と話す
プラグインのスキルが表示されない

プラグインをインストールしたのにスキルが見えない？キャッシュをクリアし、Claude Code を再起動し、再インストールしてください。

それでもスキルが表示されない場合、プラグインの構造が間違っているかもしれません。こういうときこそバリデーターツールが真価を発揮します。

ランタイムエラー

スキルは読み込まれるが、実行中に失敗します。よくある原因がいくつかあります:

依存関係の欠如: スキルが外部パッケージを使う場合、それらがインストールされている必要があります。何が必要かを Claude が知れるよう、依存関係の情報をスキルの description に追加してください。
権限の問題: スクリプトには実行権限が必要です。スキルが参照するスクリプトに対して chmod +x を実行してください。
パス区切り文字: Windows でもどこでもスラッシュを使ってください。
クイックトラブルシューティングチェックリスト
トリガーされない？ description を改善し、トリガーフレーズを追加する。
読み込まれない？ パス、ファイル名、YAML 構文を確認する。
間違ったスキルが使われる？ description を互いにより明確に区別する。
シャドウされている？ 優先順位の階層を確認し、必要なら名前を変更する。
プラグインのスキルが見えない？ キャッシュをクリアして再インストールする。
ランタイムの失敗？ 依存関係、権限、パスを確認する。
