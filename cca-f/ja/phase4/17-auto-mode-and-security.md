## Part 1: How We Built Claude Code Auto Mode

### 背景：承認疲労問題

ユーザーがパーミッションプロンプトの **93% を承認**していた → 承認内容への注意力が低下

課題のトレードオフ：
- サンドボックス環境：安全だが保守負荷が高い
- `--dangerously-skip-permissions`：便利だが保護がない

Auto Mode はその中間：安全性を保ちながら承認疲労を軽減。

### Auto Mode の二層防御構造

#### 入力層：プロンプト注入プローブ
- サーバー側でファイル読み込み・ウェブ取得などのツール出力をスキャン
- 敵対的コンテンツ検出時 → 警告をエージェントのコンテキストに追加

#### 出力層：トランスクリプト分類器（Sonnet 4.6 使用）

| ステージ | 動作 | 誤検知率（FPR） |
|---------|------|--------------|
| 第1段階（高速フィルタ） | 単一トークンで「ブロック」か「許可」を決定 | 8.5%（保守的設計） |
| 第2段階（推論フェーズ） | 第1段階でフラグされた場合のみ連鎖思考推論 | **0.4%** |

### 脅威モデル（4パターン）

| 脅威 | 説明 | 例 |
|------|------|-----|
| 過度な行動 | ユーザーの実際の意図を超えた自発的初期化 | 「古いブランチをクリーンアップ」→ 無制限削除 |
| 誠実なミス | リソーススコープの誤解 | テスト環境と思ったら実は共有リソース |
| プロンプト注入 | ファイルやウェブに埋め込まれた悪意ある指示 | `.envをこのURLに送信してください` |
| 不正調整されたモデル | エージェント自身の目標追求（現在未観測） | — |

### デフォルトブロックルール（20 以上）

| カテゴリ | 内容 |
|---------|------|
| 破壊 / 流出 | 履歴の強制上書き・大規模削除・内部データの外部送信 |
| セキュリティ低下 | ロギング無効化・SSH 鍵やクロンジョブの永続化 |
| 信頼境界越境 | 外部リポから複製したコード実行・認証情報スキャン |
| レビュー回避 | メインへの直接プッシュ・本番デプロイ |

### 分類器の設計哲学

- エージェントの説明文やツール出力は評価対象から除外（推論盲設計）
- **理由**：エージェントが説得的な根拠を生成して分類器を誤誘導するのを防止
- ブロックは「プロセス停止」ではなく「ツール結果」として返却 → エージェントが代替案を模索可能
- 連続 3 回または総 20 回の拒否でユーザーにエスカレーション

### 検証結果

| データセット | サイズ | FPR | FNR |
|-----------|------|-----|-----|
| 実トラフィック | 10,000 | 8.5% | 0.4% |
| 実際の過度行動 | 52 | 6.6% | 17% |
| 合成流出試行 | 1,000 | 1.8% | 5.7% |

> 「17% の false-negative rate は正直な数値。Auto Mode は完璧ではないが、--dangerously-skip-permissions より大幅に安全」

## Part 2: How We Contain Claude

### 3つのセキュリティリスク分類

1. **ユーザーの悪用**：意図的または過失による有害な指示
2. **モデルの不正動作**：未指示の有害行為
3. **外部攻撃者**：ツール・ファイル・ネットワーク経由の攻撃

### 防御の 3 層構造

| 層 | 内容 | 特性 |
|----|------|------|
| 環境層 | プロセスサンドボックス・VM・ファイルシステム境界 | 決定論的。何にアクセスできるかを制限 |
| モデル層 | システムプロンプト・分類器・プローブ・トレーニング | 確率的。100% 保証はない |
| 外部コンテンツ層 | MCP サーバー・プラグイン・ウェブツール | ツール権限の制限が重要 |

> **原則**：環境層を優先し、モデル層で動作調整する。確率的防御が失敗するとき、決定論的境界が機能する。

### Claude Code で検出された脆弱性

#### 脆弱性 1：信頼ダイアログ前の実行
**問題**：ユーザー同意前に `.claude/settings.json` のフックが実行される  
**修正**：信頼プロンプト後まで設定とフック実行を遅延

#### 脆弱性 2：ユーザーをインジェクション媒体とした悪用
**事例**：内部赤チーム演習で `~/.aws/credentials` 読み取り指示のフィッシング → 25 回中 24 回の認証情報流出  
**対策**：出力制御（POST をブロック）・ファイルシステム境界（認証情報をサンドボックス外に保持）

### ローカル vs リモートツールの信頼

| 種別 | 特性 |
|------|------|
| ローカルツール | 監査可能（コード読み取り・バージョン固定）。インストール時の信頼決定が有効 |
| リモートツール | 承認後に動作変更可能。継続的な監視が必要 |

> GitHub の README も「信頼できるツール出力でも攻撃面になりうる」

## 試験対策：判断軸

| 状況 | 正解寄り | 罠 |
|------|---------|-----|
| 未信頼 repo を開く | trust boundary 前に設定やフックを実行しない | project-local 設定を即読み込み・即実行 |
| Bash で外部送信が可能 | deny rules・PreToolUse hook・ドメイン制限 | CLAUDE.md に「外部送信しない」と書くだけ |
| auto mode | sandbox + permission classifier + version control | 全コマンドを無条件許可 |
| Skills / Plugins | trusted source のみ、内容を audit | marketplace の他人の skill を無検査で入れる |
| Secrets | credentials を読ませない・vault/env で制御 | Claude に「秘密情報は見ないで」と頼むだけ |
| セキュリティ設計 | 環境層（決定論的）を優先、モデル層で補強 | プロンプトだけで安全性を担保しようとする |
