## Eval（評価）の基本構造

**Eval** = AI システムへ入力を与え、出力に採点ロジックを適用して成功を測定するテスト

| 種別 | 説明 |
|------|------|
| 単一ターン評価 | プロンプト → 応答 → 採点（従来的） |
| マルチターン評価 | 複数段階のやり取り |
| エージェント評価 | 複数ターンでツール呼び出し・状態変更を伴う（最も複雑） |

### 主要用語

| 用語 | 意味 |
|------|------|
| Task | 定義された入力と成功基準を持つ単一テスト |
| Trial | タスク実行の 1 回の試行 |
| Grader | エージェント性能をスコア化するロジック |
| Transcript | 出力・ツール呼び出し・推論を含む完全な記録 |
| Outcome | 試行終了時の環境の最終状態 |
| Evaluation Harness | 端から端までの eval 実行基盤 |
| Evaluation Suite | 特定の能力測定を目的とした複数タスク集 |

> **重要**：「フライト予約エージェントが『予約しました』と表示しても、データベースに予約が存在しなければ失敗」→ Transcript と Outcome の区別が必須

## 採点方法の比較

| 採点タイプ | 強み | 弱み |
|----------|------|------|
| コードベース | 高速・低コスト・客観的・再現可能 | 有効な変動に脆弱、ニュアンス不足 |
| モデルベース | 柔軟・スケーラブル・ニュアンス豊か | 非決定論的・高コスト・キャリブレーション必要 |
| 人間採点 | 金標準品質・専門家判断と一致 | 高コスト・低速・専門家アクセス困難 |

**選択原則**：決定論的（可能なら）→ LLM（必要時）→ 人間（検証用）

## エージェント別評価手法

### コーディングエージェント
- **決定論的 grader**：ユニットテスト合格（主要）
- **成果物検証**：「修正失敗テスト成功 + 既存テスト破損なし」
- **実運用**：単体テスト + LLM ルーブリックが主流

### 対話型エージェント（サポート・営業・コーチング）
- ユーザー人格シミュレーション
- 多次元的成功基準（完了・ターン数・トーン）
- LLM ルーブリック + 状態検査 + ツール呼び出し検証

### リサーチエージェント
- 複合型採点（グラウンド検証 + カバレッジ + ソース品質）
- LLM ルーブリックは専門家との定期キャリブレーション必須

### コンピュータ使用エージェント
- WebArena：ブラウザタスク、URL/ページ状態検査、バックエンド検証
- OSWorld：OS 制御、ファイルシステム・設定・DB・UI 要素検査

## 非決定論性への対応

### pass@k
「k 回の試行のうち少なくとも 1 回正解する確率」

- k 増加で pass@k 上昇
- 用途：1 つの成功で十分な場合

### pass^k
「k 回の試行すべてが成功する確率」

- 75% の単一試行成功率 → (0.75)³ ≈ 42%（3 試行全成功）
- 用途：毎回の信頼性が必須（本番ユーザーフェーシング）

## Eval 構築ロードマップ

### Step 0：早期開始
「実際の失敗から得た 20〜50 個のシンプルなタスクが良いスタート」

大効果サイズでは小サンプルも統計有意。完成度 100% を求めず、実失敗をテストケース化。

### Step 1：手動テストから
- 現行リリース前の確認項目
- バグ報告・サポート問い合わせ
- ユーザー影響度で優先順位付け

### Step 2：曖昧性排除
「良いタスクとは、2 人のドメイン専門家が独立して同じ合否判定に至るもの」

### Step 3：堅牢な eval 環境
- 各試行をクリーンな環境から開始
- 共有状態・ファイル遺物・キャッシュを排除
- 相関失敗防止

### Step 4：Eval 飽和の監視
**Eval 飽和** = 100% 通過タスク（改善信号喪失）

例：SWE-Bench は初期 30% → 現在 80% 超（飽和接近）

### 長期運用
「トランスクリプトを読まなければ grader が正しく機能しているかわからない」

## Claude Code での Eval 実例

- 初期段階：従業員・外部ユーザーのフィードバックで高速反復
- 中期段階：簡潔性・ファイル編集など限定領域の eval を追加
- 成熟段階：過度設計などの複雑な行動評価へ拡張

## 評価の多層防御

| 手法 | タイミング | 用途 |
|------|----------|------|
| 自動化 eval | 開発中・毎回コミット | 迅速反復・回帰検査 |
| 本番監視 | デプロイ後 | 分布シフト・予期外失敗 |
| A/B テスト | 十分トラフィック後 | 有意性確認 |
| ユーザーフィードバック | 継続的 | 予期外問題検出 |
| トランスクリプト手動レビュー | 週単位サンプリング | 直感醸成・品質キャリブレーション |
| 体系的人間研究 | LLM grader 検証時 | 金標準判断 |

## 試験対策：判断軸

| 状況 | 正解寄り | 罠 |
|------|---------|-----|
| JSON 抽出品質 | tool_use + JSON Schema + validation retry + field-level eval | スキーマだけで正しいとみなす |
| 品質改善 | eval set・regression test・人間校正 | 体感でプロンプトを直す |
| review の信頼性 | 別セッション・別パスでレビュー | 同じセッションで自己レビューだけ |
| 本番品質保証 | 自動化 eval + 本番監視 + 人間サンプリングの多層構造 | 単一の eval スコアで判断 |
| 新モデル採用判断 | eval で迅速に強弱を判定 | 主観的評価だけで判断 |
