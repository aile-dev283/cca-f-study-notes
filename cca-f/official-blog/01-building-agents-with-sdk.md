# Building agents with the Claude Agent SDK（Claude Agent SDK でエージェントを構築する）

**URL:** <https://claude.com/blog/building-agents-with-the-claude-agent-sdk>  
**対応ドメイン:** D1 / D3 / D5  
**優先度:** S

---

Claude Agent SDK は、開発者が Claude Code の上に強力なエージェントを構築するのを支援するツール群です。本記事では、その始め方を順を追って説明し、私たちのベストプラクティスを共有します。

## Claude にコンピュータを与える

Claude Code の背後にある重要な設計原則は、Claude にはプログラマーが日々使うのと同じツールが必要だということです。Claude は、コードベースの中から適切なファイルを見つけ、ファイルを書き込み・編集し、コードを lint し、実行し、デバッグし、編集し、ときにはコードが成功するまでこれらの操作を反復的に行えなければなりません。

私たちは、（ターミナル経由で）ユーザーのコンピュータへのアクセスを Claude に与えることで、Claude がプログラマーと同じようにコードを書くのに必要なものを手に入れることを発見しました。

しかしこれは、Claude Code の中の Claude を _コーディング以外_ のタスクでも効果的にしました。bash コマンドを実行し、ファイルを編集し、ファイルを作成し、ファイルを検索するツールを与えることで、Claude は CSV ファイルを読み、ウェブを検索し、可視化を構築し、メトリクスを解釈し、その他あらゆる種類のデジタル作業を行えます。要するに、コンピュータを備えた汎用エージェントを作り出すのです。Claude Agent SDK の背後にある重要な設計原則は、あなたのエージェントにコンピュータを与え、人間と同じように作業できるようにすることです。

## 新しいタイプのエージェントを作る

私たちは、Claude にコンピュータを与えることで、これまでよりも効果的なエージェントを構築する能力が解き放たれると考えています。たとえば、私たちの SDK を使えば、開発者は次のようなものを構築できます。

* **金融エージェント（Finance agents）**: 外部 API へのアクセス、データの保存、計算を行うためのコード実行を通じて、あなたのポートフォリオと目標を理解し、投資の評価を支援できるエージェントを構築します。
* **パーソナルアシスタントエージェント（Personal assistant agents）**。社内のデータソースに接続し、アプリケーション横断でコンテキストを追跡することで、旅行の予約やカレンダーの管理を手伝い、さらにアポイントメントの調整やブリーフの作成などを行えるエージェントを構築します。
* **カスタマーサポートエージェント（Customer support agents）**: ユーザーデータの収集とレビュー、外部 API への接続、ユーザーへの返信、必要に応じた人間へのエスカレーションを通じて、カスタマーサービスのチケットのような高い曖昧性を持つユーザーリクエストを処理できるエージェントを構築します。
* **ディープリサーチエージェント（Deep research agents）**: ファイルシステムを検索し、複数のソースからの情報を分析・統合し、ファイル横断でデータを相互参照し、詳細なレポートを生成することで、大規模なドキュメントコレクションにわたって包括的なリサーチを実施できるエージェントを構築します。

そしてさらに多くのものを。その核心において、SDK は、あなたが自動化しようとしているあらゆるワークフロー向けのエージェントを構築するためのプリミティブを提供します。

## エージェントループを構築する

Claude Code では、Claude はしばしば特定のフィードバックループの中で動作します。すなわち、コンテキストを集める（gather context）→ アクションを取る（take action）→ 作業を検証する（verify work）→ 繰り返す（repeat）、というループです。

これは、ほかのエージェントや、それらに与えるべき能力について考えるための有用な枠組みを提供します。これを説明するために、Claude Agent SDK でメールエージェントをどのように構築しうるかという例を順に見ていきます。

### コンテキストを集める（Gather context）

エージェントを開発するとき、あなたはエージェントに単なるプロンプト以上のものを与えたいと考えます。エージェントは自分自身のコンテキストを取得し、更新できる必要があります。ここでは、SDK の機能がどのように役立つかを説明します。

#### Agentic search（エージェント的検索）とファイルシステム

ファイルシステムは、モデルのコンテキストに引き込む _ことができる_ 情報を表しています。

Claude がログやユーザーがアップロードしたファイルのような大きなファイルに遭遇すると、`grep` や `tail` のような bash スクリプトを使って、それらをどの方法でコンテキストにロードするかを判断します。本質的に、エージェントのフォルダとファイルの構造は、一種のコンテキストエンジニアリングになります。

私たちのメールエージェントは、過去の会話を 'Conversations' というフォルダに保存しているかもしれません。これによって、それらについて尋ねられたとき、コンテキストのために過去の会話を検索できるようになります。

#### Semantic search（意味的検索）

Semantic search は通常 agentic search よりも高速ですが、精度が低く、保守が難しく、透明性も低くなります。これは、関連するコンテキストを「チャンク化」し、それらのチャンクをベクトルとして埋め込み、それらのベクトルに対してクエリを実行することで概念を検索する、というものです。その限界を踏まえると、私たちはまず agentic search から始め、より高速な結果やより多くのバリエーションが必要な場合にのみ semantic search を追加することを提案します。

#### Subagents（サブエージェント）

Claude Agent SDK はデフォルトで subagent をサポートしています。Subagent は主に 2 つの理由で有用です。第一に、並列化を可能にします。複数の subagent を立ち上げて、異なるタスクを同時に処理させることができます。第二に、コンテキストの管理に役立ちます。Subagent は自分自身の隔離されたコンテキストウィンドウを使用し、その完全なコンテキストではなく、関連する情報だけをオーケストレーターに送り返します。これにより、大量の情報をふるいにかける必要があり、そのほとんどが役に立たないようなタスクに理想的になります。

私たちのメールエージェントを設計するとき、それに「検索 subagent（search subagent）」の能力を与えるかもしれません。すると、メールエージェントは複数の検索 subagent を並列に切り離して立ち上げ、それぞれがあなたのメール履歴に対して異なるクエリを実行し、メールスレッド全体ではなく関連する抜粋だけを返すようにできます。

#### Compaction（コンパクション）

エージェントが長時間にわたって動作しているとき、コンテキストの維持が重要になります。Claude Agent SDK の compact 機能は、コンテキスト上限が近づいたときに過去のメッセージを自動的に要約するため、エージェントがコンテキストを使い果たすことがありません。これは Claude Code の compact スラッシュコマンドの上に構築されています。

### アクションを取る（Take action）

コンテキストを集めたら、次はエージェントにアクションを取るための柔軟な手段を与えたいと考えます。

#### Tools（ツール）

ツールは、あなたのエージェントの実行における主要な構成要素です。ツールは Claude のコンテキストウィンドウの中で目立つ存在であり、タスクをどのように完了させるかを決めるときに Claude が考慮する主要なアクションになります。これは、コンテキスト効率を最大化するために、ツールをどう設計するかを意識すべきだということを意味します。さらなるベストプラクティスは、私たちのブログ記事「Writing effective tools for agents – with agents」で見ることができます。

そのため、あなたのツールは、エージェントに取らせたい主要なアクションであるべきです。Claude Agent SDK でカスタムツールを作る方法を学んでください。

私たちのメールエージェントでは、「`fetchInbox`」や「`searchEmails`」のようなツールを、エージェントの主要で最も頻繁なアクションとして定義するかもしれません。

#### Bash & スクリプト

Bash は、コンピュータを使ってエージェントに柔軟な作業を行わせるための汎用ツールとして有用です。

私たちのメールエージェントでは、ユーザーが重要な情報を添付ファイルに保存しているかもしれません。Claude はコードを書いて PDF をダウンロードし、それをテキストに変換し、それを横断的に検索して有用な情報を見つけることができます。

#### Code generation（コード生成）

Claude Agent SDK はコード生成に長けており、それには十分な理由があります。コードは精密で、合成可能で、無限に再利用可能であり、複雑な操作を確実に実行する必要があるエージェントにとって理想的な出力になります。

エージェントを構築するときは、次のことを考えてみてください。どのタスクがコードとして表現されることで恩恵を受けるだろうか? しばしばその答えが、大きな能力を解き放ちます。

たとえば、私たちが最近ローンチしたファイル作成機能は、完全にコード生成に依存しています。Claude は Python スクリプトを書いて Excel スプレッドシート、PowerPoint プレゼンテーション、Word ドキュメントを作成し、一貫したフォーマットと、ほかの方法では達成が難しい複雑な機能を確保します。

#### MCPs（MCP）

Model Context Protocol（MCP）は、外部サービスへの標準化された統合を提供し、認証や API 呼び出しを自動的に処理します。これは、カスタムの統合コードを書いたり、OAuth フローを自分で管理したりすることなく、Slack、GitHub、Google Drive、Asana のようなツールにエージェントを接続できることを意味します。

私たちのメールエージェントでは、チームのコンテキストを理解するために Slack メッセージを検索したり、誰かがすでにカスタマーリクエストの対応に割り当てられているかを確認するために Asana のタスクをチェックしたりしたいかもしれません。MCP サーバーを使えば、これらの統合は箱から出してすぐに機能します。エージェントは単にツールを呼び出すだけでよく、残りは MCP が処理してくれます。

成長を続ける MCP エコシステムは、構築済みの統合が利用可能になるにつれて、新しい能力を素早くエージェントに追加できることを意味し、あなたはエージェントの振る舞いに集中できるようになります。

### 作業を検証する（Verify your work）

Claude Code SDK は、自身の作業を評価することでエージェントループを締めくくります。自分自身の出力をチェックして改善できるエージェントは、本質的により信頼性が高くなります。問題が積み重なる前に間違いを捉え、ずれが生じたときに自己修正し、反復するにつれて良くなっていきます。

鍵となるのは、Claude に自身の作業を評価するための具体的な手段を与えることです。ここでは、私たちが効果的だと分かった 3 つのアプローチを紹介します。

#### ルールを定義する（Defining rules）

フィードバックの最良の形は、出力に対して明確に定義されたルールを提供し、次にどのルールがなぜ失敗したのかを説明することです。

コードの linting は、ルールベースのフィードバックの優れた形です。フィードバックは詳細であればあるほど良いです。たとえば、純粋な JavaScript を生成するよりも、TypeScript を生成して lint する方が通常は優れています。なぜなら、それは複数の追加的なフィードバックの層を提供してくれるからです。

メールを生成するとき、メールアドレスが有効であること（そうでなければエラーをスローする）と、ユーザーが以前にその相手にメールを送ったことがあること（そうであれば警告をスローする）を Claude にチェックさせたいかもしれません。

#### 視覚的フィードバック（Visual feedback）

UI 生成やテストのような視覚的タスクを完了させるためにエージェントを使うとき、（スクリーンショットやレンダーの形での）視覚的フィードバックが役立ちます。たとえば、HTML フォーマットのメールを送る場合、生成したメールをスクリーンショットに撮ってモデルに戻し、視覚的な検証と反復的な改善を行うことができます。そうすればモデルは、視覚的な出力が要求されたものと一致するかどうかをチェックします。

たとえば次のような観点です。

* **レイアウト（Layout）** - 要素は正しく配置されているか? 余白は適切か?
* **スタイリング（Styling）** - 色、フォント、フォーマットは意図したとおりに見えるか?
* **コンテンツ階層（Content hierarchy）** - 情報は適切な強調を伴って正しい順序で提示されているか?
* **レスポンシブ性（Responsiveness）** - 壊れていたり窮屈に見えたりしないか?（ただし 1 枚のスクリーンショットでは得られるビューポート情報は限られる）

Playwright のような MCP サーバーを使えば、この視覚的フィードバックループを自動化できます。レンダリングされた HTML のスクリーンショットを撮ったり、異なるビューポートサイズをキャプチャしたり、さらにはインタラクティブな要素をテストしたりすることを、すべてエージェントのワークフローの中で行えます。

#### LLM をジャッジとして使う（LLM as a judge）

別の言語モデルに、ファジーなルールに基づいてエージェントの出力を「ジャッジ」させることもできます。これは一般的にあまり堅牢な方法ではなく、レイテンシ面で大きなトレードオフを伴うことがありますが、わずかな性能向上でもそのコストに見合うアプリケーションにとっては役立ちます。

私たちのメールエージェントは、ドラフトのトーンを別の subagent にジャッジさせて、それがユーザーの過去のメッセージとよく合っているかどうかを確認するかもしれません。

## エージェントをテストして改善する

エージェントループを数回通したら、私たちはエージェントをテストし、それがタスクに対して十分に装備されていることを確認することを推奨します。エージェントを改善する最良の方法は、その出力を、特に失敗したケースを注意深く見て、エージェントの立場に身を置いてみることです。それはこの仕事に適したツールを持っているだろうか?

エージェントが仕事をこなすのに十分に装備されているかどうかを評価する際に問うべき、ほかのいくつかの質問を挙げます。

* エージェントがタスクを誤解している場合、重要な情報が欠けているのかもしれません。検索 API の構造を変更して、エージェントが知る必要のあることを見つけやすくできないでしょうか?
* エージェントがタスクで繰り返し失敗する場合、その失敗を特定して修正するための形式的なルールをツール呼び出しに追加できないでしょうか?
* エージェントが自分のエラーを修正できない場合、問題に別の方法でアプローチするための、より有用または創造的なツールを与えられないでしょうか?
* 機能を追加するにつれてエージェントの性能がばらつく場合、顧客の利用実態に基づいて、プログラム的な評価（evals）のための代表的なテストセットを構築しましょう。

## 始めるには（Getting started）

Claude Agent SDK は、ファイルを書き、コマンドを実行し、自身の作業を反復できるコンピュータへのアクセスを Claude に与えることで、自律エージェントの構築をより容易にします。

エージェントループ（コンテキストを集める、アクションを取る、そして作業を検証する）を念頭に置くことで、デプロイと反復が容易な信頼性の高いエージェントを構築できます。

Claude Agent SDK は今日から使い始められます。すでに SDK の上で構築している開発者には、このガイドに従って最新バージョンへ移行することを推奨します。
