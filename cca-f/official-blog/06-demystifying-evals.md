# Demystifying evals for AI agents（AIエージェントの評価を解き明かす）

**URL:** <https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents>  
**対応ドメイン:** D4 / D5  
**優先度:** A

---

## はじめに

優れた評価（eval）は、チームがより自信を持ってAIエージェントを出荷する助けになる。評価がなければ、リアクティブなループにはまり込みやすい。つまり、本番環境でしか問題に気づけず、そこで1つの不具合を直すと別の不具合が生まれてしまう、という状況だ。評価は、問題や挙動の変化がユーザーに影響を及ぼす前に可視化してくれるものであり、その価値はエージェントのライフサイクル全体を通じて複利的に積み上がっていく。

[Building effective agents](https://www.anthropic.com/engineering/building-effective-agents) で述べたとおり、エージェントは多数のターンにわたって動作する。すなわち、ツールを呼び出し、状態を変更し、中間結果に基づいて適応していく。AIエージェントを有用にしているまさにその能力――自律性、知性、柔軟性――が、同時にエージェントの評価を難しくもしている。

私たちは社内での取り組みや、エージェント開発の最前線にいる顧客との協業を通じて、エージェントに対してより厳密で有用な評価を設計する方法を学んできた。以下では、実世界での展開において、さまざまなエージェントアーキテクチャやユースケースにわたって有効だった知見を紹介する。

## 評価の構造

**評価**（"eval"）とは、AIシステムに対するテストである。AIに入力を与え、その出力に採点ロジックを適用して成功を測定する。本稿では、実際のユーザーを使わずに開発中に実行できる **自動評価（automated evals）** に焦点を当てる。

**単一ターン評価（single-turn evaluations）** はシンプルだ。プロンプト、応答、そして採点ロジックである。初期のLLMでは、単一ターンで非エージェント的な評価が主要な評価手法だった。AIの能力が進歩するにつれ、**マルチターン評価（multi-turn evaluations）** がますます一般的になってきた。

シンプルな評価では、エージェントがプロンプトを処理し、grader（採点器）が出力が期待どおりかをチェックする。より複雑なマルチターン評価では、コーディングエージェントがツール、タスク（この場合はMCPサーバーの構築）、および環境を受け取り、「エージェントループ」（ツール呼び出しと推論）を実行し、実装によって環境を更新する。その後、採点ではユニットテストを使って、動作するMCPサーバーを検証する。

**エージェント評価（agent evaluations）** はさらに複雑である。エージェントは多数のターンにわたってツールを使い、環境内の状態を変更しながら適応していく――つまり、ミスが伝播して積み重なる可能性がある。さらにフロンティアモデルは、静的な評価の限界を超える創造的な解を見つけることもある。たとえば、Opus 4.5 はフライト予約に関する [𝜏2-bench](https://github.com/sierra-research/tau2-bench) の問題を、ポリシーの抜け穴を[発見](https://www.anthropic.com/news/claude-opus-4-5) することで解決した。記述されたとおりの評価では「失敗」したが、実際にはユーザーにとってより良い解を見出していたのである。

エージェント評価を構築する際、私たちは以下の定義を用いる。

*   **task**（別名 **problem** または **test case**）とは、定義された入力と成功基準を持つ単一のテストである。
*   タスクへの各試行が **trial** である。モデルの出力は実行ごとに変動するため、より一貫した結果を得るために複数のtrialを実行する。
*   **grader** は、エージェントのパフォーマンスのある側面を採点するロジックである。1つのタスクは複数のgraderを持つことができ、各graderは複数のアサーション（**checks** と呼ばれることもある）を含む。
*   **transcript**（**trace** または **trajectory** とも呼ばれる）は、出力、ツール呼び出し、推論、中間結果、その他あらゆるインタラクションを含む、trialの完全な記録である。Anthropic API の場合、これは評価実行の終了時点における完全な messages 配列であり、評価中のAPIへのすべての呼び出しと、返されたすべての応答を含む。
*   **outcome** は、trial終了時の環境における最終状態である。フライト予約エージェントは transcript の最後に「フライトを予約しました」と言うかもしれないが、outcome は環境の SQL データベースに予約が存在するかどうかである。
*   **evaluation harness** は、評価をエンドツーエンドで実行するインフラである。指示やツールを提供し、タスクを並行して実行し、すべてのステップを記録し、出力を採点し、結果を集約する。
*   **agent harness**（または **scaffold**）は、モデルがエージェントとして動作することを可能にするシステムである。入力を処理し、ツール呼び出しをオーケストレーションし、結果を返す。私たちが「エージェント」を評価するとき、評価しているのは harness _と_ モデルが連携して動作したものである。たとえば、[Claude Code](https://claude.com/product/claude-code) は柔軟な agent harness であり、私たちはその中核プリミティブを [Agent SDK](https://platform.claude.com/docs/en/agent-sdk/overview) を通じて利用し、[長時間稼働するエージェントハーネス](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) を構築した。
*   **evaluation suite** は、特定の能力や挙動を測定するために設計されたタスクの集合である。スイート内のタスクは通常、広い目標を共有する。たとえば、カスタマーサポートの評価スイートは、返金、キャンセル、エスカレーションをテストするかもしれない。

## なぜ評価を構築するのか？

チームが最初にエージェントを構築し始めるとき、手動テスト、[dogfooding](https://en.wikipedia.org/wiki/Eating_your_own_dog_food)、そして直感の組み合わせで、驚くほど遠くまで進むことができる。より厳密な評価は、出荷を遅らせるオーバーヘッドのようにすら見えるかもしれない。しかし、初期のプロトタイピング段階を過ぎ、ひとたびエージェントが本番環境に入りスケールし始めると、評価なしでの構築は破綻し始める。

破綻点が訪れるのは多くの場合、変更後にエージェントの調子が悪くなったとユーザーが報告し、チームが「計器なしで飛んでいる」状態になって、当てずっぽうで試すしか検証手段がなくなったときである。評価がなければ、デバッグはリアクティブになる。すなわち、苦情を待ち、手動で再現し、バグを修正し、他に何も後退していないことを祈る、というやり方だ。チームは、本物のリグレッションをノイズと区別すること、出荷前に何百ものシナリオに対して変更を自動テストすること、改善を測定すること、のいずれもできない。

私たちは、この進展が何度も繰り返されるのを見てきた。たとえば、Claude Code は当初、Anthropic の従業員や外部ユーザーからのフィードバックに基づく高速な反復から始まった。後に評価を追加した――最初は簡潔さやファイル編集といった狭い領域について、次に過剰設計（over-engineering）のようなより複雑な挙動についてである。これらの評価は、問題の特定、改善の指針、そしてリサーチとプロダクトのコラボレーションの焦点化に役立った。本番監視、A/Bテスト、ユーザーリサーチなどと組み合わせることで、評価は Claude Code をスケールしながら改善し続けるためのシグナルを提供してくれる。

評価を書くことは、エージェントのライフサイクルのどの段階でも有用である。初期には、評価はプロダクトチームに対して、エージェントにとっての成功とは何かを明確に定義することを強いる。後の段階では、一貫した品質基準を維持するのに役立つ。

[Descript](https://www.descript.com/) のエージェントはユーザーの動画編集を支援するため、彼らは成功する編集ワークフローの3つの次元を中心に評価を構築した。すなわち、壊さないこと、頼まれたことを実行すること、それをうまく実行すること、である。彼らは手動採点から、プロダクトチームが定義した基準と定期的な人間によるキャリブレーションを伴うLLM graderへと進化させ、現在では品質ベンチマーク用とリグレッションテスト用の2つの別々のスイートを定期的に実行している。[Bolt](https://bolt.new/) のAIチームは、すでに広く使われるエージェントを持った後で、より遅い段階から評価の構築を始めた。3か月で、彼らはエージェントを実行して静的解析で出力を採点し、ブラウザエージェントを使ってアプリをテストし、指示追従などの挙動についてLLMジャッジを用いる評価システムを構築した。

開発の最初から評価を作るチームもあれば、評価がエージェント改善のボトルネックになるスケール段階で一度に追加するチームもある。評価は、期待される挙動を明示的にエンコードするために、エージェント開発の初期段階で特に有用である。同じ初期仕様を読んだ2人のエンジニアが、AIがエッジケースをどう扱うべきかについて異なる解釈にたどり着くこともありうる。評価スイートはこの曖昧さを解消する。いつ作られたかにかかわらず、評価は開発の加速に役立つ。

評価はまた、新しいモデルをどれだけ素早く採用できるかをも左右する。より強力なモデルが登場したとき、評価を持たないチームは数週間のテストに直面する一方、評価を持つ競合は、そのモデルの強みを素早く見極め、プロンプトを調整し、数日でアップグレードできる。

評価がいったん存在すれば、ベースラインとリグレッションテストが無料で手に入る。レイテンシ、トークン使用量、タスクあたりのコスト、エラー率を、静的なタスク群に対して追跡できるのだ。評価はまた、プロダクトチームとリサーチチームの間の最も帯域幅の広いコミュニケーションチャネルにもなりうる。研究者が最適化対象とできるメトリクスを定義するからである。明らかに、評価にはリグレッションや改善の追跡を超えた、広範な恩恵がある。コストは前払いで目に見える一方、恩恵は後から積み上がるため、その複利的な価値は見落とされやすい。

## AIエージェントの評価方法

今日、大規模に展開されているエージェントには、コーディングエージェント、リサーチエージェント、コンピュータ使用エージェント、対話型エージェントなど、いくつかの一般的なタイプがある。各タイプはさまざまな業界にわたって展開されうるが、いずれも類似の手法で評価できる。評価をゼロから発明する必要はない。以下のセクションでは、いくつかのエージェントタイプに対する実証済みの手法を説明する。これらの手法を土台として用い、それを自分のドメインへと拡張してほしい。

### エージェント向けgraderの種類

エージェント評価は通常、コードベース、モデルベース、人間という3種類のgraderを組み合わせる。各graderは、transcriptまたはoutcomeのいずれかの一部を評価する。効果的な評価設計の不可欠な要素は、仕事に適したgraderを選ぶことである。

#### コードベースのgrader

| **手法** | **強み** | **弱み** |
|---|---|---|
| • 文字列一致チェック（完全一致、正規表現、ファジーなど）  <br/>• バイナリテスト（fail-to-pass、pass-to-pass）  <br/>• 静的解析（lint、型、セキュリティ）  <br/>• outcomeの検証  <br/>• ツール呼び出しの検証（使用ツール、パラメータ）  <br/>• transcriptの分析（要したターン数、トークン使用量） | • 高速  <br/>• 安価  <br/>• 客観的  <br/>• 再現可能  <br/>• デバッグが容易  <br/>• 特定の条件を検証できる | • 期待されるパターンと完全には一致しない、有効な変動に対して脆弱  <br/>• ニュアンスを欠く  <br/>• より主観的なタスクの一部の評価には限界がある |

#### モデルベースのgrader

| **手法** | **強み** | **弱み** |
|---|---|---|
| • ルーブリックベースのスコアリング <br/>• 自然言語アサーション <br/>• ペアワイズ比較 <br/>• リファレンスベースの評価 <br/>• 複数ジャッジによる合議 | • 柔軟 <br/>• スケーラブル <br/>• ニュアンスを捉える <br/>• オープンエンドなタスクを扱える <br/>• 自由形式の出力を扱える | • 非決定論的 <br/>• コードより高コスト <br/>• 精度のために人間のgraderとのキャリブレーションが必要 |

#### 人間のgrader

| **手法** | **強み** | **弱み** |
|---|---|---|
| • SME（専門家）レビュー <br/>• クラウドソーシングによる判断 <br/>• スポットチェックのサンプリング <br/>• A/Bテスト <br/>• アノテーター間一致 | • ゴールドスタンダードの品質 <br/>• 専門家ユーザーの判断と一致 <br/>• モデルベースのgraderのキャリブレーションに使われる | • 高コスト <br/>• 低速 <br/>• 大規模には人間の専門家へのアクセスを要することが多い |

各タスクについて、スコアリングは重み付き（複数graderのスコアの合計が閾値に達する必要がある）、バイナリ（すべてのgraderが合格する必要がある）、またはその両者のハイブリッドにできる。

### 能力評価 vs. リグレッション評価

**能力評価（capability evals）すなわち「品質」評価** は、「このエージェントは何をうまくできるか？」を問う。これらは低い合格率から始めるべきであり、エージェントが苦戦するタスクを狙い、チームに登るべき坂を与える。

**リグレッション評価（regression evals）** は、「このエージェントはこれまで扱えていたすべてのタスクを依然として扱えるか？」を問い、ほぼ100%の合格率を持つべきである。これらは後退を防ぐ。スコアの低下は、何かが壊れていて改善が必要であることを示すシグナルとなるからだ。チームが能力評価で坂を登るにつれ、変更が他の場所で問題を引き起こさないことを確認するため、リグレッション評価も同時に実行することが重要である。

エージェントがローンチされ最適化された後、高い合格率を持つ能力評価は、ドリフトを捉えるために継続的に実行されるリグレッションスイートへと「卒業」できる。かつて「そもそもこれができるのか？」を測っていたタスクは、その後「これを依然として確実にできるのか？」を測るようになる。

### コーディングエージェントの評価

**コーディングエージェント** は、コードを書き、テストし、デバッグし、人間の開発者と同じようにコードベースをナビゲートしてコマンドを実行する。現代のコーディングエージェントに対する効果的な評価は通常、十分に仕様化されたタスク、安定したテスト環境、そして生成コードに対する徹底的なテストに依拠する。

決定論的なgraderはコーディングエージェントにとって自然である。なぜなら、ソフトウェアは一般に評価しやすいからだ。すなわち、コードが動作するか、テストが通るか、である。広く使われている2つのコーディングエージェントベンチマーク、[SWE-bench Verified](https://www.swebench.com/SWE-bench/) と [Terminal-Bench](https://www.tbench.ai/) は、このアプローチに従っている。SWE-bench Verified は、人気のあるPythonリポジトリのGitHub issueをエージェントに与え、テストスイートを実行して解を採点する。解が合格するのは、既存のテストを壊さずに失敗していたテストを修正した場合のみである。LLMはこの評価において、わずか1年で40%から80%超へと進歩した。Terminal-Bench は異なる路線をとる。すなわち、Linuxカーネルをソースからビルドする、MLモデルを訓練するといった、エンドツーエンドの技術的タスクをテストする。

コーディングタスクの主要な _outcome_ を検証する合否テストの集合を得たら、transcript も採点することがしばしば有用である。たとえば、ヒューリスティックベースのコード品質ルールは、テストの合格以外の観点から生成コードを評価でき、明確なルーブリックを持つモデルベースのgraderは、エージェントがどのようにツールを呼び出すか、ユーザーとどう対話するかといった挙動を評価できる。

**例：コーディングエージェントの理論的な評価**

エージェントが認証バイパスの脆弱性を修正しなければならないコーディングタスクを考える。下記の例示的なYAMLファイルに示すように、このエージェントはgraderとメトリクスの両方を用いて評価できる。

```yaml
task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token
```

なお、この例は説明のために利用可能なgraderの全範囲を示している。実際には、コーディング評価は通常、正しさの検証にユニットテストを、全体的なコード品質の評価にLLMルーブリックを用い、追加のgraderやメトリクスは必要に応じてのみ加える。

### 対話型エージェントの評価

**対話型エージェント** は、サポート、営業、コーチングといった領域でユーザーと対話する。従来のチャットボットとは異なり、状態を維持し、ツールを使い、会話の途中でアクションを起こす。コーディングエージェントやリサーチエージェントもユーザーとの多数のターンのやり取りを伴いうるが、対話型エージェントは独特の課題を提示する。すなわち、対話そのものの品質が、評価対象の一部だということである。対話型エージェントに対する効果的な評価は通常、検証可能な最終状態のoutcomeと、タスク完了および対話品質の両方を捉えるルーブリックに依拠する。他のほとんどの評価とは異なり、ユーザーをシミュレートするために2つ目のLLMを必要とすることが多い。私たちはこのアプローチを [アライメント監査エージェント](https://alignment.anthropic.com/2025/automated-auditing/) で用い、長期にわたる敵対的な会話を通じてモデルにストレステストを行っている。

対話型エージェントの成功は多次元でありうる。すなわち、チケットは解決されたか（状態チェック）、10ターン未満で終わったか（transcriptの制約）、トーンは適切だったか（LLMルーブリック）、といった具合だ。多次元性を取り入れた2つのベンチマークが [𝜏-Bench](https://arxiv.org/abs/2406.12045) とその後継 [τ2-Bench](https://arxiv.org/abs/2506.07982) である。これらは小売サポートや航空券予約といった領域にわたってマルチターンの対話をシミュレートし、一方のモデルがユーザーのペルソナを演じるなか、エージェントが現実的なシナリオをナビゲートする。

**例：対話型エージェントの理論的な評価**

エージェントが不満を抱える顧客の返金を処理しなければならないサポートタスクを考える。

```yaml
graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}
      refunds: {status: processed}
  - type: tool_calls
    required:
      - {tool: verify_identity}
      - {tool: process_refund, params: {amount: "<=100"}}
      - {tool: send_confirmation}
  - type: transcript
    max_turns: 10
tracked_metrics:
  - type: transcript
    metrics:
      - n_turns
      - n_toolcalls
      - n_total_tokens
  - type: latency
    metrics:
      - time_to_first_token
      - output_tokens_per_sec
      - time_to_last_token
```

コーディングエージェントの例と同様に、このタスクは説明のために複数のgraderタイプを示している。実際には、対話型エージェントの評価は通常、コミュニケーション品質と目標達成の両方を評価するためにモデルベースのgraderを用いる。多くのタスク――質問への回答など――は複数の「正しい」解を持ちうるからである。

### リサーチエージェントの評価

**リサーチエージェント** は、情報を収集し、統合し、分析して、回答やレポートといった出力を生成する。ユニットテストがバイナリの合否シグナルを提供するコーディングエージェントとは異なり、リサーチの品質はタスクとの関係でしか判断できない。「包括的」「十分に出典が示されている」あるいは「正しい」とは何かは、文脈に依存する。市場調査、買収のためのデューデリジェンス、科学レポートは、それぞれ異なる基準を要する。

リサーチ評価は独特の課題に直面する。専門家は、ある統合が包括的かどうかについて意見が分かれることがあり、参照コンテンツが絶えず変化するためグラウンドトゥルース（正解）が移ろい、そしてより長くオープンエンドな出力は誤りの余地を増やす。たとえば [BrowseComp](http://arxiv.org/abs/2504.12516) のようなベンチマークは、AIエージェントがオープンウェブ全体から干し草の中の針を見つけられるかをテストする――検証は容易だが解くのは難しいように設計された問いである。

リサーチエージェントの評価を構築する1つの戦略は、graderタイプを組み合わせることである。グラウンドネス（根拠付け）チェックは、主張が取得したソースによって裏付けられていることを検証し、カバレッジチェックは、良い回答が含むべき主要な事実を定義し、ソース品質チェックは、参照されたソースが――単に最初に取得されたものではなく――権威あるものであることを確認する。客観的に正しい答えがあるタスク（「X社の第3四半期の売上はいくらだったか？」）には、完全一致が有効である。LLMは、裏付けのない主張やカバレッジのギャップにフラグを立てられるだけでなく、オープンエンドな統合の一貫性と完全性を検証することもできる。

リサーチ品質の主観的な性質を踏まえると、これらのエージェントを効果的に採点するには、LLMベースのルーブリックを専門家の人間の判断に対して頻繁にキャリブレーションすべきである。

### コンピュータ使用エージェント

**コンピュータ使用エージェント** は、APIやコード実行を通じてではなく、人間と同じインターフェース――スクリーンショット、マウスクリック、キーボード入力、スクロール――を通じてソフトウェアと対話する。デザインツールからレガシーなエンタープライズソフトウェアまで、グラフィカルユーザーインターフェース（GUI）を持つあらゆるアプリケーションを使うことができる。評価には、エージェントがソフトウェアアプリケーションを使える現実またはサンドボックス化された環境でエージェントを実行し、意図された結果を達成したかをチェックすることが必要である。たとえば [WebArena](https://arxiv.org/abs/2307.13854) は、ブラウザベースのタスクをテストし、URLとページ状態のチェックを用いてエージェントが正しくナビゲートしたことを検証するとともに、データを変更するタスクについてはバックエンドの状態検証を行う（確認ページが表示されただけでなく、実際に注文が行われたことを確認する）。[OSWorld](https://os-world.github.io/) はこれをOS全体の制御へと拡張し、タスク完了後に多様な成果物――ファイルシステムの状態、アプリケーション設定、データベースの内容、UI要素のプロパティ――を検査する評価スクリプトを備える。

ブラウザ使用エージェントは、トークン効率とレイテンシのバランスを必要とする。DOMベースのインタラクションは高速に実行されるが多くのトークンを消費し、スクリーンショットベースのインタラクションはより遅いがトークン効率が高い。たとえば、ClaudeにWikipediaを要約させるときは、DOMからテキストを抽出するほうが効率的である。Amazonで新しいノートパソコンケースを探すときは、スクリーンショットを撮るほうが効率的である（DOM全体を抽出するのはトークン集約的だからだ）。私たちのClaude for Chrome製品では、エージェントがそれぞれの文脈に適したツールを選択しているかをチェックする評価を開発した。これにより、ブラウザベースのタスクをより速く、より正確に完了できるようになった。

### エージェント評価における非決定論性の考え方

エージェントのタイプにかかわらず、エージェントの挙動は実行ごとに変動するため、評価結果は一見そう見えるよりも解釈が難しい。各タスクは固有の成功率を持ち――あるタスクでは90%、別のタスクでは50%といった具合に――ある評価実行で合格したタスクが、次の実行では失敗することもありうる。私たちが測定したいのは、エージェントがあるタスクで _どれくらいの頻度で_（trialのどの割合で）成功するか、である場合もある。

このニュアンスを捉えるのに役立つ2つのメトリクスがある。

[**pass@k**](https://proceedings.neurips.cc/paper/2019/file/7298332f04ac004a0ca44cc69ecf6f6b-Paper.pdf) は、エージェントが _k_ 回の試行のうち少なくとも1回正しい解を得る尤度を測定する。_k_ が増えるにつれ、pass@k のスコアは上昇する。「シュート機会」が多いほど、少なくとも1回成功するオッズが高くなるからだ。50% pass@1 というスコアは、モデルが評価内のタスクの半分を最初の試行で成功させることを意味する。コーディングでは、私たちはエージェントが最初の試行で解を見つけること――pass@1――に最も関心を持つことが多い。それ以外のケースでは、1つでも機能すれば、多数の解を提案することが妥当な場合もある。

[**pass^k**](https://arxiv.org/abs/2406.12045) は、_k_ 回の trial _すべて_ が成功する確率を測定する。_k_ が増えるにつれ、pass^k は低下する。より多くの trial にわたって一貫性を要求するのは、クリアするのがより難しいバーだからである。エージェントの trial あたりの成功率が75%で、3回の trial を実行した場合、3回すべてに合格する確率は (0.75)³ ≈ 42% である。このメトリクスは、ユーザーが毎回信頼できる挙動を期待する顧客対面エージェントにおいて、とりわけ重要である。

pass@k と pass^k は、trial が増えるにつれて乖離していく。k=1 では両者は同一である（どちらも trial あたりの成功率に等しい）。k=10 までには、両者は正反対の物語を語る。pass@k は100%に近づく一方、pass^k は0%へと落ち込む。

両方のメトリクスは有用であり、どちらを使うかはプロダクト要件に依存する。1回の成功が重要なツールには pass@k を、一貫性が不可欠なエージェントには pass^k を用いる。

## ゼロから1へ：エージェントのための優れた評価へのロードマップ

このセクションでは、評価ゼロの状態から信頼できる評価へと至るための、実践的で現場で検証されたアドバイスを示す。これを、評価駆動のエージェント開発のためのロードマップだと考えてほしい。すなわち、成功を早期に定義し、それを明確に測定し、継続的に反復する、ということである。

### 初期の評価データセット用にタスクを収集する

**Step 0. 早期に始める**

チームが、数百ものタスクが必要だと考えて評価の構築を遅らせるのをよく見かける。実際には、実際の失敗から引き出した20〜50個のシンプルなタスクが、素晴らしいスタートになる。何しろ、エージェント開発の初期では、システムへの各変更がしばしば明確で目立つインパクトを持ち、この大きな効果量は、小さなサンプルサイズで十分であることを意味する。より成熟したエージェントは、より小さな効果を検出するために、より大規模で難しい評価を必要とするかもしれないが、初期には80/20のアプローチをとるのが最善である。評価は、待てば待つほど構築が難しくなる。初期には、プロダクト要件が自然にテストケースへと翻訳される。待ちすぎると、稼働中のシステムから成功基準を逆算する羽目になる。

**Step 1. すでに手動でテストしているものから始める**

開発中に実行している手動チェック――各リリース前に検証する挙動や、エンドユーザーが試す一般的なタスク――から始めよう。すでに本番環境にいるなら、バグトラッカーやサポートキューを見る。ユーザーが報告した失敗をテストケースに変換することで、スイートが実際の使われ方を反映するようになる。ユーザーへのインパクトで優先順位付けすることは、重要なところに労力を投資する助けになる。

**Step 2: リファレンス解付きで曖昧さのないタスクを書く**

タスクの品質を正しく仕上げることは、見かけより難しい。良いタスクとは、2人のドメイン専門家が独立して同じ合否判定にたどり着くようなものである。彼ら自身がそのタスクに合格できるだろうか？ もしできないなら、そのタスクは改良を要する。タスク仕様における曖昧さは、メトリクスにおけるノイズになる。同じことが、モデルベースのgraderの基準にも当てはまる。曖昧なルーブリックは一貫性のない判断を生む。

各タスクは、指示に正しく従うエージェントが合格できるものであるべきだ。これは微妙でありうる。たとえば、Terminal-Bench の監査では、あるタスクがエージェントにスクリプトを書くよう求めながらファイルパスを指定しておらず、テストがそのスクリプトに特定のファイルパスを仮定している場合、エージェントは自分には何の落ち度もなく失敗しうることが明らかになった。graderがチェックするすべては、タスクの記述から明確であるべきだ。エージェントが曖昧な仕様のせいで失敗してはならない。フロンティアモデルの場合、多数の trial にわたる0%の合格率（すなわち 0% pass@100）は、ほとんどの場合、能力のないエージェントではなく、壊れたタスクのシグナルであり、タスク仕様とgraderを再確認すべき兆候である。各タスクについて、リファレンス解――すべてのgraderに合格する、既知の機能する出力――を作ることが有用である。これは、タスクが解けることを証明し、graderが正しく構成されていることを検証する。

**Step 3: バランスのとれた問題セットを構築する**

ある挙動が _起こるべき_ ケースと _起こるべきでない_ ケースの両方をテストしよう。片側だけの評価は片側だけの最適化を生む。たとえば、エージェントが検索すべきときに検索するかどうかだけをテストすると、ほとんどあらゆることについて検索するエージェントになってしまうかもしれない。[クラス不均衡](https://developers.google.com/machine-learning/crash-course/overfitting/imbalanced-datasets) な評価は避けるよう努めよう。私たちは [Claude.ai](http://claude.ai/) でウェブ検索の評価を構築した際、これを身をもって学んだ。課題は、適切なときには広範なリサーチを行う能力を保ちつつ、検索すべきでないときにモデルが検索するのを防ぐことだった。チームは両方向をカバーする評価を構築した。すなわち、モデルが検索すべきクエリ（天気を調べるなど）と、既存の知識から答えるべきクエリ（「Appleを創業したのは誰か？」など）である。アンダートリガリング（検索すべきときに検索しない）とオーバートリガリング（検索すべきでないときに検索する）の間で適切なバランスを取ることは難しく、プロンプトと評価の両方に対して何度もの改良を要した。新しい事例問題が出てくるにつれ、私たちはカバレッジを改善するために評価を追加し続けている。

### 評価ハーネスとgraderを設計する

**Step 4: 安定した環境を備えた堅牢な評価ハーネスを構築する**

評価内のエージェントが本番で使われるエージェントとおおよそ同じように機能すること、そして環境そのものがさらなるノイズを持ち込まないことが不可欠である。各 trial は、クリーンな環境から始めることで「隔離」されるべきだ。実行間の不要な共有状態（残ったファイル、キャッシュデータ、リソース枯渇）は、エージェントのパフォーマンスではなくインフラの不安定さに起因する相関した失敗を引き起こしうる。共有状態はまた、パフォーマンスを人為的に水増しすることもある。たとえば、一部の社内評価では、Claudeが過去の trial の git 履歴を調べることで、いくつかのタスクで不当な優位を得るのを観察した。複数の異なる trial が、環境の同じ制約（限られたCPUメモリなど）のせいで失敗する場合、これらの trial は同じ要因に影響されているため独立ではなく、評価結果はエージェントのパフォーマンスを測定するうえで信頼できないものになる。

**Step 5: graderを慎重に設計する**

上述のとおり、優れた評価設計には、エージェントとタスクに最適なgraderを選ぶことが含まれる。私たちは、可能なところでは決定論的なgraderを、必要なところや追加の柔軟性のためにはLLMのgraderを選び、追加の検証のためには人間のgraderを慎重に用いることを推奨する。

エージェントが、正しい順序でのツール呼び出しのシーケンスといった、非常に特定的なステップに従ったかをチェックしたくなる、という一般的な本能がある。私たちは、このアプローチは硬直的すぎ、過度に脆弱なテストにつながることを発見した。エージェントは、評価設計者が予期しなかった有効なアプローチを頻繁に見つけ出すからである。創造性を不必要に罰しないために、エージェントがたどった経路ではなく、エージェントが生み出したものを採点するほうがしばしば良い。

複数のコンポーネントを持つタスクには、部分点を組み込もう。問題を正しく特定し、顧客を確認したものの、返金の処理に失敗したサポートエージェントは、即座に失敗したエージェントよりも有意に優れている。この成功の連続性を結果に表現することが重要である。

モデルによる採点は、精度を検証するために慎重な反復をしばしば要する。LLM-as-judge のgraderは、人間の採点とモデルの採点の間に乖離がほとんどないという確信を得るために、人間の専門家と緊密にキャリブレーションされるべきである。ハルシネーションを避けるために、LLMに逃げ道を与えよう。たとえば、十分な情報がないときには「Unknown」を返すよう指示するのである。タスクの各次元を採点する明確で構造化されたルーブリックを作り、その上で、1つのLLMですべての次元を採点するのではなく、各次元を独立したLLM-as-judgeで採点することも役立つ。システムが堅牢になれば、人間によるレビューは時折用いるだけで十分である。

一部の評価には微妙な失敗モードがあり、エージェントが採点のバグ、agent harnessの制約、または曖昧さのせいでタスクを解けず、良好なエージェントパフォーマンスがあっても低いスコアになる。洗練されたチームでさえ、これらの問題を見逃しうる。たとえば、[Opus 4.5 は CORE-Bench で当初42%を記録](https://x.com/sayashk/status/1996334941832089732?s=46&t=c5pEvnVdVbMkcD_rcCHplg) していたが、Anthropic の研究者が複数の問題を見つけた。すなわち、「96.124991…」を期待しているのに「96.12」を罰する硬直的な採点、曖昧なタスク仕様、そして正確に再現することが不可能な確率的タスクである。バグを修正し、より制約の少ない scaffold を用いた後、Opus 4.5 のスコアは95%へと跳ね上がった。同様に、[METR は発見した](https://x.com/metr_evals/status/2001473506442375645?s=46)。彼らの time horizon ベンチマークにおいて、エージェントに記述されたスコア閾値へと最適化するよう求めながら、採点はその閾値を超えることを要求する、という誤設定されたタスクが複数あったのだ。これは、指示に従ったClaudeのようなモデルを罰する一方、記述された目標を無視したモデルがより良いスコアを得る結果になった。タスクとgraderを注意深く再確認することは、これらの問題を避ける助けになる。

graderを、バイパスやハックに耐性のあるものにしよう。エージェントが評価を簡単に「ズル」できてはならない。タスクとgraderは、意図しない抜け穴を突くのではなく、合格には本当に問題を解くことが必要となるよう設計されるべきである。

### 評価を長期にわたって維持し活用する

**Step 6: transcriptをチェックする**

多数の trial の transcript と採点を読まない限り、graderがうまく機能しているかはわからない。Anthropic では、評価のtranscriptを閲覧するためのツールに投資し、定期的に時間をとってそれらを読んでいる。タスクが失敗したとき、transcriptは、エージェントが本当にミスを犯したのか、それともあなたのgraderが有効な解を却下したのかを教えてくれる。それはまた、エージェントや評価の挙動に関する重要な詳細をしばしば浮かび上がらせる。

失敗は公正に見えるべきである。すなわち、エージェントが何を、なぜ間違えたのかが明確であるべきだ。スコアが伸びないとき、私たちはそれがエージェントのパフォーマンスによるものであり、評価のせいではないという確信を必要とする。transcriptを読むことは、あなたの評価が本当に重要なことを測定していることを検証する方法であり、エージェント開発にとって重要なスキルである。

**Step 7: 能力評価の飽和を監視する**

100%の評価はリグレッションを追跡するが、改善のためのシグナルは提供しない。**評価の飽和（eval saturation）** は、エージェントが解ける可能性のあるすべてのタスクに合格し、改善の余地が残らないときに生じる。たとえば、SWE-Bench Verified のスコアは今年30%から始まり、フロンティアモデルは現在80%超で飽和に近づいている。評価が飽和に近づくにつれ、最も難しいタスクだけが残るため、進歩も鈍化する。これは結果を欺瞞的なものにしうる。大きな能力の改善が、スコアの小さな増加として現れるからだ。たとえば、コードレビューのスタートアップ [Qodo](https://www.qodo.ai/) は、当初 Opus 4.5 に感心しなかった。彼らのワンショットのコーディング評価が、より長く複雑なタスクでの利得を捉えていなかったからである。これを受けて、彼らは新しいエージェント型の評価フレームワークを開発し、進歩のはるかに明確な姿を得た。

原則として、私たちは、誰かが評価の詳細を掘り下げていくつかのtranscriptを読むまで、評価スコアを額面どおりには受け取らない。採点が不公正であったり、タスクが曖昧であったり、有効な解が罰せられたり、harnessがモデルを制約したりしている場合、その評価は見直されるべきである。

**Step 8: オープンな貢献とメンテナンスによって評価スイートを長期にわたって健全に保つ**

評価スイートは、有用であり続けるために継続的な注意と明確なオーナーシップを必要とする、生きた成果物である。

Anthropic では、評価メンテナンスのさまざまなアプローチを試した。最も効果的だと判明したのは、中核インフラを所有する専任の評価チームを設置する一方、ドメイン専門家とプロダクトチームがほとんどの評価タスクを貢献し、評価を自ら実行する、というやり方だった。

AIプロダクトチームにとって、評価を所有し反復することは、ユニットテストの維持と同じくらい日常的なことであるべきだ。チームは、初期テストでは「機能する」が、よく設計された評価なら早期に浮かび上がらせていたはずの暗黙の期待を満たせないAI機能に、何週間も浪費しかねない。評価タスクを定義することは、プロダクト要件が構築を始められるほど具体的かをストレステストする最良の方法の1つである。

私たちは、評価駆動開発（eval-driven development）を実践することを推奨する。すなわち、エージェントがまだ実現できないうちに、計画された能力を定義するための評価を構築し、その後、エージェントがうまく機能するまで反復するのである。社内では、今日「十分に機能する」が、数か月後にモデルができることへの賭けである機能を、しばしば構築する。低い合格率から始まる能力評価は、これを可視化する。新しいモデルが投入されたとき、スイートを実行すれば、どの賭けが報われたかが素早く明らかになる。

プロダクト要件とユーザーに最も近い人々こそ、成功を定義するのに最適な立場にいる。現在のモデル能力をもってすれば、プロダクトマネージャー、カスタマーサクセスマネージャー、あるいは営業担当者が、Claude Code を使って評価タスクをPRとして貢献できる――そうさせよう！ あるいは、さらに良いことに、積極的にそれを可能にしよう。

_効果的な評価を作るプロセス。_

## 評価が、エージェントの全体的な理解のための他の手法とどう組み合わさるか

自動評価は、本番にデプロイしたり実際のユーザーに影響を与えたりすることなく、何千ものタスクでエージェントに対して実行できる。しかしこれは、エージェントのパフォーマンスを理解する多くの方法のうちの1つにすぎない。完全な姿には、本番監視、ユーザーフィードバック、A/Bテスト、手動でのtranscriptレビュー、そして体系的な人間による評価が含まれる。

### AIエージェントのパフォーマンスを理解するためのアプローチの概観

| **手法** | **長所** | **短所** |
|---|---|---|
| **自動評価** <br/>実際のユーザーを使わずにプログラム的にテストを実行する | • より速い反復 <br/>• 完全に再現可能 <br/>• ユーザーへの影響なし <br/>• すべてのコミットで実行できる <br/>• 本番デプロイを要さずに大規模にシナリオをテストできる | • 構築により多くの前払い投資を要する <br/>• ドリフトを避けるため、プロダクトとモデルの進化に応じた継続的なメンテナンスを要する <br/>• 実際の使用パターンと一致しない場合、偽りの自信を生みうる |
| **本番監視** <br/>稼働中のシステムでメトリクスとエラーを追跡する | • 大規模に実際のユーザー挙動を明らかにする <br/>• 合成的な評価が見逃す問題を捉える <br/>• エージェントが実際にどう振る舞うかについてのグラウンドトゥルースを提供する | • リアクティブ。問題があなたが知る前にユーザーに届く <br/>• シグナルがノイジーでありうる <br/>• 計装への投資を要する <br/>• 採点のためのグラウンドトゥルースを欠く |
| **A/Bテスト** <br/>実際のユーザートラフィックでバリアントを比較する | • 実際のユーザーアウトカム（リテンション、タスク完了）を測定する <br/>• 交絡をコントロールする <br/>• スケーラブルで体系的 | • 遅い。有意性に達するのに数日から数週間かかり、十分なトラフィックを要する <br/>• デプロイした変更しかテストできない <br/>• transcriptを徹底的にレビューできなければ、メトリクスの変化の根底にある「なぜ」についてのシグナルが乏しい |
| **ユーザーフィードバック** <br/>低評価やバグ報告のような明示的なシグナル | • 予期しなかった問題を浮かび上がらせる <br/>• 実際の人間のユーザーからの本物の事例を伴う <br/>• フィードバックはしばしばプロダクト目標と相関する | • まばらで自己選択的 <br/>• 深刻な問題に偏る <br/>• ユーザーは何かが失敗した _理由_ をめったに説明しない <br/>• 自動化されていない <br/>• 問題の検出を主にユーザーに頼ることは、ユーザーに負の影響を及ぼしうる |
| **手動でのtranscriptレビュー** <br/>人間がエージェントの会話を読み通す | • 失敗モードへの直感を養う <br/>• 自動チェックが見逃す微妙な品質問題を捉える <br/>• 「良い」とはどういうものかをキャリブレーションし、詳細を把握する助けになる | • 時間集約的 <br/>• スケールしない <br/>• カバレッジが一貫しない <br/>• レビュアーの疲労や、異なるレビュアーがシグナルの品質に影響しうる <br/>• 通常、明確な定量的採点ではなく定性的なシグナルしか得られない |
| **体系的な人間による研究** <br/>訓練された評価者によるエージェント出力の構造化された採点 | • 複数の人間の評価者によるゴールドスタンダードの品質判断 <br/>• 主観的または曖昧なタスクを扱える <br/>• モデルベースのgraderを改善するためのシグナルを提供する | • 比較的高コストで、ターンアラウンドが遅い <br/>• 頻繁に実行するのが難しい <br/>• 評価者間の不一致は調整を要する <br/>• 複雑なドメイン（法務、金融、ヘルスケア）は研究を行う人間の専門家を要する |

これらの手法は、エージェント開発の異なる段階に対応する。自動評価は、ローンチ前やCI/CDで特に有用であり、品質問題に対する第一の防衛線として、各エージェント変更やモデルアップグレードのたびに実行される。本番監視はローンチ後に始動し、分布のドリフトや予期しない実世界の失敗を検出する。A/Bテストは、十分なトラフィックを得たら、重要な変更を検証する。ユーザーフィードバックとtranscriptレビューは、ギャップを埋める継続的な実践である。すなわち、フィードバックを絶えずトリアージし、毎週読むtranscriptをサンプリングし、必要に応じてさらに掘り下げる。体系的な人間による研究は、LLMのgraderをキャリブレーションする場合や、人間の合意が参照基準となる主観的な出力を評価する場合のために取っておく。

安全工学の [スイスチーズモデル](https://en.wikipedia.org/wiki/Swiss_cheese_model) のように、単一の評価レイヤーがすべての問題を捉えるわけではない。複数の手法を組み合わせれば、1つのレイヤーをすり抜けた失敗が、別のレイヤーで捕まる。

最も効果的なチームは、これらの手法を組み合わせる。すなわち、速い反復のための自動評価、グラウンドトゥルースのための本番監視、そしてキャリブレーションのための定期的な人間によるレビューである。

## 結論

評価を持たないチームは、リアクティブなループにはまり込む――1つの失敗を直すと別の失敗が生まれ、本物のリグレッションをノイズと区別できなくなる。早期に投資するチームは、その逆を見出す。すなわち、失敗がテストケースになり、テストケースがリグレッションを防ぎ、メトリクスが当てずっぽうに取って代わるにつれて、開発が加速するのだ。評価は、チーム全体に登るべき明確な坂を与え、「エージェントの調子が悪い」を、行動可能な何かへと変える。その価値は複利的に積み上がるが、それは評価を後付けではなく中核的な構成要素として扱う場合に限られる。

パターンはエージェントのタイプによって異なるが、ここで述べた基本原則は不変である。早期に始め、完璧なスイートを待たないこと。あなたが目にする失敗から現実的なタスクを調達すること。曖昧さのない、堅牢な成功基準を定義すること。graderを慎重に設計し、複数のタイプを組み合わせること。問題がモデルにとって十分に難しいことを確認すること。シグナル対ノイズ比を改善するために評価を反復すること。そしてtranscriptを読むこと！

AIエージェントの評価は、いまだ黎明期にある、急速に進化する分野である。エージェントがより長いタスクを引き受け、マルチエージェントシステムで協働し、ますます主観的な仕事を扱うようになるにつれ、私たちは手法を適応させていく必要がある。私たちは、さらに学ぶにつれてベストプラクティスを共有し続けていく。

## 付録：評価フレームワーク

いくつかのオープンソースおよび商用のフレームワークは、チームがインフラをゼロから構築せずにエージェント評価を実装するのを助けてくれる。適切な選択は、あなたのエージェントのタイプ、既存のスタック、そしてオフライン評価、本番のオブザーバビリティ、あるいはその両方のどれを必要とするかに依存する。

[Harbor](https://harborframework.com/) は、コンテナ化された環境でエージェントを実行するために設計されており、クラウドプロバイダー全体で trial を大規模に実行するためのインフラと、タスクとgraderを定義するための標準化されたフォーマットを備える。Terminal-Bench 2.0 のような人気のベンチマークは Harbor のレジストリを通じて提供されており、確立されたベンチマークをカスタム評価スイートとともに容易に実行できる。

[Braintrust](https://www.braintrust.dev/) は、オフライン評価を本番のオブザーバビリティおよび実験追跡と組み合わせるプラットフォームであり――開発中の反復と本番での品質監視の両方を必要とするチームに有用である。その `autoevals` ライブラリには、事実性、関連性、その他の一般的な次元のための事前構築されたスコアラーが含まれる。

[LangSmith](https://docs.langchain.com/langsmith/evaluation) は、トレーシング、オフラインおよびオンラインの評価、そしてデータセット管理を、LangChain エコシステムとの緊密な統合とともに提供する。[Langfuse](https://langfuse.com/) は、データレジデンシー要件を持つチームのために、セルフホスト型のオープンソースの代替として同様の機能を提供する。

[Arize](https://arize.com/) は、LLMのトレーシング、デバッグ、オフラインまたはオンラインの評価のためのオープンソースプラットフォームである Phoenix と、Phoenix をスケール、最適化、監視のために拡張する SaaS 製品である AX を提供する。

多くのチームは、複数のツールを組み合わせたり、独自の評価フレームワークを自作したり、あるいは出発点として単純な評価スクリプトを使ったりする。私たちは、フレームワークが進歩を加速し標準化する価値ある手段になりうる一方で、それらはそこに通すあなたの評価タスクの質を超えることはない、と考えている。多くの場合、自分のワークフローに合うフレームワークを素早く選び、それから高品質なテストケースとgraderを反復することによって、エネルギーを評価そのものに投じるのが最善である。
