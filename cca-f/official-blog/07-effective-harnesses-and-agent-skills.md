# Effective harnesses for long-running agents（長期実行エージェントのための効果的なハーネス）

**URL:** <https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents>  
**対応ドメイン:** D1 / D5 / D3  
**優先度:** A/B

---

2025年11月26日公開

エージェントは依然として、多数のコンテキストウィンドウをまたいで作業することに課題を抱えています。私たちは、長期実行エージェントのためのより効果的なハーネスを作るにあたり、人間のエンジニアからヒントを得ました。

AIエージェントの能力が高まるにつれ、開発者はエージェントに対して、数時間、あるいは数日にわたる作業を要する複雑なタスクをますます任せるようになっています。しかし、複数のコンテキストウィンドウをまたいでエージェントに一貫した進捗を上げさせることは、依然として未解決の問題です。

長期実行エージェントの根本的な課題は、エージェントが離散的なセッションで作業しなければならず、新しいセッションはそれ以前に起きたことの記憶を持たずに始まるという点にあります。シフト制で働くエンジニアによって運営されているソフトウェアプロジェクトを想像してみてください。新しく入るエンジニアは、前のシフトで何が起きたかの記憶を一切持たずにやって来ます。コンテキストウィンドウには限りがあり、ほとんどの複雑なプロジェクトは単一のウィンドウ内で完了できないため、エージェントにはコーディングセッション間のギャップを橋渡しする手段が必要です。

私たちは、Claude Agent SDK が多数のコンテキストウィンドウをまたいで効果的に作業できるようにするための、二段構えの解決策を開発しました。それは、初回実行時に環境をセットアップする初期化エージェント（initializer agent）と、毎セッションで段階的な進捗を上げることを任務とし、次のセッションのために明確な成果物を残すコーディングエージェント（coding agent）です。付属の quickstart にコード例が掲載されています。

## 長期実行エージェントの問題

Claude Agent SDK は、コーディングはもちろん、モデルがツールを使ってコンテキストを収集し、計画を立て、実行することを必要とするその他のタスクにも長けた、強力で汎用的なエージェントハーネスです。これには compaction（圧縮）などのコンテキスト管理機能があり、コンテキストウィンドウを使い尽くすことなくエージェントがタスクに取り組めるようにします。理論的には、このセットアップがあれば、エージェントは任意に長い時間にわたって有用な作業を続けることが可能なはずです。

しかし、compaction だけでは不十分です。何の工夫もなければ、Opus 4.5 のような最先端のコーディングモデルであっても、Claude Agent SDK 上で複数のコンテキストウィンドウにわたってループ実行された場合、「claude.ai のクローンを作って」といった高レベルなプロンプトだけを与えられただけでは、本番品質のウェブアプリを構築するには至りません。

Claude の失敗は2つのパターンとして現れました。第一に、エージェントは一度にあまりに多くのことをやろうとする傾向、すなわち本質的にアプリをワンショットで作ろうとする傾向がありました。これはしばしば、モデルが実装の途中でコンテキストを使い果たし、機能が半分しか実装されておらず、かつドキュメント化もされていない状態で次のセッションを始めさせる結果につながりました。すると、エージェントは何が起きたのかを推測しなければならず、基本的なアプリを再び動作させようとするのに多大な時間を費やすことになりました。これは compaction があっても発生します。compaction は必ずしも完全に明確な指示を次のエージェントに引き継ぐわけではないからです。

第二の失敗モードは、プロジェクトの後半でよく発生しました。いくつかの機能がすでに構築された後、後続のエージェントのインスタンスがあたりを見回し、進捗があったことを確認して、作業は完了したと宣言してしまうのです。

これは問題を2つの部分に分解します。第一に、与えられたプロンプトが必要とするすべての機能の土台を築く初期環境をセットアップする必要があり、これによってエージェントがステップ・バイ・ステップ、機能ごとに作業を進められるようにします。第二に、各エージェントに対して、目標に向けて段階的な進捗を上げると同時に、セッションの終わりには環境をクリーンな状態（clean state）で残すように促すべきです。「クリーンな状態」とは、main ブランチへのマージに適した類のコードを意味します。すなわち、重大なバグがなく、コードが整然としていてよくドキュメント化されており、概して開発者が無関係な散らかりを片付ける必要なく新しい機能の作業に容易に着手できる状態です。

社内で実験する中で、私たちはこれらの問題に二部構成の解決策で対処しました。

1. 初期化エージェント（Initializer agent）: 最初のエージェントセッションは、初期環境をセットアップするようモデルに求める専用のプロンプトを使用します。すなわち、init.sh スクリプト、エージェントが行ったことのログを保持する claude-progress.txt ファイル、そしてどのファイルが追加されたかを示す最初の git commit です。

2. コーディングエージェント（Coding agent）: 以降のすべてのセッションは、モデルに段階的な進捗を上げ、その後に構造化された更新を残すよう求めます。

ここでの重要な洞察は、新しいコンテキストウィンドウで作業を始める際に、エージェントが作業の状態を素早く理解できる方法を見つけたことであり、これは git 履歴と並んで claude-progress.txt ファイルによって実現されます。これらのプラクティスのヒントは、有能なソフトウェアエンジニアが日々行っていることを知ることから得られました。

## 環境管理

更新された Claude 4 プロンプティングガイドにおいて、私たちは複数コンテキストウィンドウのワークフローに関するいくつかのベストプラクティスを共有しました。その中には、「最初のコンテキストウィンドウのためには異なるプロンプト」を使用するハーネス構造が含まれていました。この「異なるプロンプト」は、初期化エージェントに対して、将来のコーディングエージェントが効果的に作業するために必要となるすべてのコンテキストを備えた環境をセットアップするよう要求します。ここでは、そのような環境の主要なコンポーネントのいくつかについて、より深く掘り下げます。

### 機能リスト

エージェントがアプリをワンショットで作ろうとしたり、プロジェクトを早まって完了とみなしたりする問題に対処するため、私たちは初期化エージェントに対して、ユーザーの最初のプロンプトを拡張する形で、機能要件を網羅的に記したファイルを書くよう促しました。claude.ai クローンの例では、これは200を超える機能を意味しました。たとえば「ユーザーは新しいチャットを開き、クエリを入力し、Enter を押して、AIの応答を見ることができる」といったものです。これらの機能はすべて、当初「failing（失敗）」とマークされており、後続のコーディングエージェントが完全な機能性がどのようなものかを明確に把握できるようになっていました。

```json
{
    "category": "functional",
    "description": "New chat button creates a fresh conversation",
    "steps": [
      "Navigate to main interface",
      "Click the 'New Chat' button",
      "Verify a new conversation is created",
      "Check that chat area shows welcome state",
      "Verify conversation appears in sidebar"
    ],
    "passes": false
}
```

私たちはコーディングエージェントに対して、このファイルを編集する際は passes フィールドのステータスを変更することによってのみ行うよう促し、「テストを削除または編集することは受け入れられない。なぜならそれは機能の欠落やバグにつながる可能性があるからだ」といった強い言い回しの指示を使用します。いくつかの実験を経て、私たちはこれに JSON を使うことに落ち着きました。というのも、モデルは Markdown ファイルと比べて、JSON ファイルを不適切に変更したり上書きしたりする可能性が低いからです。

### 段階的な進捗

この初期環境の足場が整った上で、次の反復であるコーディングエージェントには、一度に1つの機能だけに取り組むよう求めました。この段階的なアプローチは、エージェントが一度にあまりに多くのことをやろうとする傾向に対処する上で、決定的に重要であることが判明しました。

段階的に作業するようになっても、モデルがコード変更を行った後に環境をクリーンな状態で残すことは依然として不可欠です。私たちの実験では、この振る舞いを引き出す最良の方法は、モデルに対して、説明的なコミットメッセージとともに進捗を git にコミットし、進捗ファイルに進捗の要約を書くよう求めることだとわかりました。これにより、モデルは git を使って不適切なコード変更を元に戻し、コードベースの動作する状態を復元できるようになりました。

これらのアプローチは効率性も高めました。というのも、エージェントが何が起きたのかを推測したり、基本的なアプリを再び動作させようとするのに時間を費やしたりする必要をなくしたからです。

### テスト

私たちが観察したもう1つの主要な失敗モードは、Claude が適切なテストをせずに機能を完了とマークしてしまう傾向でした。明示的なプロンプトがない場合、Claude はコード変更を行い、さらには開発サーバーに対してユニットテストや curl コマンドでテストを行うことさえありましたが、その機能がエンドツーエンドでは動作していないことを認識できませんでした。

ウェブアプリを構築するケースにおいては、ブラウザ自動化ツールを使い、人間のユーザーが行うようにすべてのテストを行うよう明示的に促されると、Claude はおおむね機能をエンドツーエンドで検証することにうまく対応しました。

Claude が claude.ai クローンをテストする中で Puppeteer MCP サーバーを通じて撮影したスクリーンショット。

Claude にこの種のテストツールを提供することは、パフォーマンスを劇的に向上させました。というのも、エージェントがコードだけからは明らかでなかったバグを特定し、修正できるようになったからです。

いくつかの問題は依然として残っています。たとえば、Claude のビジョン（視覚）やブラウザ自動化ツールの限界により、あらゆる種類のバグを特定するのが難しいといった点です。たとえば、Claude は Puppeteer MCP を通じてブラウザネイティブのアラートモーダルを見ることができず、その結果、これらのモーダルに依存する機能はバグが多くなる傾向がありました。

## 状況把握（Getting up to speed）

以上のすべてが整った上で、すべてのコーディングエージェントは、自分の置かれた状況を把握するために一連のステップを実行するよう促されます。中にはかなり基本的なものもありますが、それでも役に立ちます。

1. pwd を実行して、自分が作業しているディレクトリを確認する。あなたはこのディレクトリ内のファイルしか編集できない。
2. git ログと進捗ファイルを読み、最近何が作業されたかを把握する。
3. 機能リストファイルを読み、まだ完了していない最優先の機能を選んで取り組む。

このアプローチは、Claude がコードのテスト方法を考え出す必要がなくなるため、毎セッションでいくらかのトークンを節約します。また、初期化エージェントに対して、開発サーバーを実行できる init.sh スクリプトを書くよう求め、新しい機能を実装する前に基本的なエンドツーエンドテストを通すようにするのも役立ちます。

claude.ai クローンのケースでは、これはエージェントが常にローカルの開発サーバーを起動し、Puppeteer MCP を使って新しいチャットを開始し、メッセージを送信し、応答を受け取ることを意味しました。これにより、Claude はアプリが壊れた状態で残されていないかを素早く特定し、既存のバグを直ちに修正できるようになりました。もしエージェントが代わりに新しい機能の実装を始めていたら、おそらく問題を悪化させていたでしょう。

これらすべてを踏まえると、典型的なセッションは以下のアシスタントメッセージで始まります。

```
[Assistant] I'll start by getting my bearings and understanding the current state of the project.
[Tool Use] <bash - pwd>
[Tool Use] <read - claude-progress.txt>
[Tool Use] <read - feature_list.json>
[Assistant] Let me check the git log to see recent work.
[Tool Use] <bash - git log --oneline -20>
[Assistant] Now let me check if there's an init.sh script to restart the servers.
<Starts the development server>
[Assistant] Excellent! Now let me navigate to the application and verify that some fundamental features are still working.
<Tests basic functionality>
[Assistant] Based on my verification testing, I can see that the fundamental functionality is working well. The core chat features, theme switching, conversation loading, and error handling are all functioning correctly. Now let me review the tests.json file more comprehensively to understand what needs to be implemented next.
<Starts work on a new feature>
```

### エージェントの失敗モードと解決策

| 問題 | 初期化エージェントの振る舞い | コーディングエージェントの振る舞い |
|---------|----------------------------|----------------------|
| Claude がプロジェクト全体に対して早すぎる段階で勝利宣言してしまう。 | 機能リストファイルをセットアップする。すなわち、入力仕様に基づいて、エンドツーエンドの機能説明のリストを記した構造化された JSON ファイルをセットアップする。 | セッションの開始時に機能リストファイルを読む。取り組む機能を1つだけ選ぶ。 |
| Claude が環境をバグやドキュメント化されていない進捗のある状態で残してしまう。 | 最初の git リポジトリと進捗ノートファイルが書かれる。 | セッションを進捗ノートファイルと git コミットログを読むことから始め、開発サーバーで基本的なテストを実行してドキュメント化されていないバグを検出する。セッションは git コミットと進捗の更新を書くことで終える。 |
| Claude が機能を早まって完了とマークしてしまう。 | 機能リストファイルをセットアップする。 | すべての機能を自己検証する。慎重なテストの後でのみ機能を「passing」とマークする。 |
| Claude がアプリの実行方法を考え出すのに時間を費やさなければならない。 | 開発サーバーを実行できる init.sh スクリプトを書く。 | セッションを init.sh を読むことから始める。 |

## 今後の取り組み（Future work）

この研究は、長期実行エージェントのハーネスにおいて、モデルが多数のコンテキストウィンドウをまたいで段階的な進捗を上げられるようにするための、1つの可能な解決策のセットを示しています。しかし、未解決の問題は残っています。

最も顕著なのは、単一の汎用コーディングエージェントがコンテキストをまたいで最も良いパフォーマンスを発揮するのか、それともマルチエージェントアーキテクチャによってより良いパフォーマンスを達成できるのかが、依然として不明であるという点です。テストエージェント、品質保証（QA）エージェント、コードクリーンアップエージェントといった専門化されたエージェントが、ソフトウェア開発ライフサイクル全体のサブタスクにおいて、さらに良い仕事をする可能性は十分に考えられます。

加えて、このデモはフルスタックのウェブアプリ開発に最適化されています。今後の方向性としては、これらの知見を他の分野に一般化することが挙げられます。これらの教訓の一部またはすべては、たとえば科学研究や金融モデリングなどで必要とされる種類の長期実行のエージェント的タスクにも適用できる可能性が高いでしょう。
