# Writing effective tools for agents — with agents（エージェントのための効果的なツールの書き方 — エージェントとともに）

**URL:** <https://www.anthropic.com/engineering/writing-tools-for-agents>  
**対応ドメイン:** D2 / D4  
**優先度:** S

---

公開日: 2025年9月11日

エージェントは、私たちが与えるツールと同じだけの効果しか発揮できません。本記事では、高品質なツールと評価（evaluation）の書き方、そして Claude を使って Claude 自身のためにツールを最適化することでパフォーマンスを高める方法を共有します。

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro) は、現実世界のタスクを解決するために、LLM エージェントに数百ものツールを与えて強化できる可能性があります。しかし、それらのツールを最大限に効果的にするにはどうすればよいのでしょうか。

本記事では、さまざまなエージェント型 AI システムにおいてパフォーマンスを向上させるための、私たちの最も効果的な手法を説明します。

まず、次の方法から取り上げます。

* ツールのプロトタイプを構築してテストする
* エージェントを用いて、ツールの包括的な評価を作成・実行する
* Claude Code のようなエージェントと協働して、ツールのパフォーマンスを自動的に向上させる

最後に、これまでの過程で見出した、高品質なツールを書くための重要な原則をまとめます。

* 実装すべき（そして実装すべきでない）適切なツールを選ぶ
* 機能の明確な境界を定義するためにツールを namespacing する
* ツールからエージェントへ意味のあるコンテキストを返す
* トークン効率の観点からツールのレスポンスを最適化する
* ツールの説明（description）と仕様（spec）をプロンプトエンジニアリングする

## ツールとは何か

コンピューティングにおいて、決定論的（deterministic）なシステムは、同一の入力が与えられれば毎回同じ出力を生成します。一方、エージェントのような *非決定論的（non-deterministic）* なシステムは、同じ開始条件であっても多様なレスポンスを生成しうります。

従来のソフトウェアを書くとき、私たちは決定論的システム同士の間で契約（contract）を確立しています。たとえば、`getWeather("NYC")` のような関数呼び出しは、呼び出されるたびに毎回まったく同じ方法でニューヨーク市の天気を取得します。

ツールは、決定論的システムと非決定論的なエージェントとの間の契約を反映した、新しい種類のソフトウェアです。ユーザーが「今日は傘を持っていくべき？」と尋ねたとき、エージェントは天気ツールを呼び出すかもしれませんし、一般知識から答えるかもしれませんし、あるいは先に場所を確認する質問をするかもしれません。場合によっては、エージェントがハルシネーションを起こしたり、ツールの使い方を把握できなかったりすることさえあります。

これは、エージェント向けにソフトウェアを書く際のアプローチを根本的に考え直す必要があることを意味します。つまり、他の開発者やシステム向けに関数や API を書くのと同じやり方でツールや [MCP サーバー](https://modelcontextprotocol.io/) を書くのではなく、エージェントのために設計する必要があるということです。

私たちの目標は、エージェントがツールを使ってさまざまな成功戦略を追求することで、幅広いタスクを効果的に解決できる範囲（surface area）を拡大することです。幸いなことに、私たちの経験では、エージェントにとって最も「人間工学的（ergonomic）」なツールは、人間にとっても驚くほど直感的に把握できるものになる傾向があります。

## ツールの書き方

このセクションでは、エージェントに与えるツールを書くこと、そして改善することの両方において、エージェントとどのように協働できるかを説明します。まずはツールのクイックプロトタイプを立ち上げ、ローカルでテストすることから始めます。次に、その後の変更を測定するために包括的な評価を実行します。エージェントと並走しながら、評価と改善のプロセスを繰り返すことで、エージェントが現実世界のタスクで高いパフォーマンスを達成するまで進めていけます。

### プロトタイプの構築

実際に自分で手を動かしてみなければ、どのツールがエージェントにとって人間工学的で、どのツールがそうでないかを予測するのは難しいものです。まずはツールのクイックプロトタイプを立ち上げましょう。ツールの作成に [Claude Code](https://www.anthropic.com/claude-code) を使う場合（場合によっては one-shot で）、ツールが依存するソフトウェアライブラリ・API・SDK（場合によっては [MCP SDK](https://modelcontextprotocol.io/docs/sdk) を含む）のドキュメントを Claude に与えると役立ちます。LLM フレンドリーなドキュメントは、公式ドキュメントサイト上のフラットな `llms.txt` ファイルとして提供されていることがよくあります（私たちの [API のもの](https://docs.anthropic.com/llms.txt) はこちらです）。

ツールを [ローカル MCP サーバー](https://modelcontextprotocol.io/docs/develop/connect-local-servers) や [Desktop extension](https://www.anthropic.com/engineering/desktop-extensions)（DXT）でラップすると、Claude Code または Claude Desktop アプリでツールに接続してテストできるようになります。

ローカル MCP サーバーを Claude Code に接続するには、`claude mcp add <name> <command> [args...]` を実行します。

ローカル MCP サーバーまたは DXT を Claude Desktop アプリに接続するには、それぞれ `Settings > Developer` または `Settings > Extensions` に移動します。

ツールは、プログラムによるテストのために、[Anthropic API](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/overview) の呼び出しに直接渡すこともできます。

ツールを自分自身でテストして、粗い部分（rough edges）を特定しましょう。ユーザーからフィードバックを集め、ツールで実現したいユースケースやプロンプトについての直感を養いましょう。

### 評価の実行

次に、評価を実行して Claude がツールをどれだけうまく使えるかを測定する必要があります。まずは、現実世界の用途に根ざした評価タスクを大量に生成することから始めます。結果の分析やツールの改善方法の判断を手伝ってもらうために、エージェントと協働することをお勧めします。このプロセスのエンドツーエンドの流れは、私たちの [tool evaluation cookbook](https://platform.claude.com/cookbook/tool-evaluation-tool-evaluation) でご覧いただけます。

**評価タスクの生成**

初期のプロトタイプがあれば、Claude Code はツールを素早く探索し、何十ものプロンプトとレスポンスのペアを作成できます。プロンプトは現実世界の用途に着想を得たものであるべきで、現実的なデータソースやサービス（たとえば社内のナレッジベースやマイクロサービス）に基づくべきです。ツールを十分な複雑さでストレステストできない、過度に単純で表面的な「サンドボックス」環境は避けることをお勧めします。優れた評価タスクは、複数回（場合によっては何十回も）のツール呼び出しを必要とすることがあります。

優れたタスクの例をいくつか挙げます。

* 最新の Acme Corp プロジェクトについて話し合うために、来週 Jane とのミーティングをスケジュールしてください。前回のプロジェクト計画ミーティングのメモを添付し、会議室を予約してください。
* 顧客 ID 9182 が、1 回の購入試行に対して 3 回課金されたと報告しました。関連するすべてのログエントリを見つけ、同じ問題の影響を受けた他の顧客がいないか判断してください。
* 顧客 Sarah Chen が解約リクエストを送信しました。リテンションオファーを準備してください。次のことを判断してください。(1) なぜ離脱しようとしているのか、(2) どのようなリテンションオファーが最も魅力的か、(3) オファーを出す前に把握しておくべきリスク要因。

そして、より弱いタスクの例をいくつか挙げます。

* 来週 jane@acme.corp とのミーティングをスケジュールしてください。
* 支払いログから `purchase_complete` と `customer_id=9182` を検索してください。
* 顧客 ID 45892 による解約リクエストを見つけてください。

各評価プロンプトには、検証可能なレスポンスまたは結果（outcome）を組み合わせるべきです。検証器（verifier）は、グラウンドトゥルースとサンプリングされたレスポンスとの間の単純な完全一致文字列比較のように簡素なものでも、Claude にレスポンスを判定させるような高度なものでもかまいません。フォーマット・句読点・有効な言い換えといった些末な違いによって正しいレスポンスを却下してしまう、過度に厳格な検証器は避けましょう。

各プロンプトとレスポンスのペアについては、任意で、タスクを解く際にエージェントが呼び出すと期待するツールを指定することもできます。これにより、評価中にエージェントが各ツールの目的をうまく把握できているかどうかを測定できます。ただし、タスクを正しく解くための有効な経路は複数存在しうるため、戦略を過度に指定したり、特定の戦略に過剰適合（overfit）させたりしないようにしてください。

**評価の実行**

評価は、LLM API を直接呼び出してプログラムで実行することをお勧めします。シンプルな agentic loop（LLM API 呼び出しとツール呼び出しを交互に繰り返す `while` ループ）を使い、評価タスクごとに 1 つのループを回します。各評価エージェントには、単一のタスクプロンプトとあなたのツールを与えるべきです。

評価エージェントのシステムプロンプトでは、（検証のための）構造化されたレスポンスブロックだけでなく、推論（reasoning）ブロックとフィードバック（feedback）ブロックも出力するよう指示することをお勧めします。これらをツール呼び出しブロックやレスポンスブロックの *前に* 出力させるよう指示すると、chain-of-thought（CoT）の挙動が引き出され、LLM の実効的な知能が高まる可能性があります。

評価を Claude で実行している場合は、[interleaved thinking](https://docs.anthropic.com/en/docs/build-with-claude/extended-thinking#interleaved-thinking) をオンにすることで、同様の機能を「すぐに使える」形で利用できます。これにより、エージェントが特定のツールを呼び出す／呼び出さない理由を探り、ツールの説明や仕様における具体的な改善点を浮き彫りにできます。

トップレベルの精度（accuracy）に加えて、個々のツール呼び出しおよびタスクの総実行時間、ツール呼び出しの総数、総トークン消費量、ツールエラーといった、その他のメトリクスを収集することをお勧めします。ツール呼び出しを追跡することで、エージェントが追求する一般的なワークフローを明らかにでき、ツールを統合（consolidate）する機会をいくらか見出せます。

**結果の分析**

エージェントは、矛盾したツールの説明から非効率なツールの実装、紛らわしいツールのスキーマに至るまで、あらゆる問題を見つけ出しフィードバックを提供してくれる、頼れるパートナーです。ただし、エージェントがフィードバックやレスポンスにおいて *省略している* ことのほうが、含めていることよりも重要である場合が多いことを心に留めておいてください。LLM は必ずしも[本心を語るとは限りません](https://www.anthropic.com/research/tracing-thoughts-language-model)。

エージェントがどこで行き詰まったり混乱したりするかを観察しましょう。評価エージェントの推論やフィードバック（あるいは CoT）を読み込み、粗い部分を特定しましょう。生のトランスクリプト（ツール呼び出しやツールのレスポンスを含む）を確認し、エージェントの CoT に明示的に記述されていない挙動を捉えましょう。行間を読むこと。評価エージェントが必ずしも正解や戦略を知っているわけではないことを忘れないでください。

ツール呼び出しのメトリクスを分析しましょう。冗長なツール呼び出しが多い場合は、ページネーションやトークン制限パラメータの適正化（rightsizing）が必要であることを示唆しているかもしれません。無効なパラメータによるツールエラーが多い場合は、ツールにより明確な説明やより良い例が必要であることを示唆しているかもしれません。Claude の [web search ツール](https://www.anthropic.com/news/web-search) をローンチしたとき、私たちは Claude が不必要にツールの `query` パラメータへ `2025` を付け加えており、それが検索結果を偏らせてパフォーマンスを低下させていることを特定しました（ツールの説明を改善することで、Claude を正しい方向へ導きました）。

### エージェントとの協働

エージェントに結果を分析させ、あなたのためにツールを改善させることさえできます。評価エージェントから得たトランスクリプトを連結して、Claude Code に貼り付けるだけです。Claude はトランスクリプトの分析と、多数のツールを一度にリファクタリングすることの達人です。たとえば、新しい変更が加えられた際に、ツールの実装と説明の整合性が保たれるようにすることなどです。

実のところ、本記事のアドバイスのほとんどは、Claude Code を使って私たちの内部ツールの実装を繰り返し最適化することから得られました。私たちの評価は、実際のプロジェクト・ドキュメント・メッセージを含む、社内ワークフローの複雑さを反映した社内ワークスペースの上に構築されました。

私たちは、「トレーニング」用の評価に過剰適合していないことを確認するために、ホールドアウト（held-out）テストセットに頼りました。これらのテストセットからは、「エキスパート」によるツール実装で達成できた水準を超えて、さらなるパフォーマンス改善を引き出せることが明らかになりました。それらのツールが研究者によって手作業で書かれたものであれ、Claude 自身によって生成されたものであれ、同様でした。

## 効果的なツールを書くための原則

このセクションでは、私たちの学びを、効果的なツールを書くためのいくつかの指針となる原則へと凝縮します。

### エージェントに適したツールを選ぶ

ツールが多ければ必ずしも良い結果につながるわけではありません。私たちが観察してきたよくある誤りは、既存のソフトウェア機能や API エンドポイントを単にラップしただけのツール（そのツールがエージェントに適しているかどうかにかかわらず）です。これは、エージェントが従来のソフトウェアとは異なる「アフォーダンス（affordances）」を持つ、つまりそれらのツールで取りうる潜在的なアクションを認識する仕方が異なるためです。

LLM エージェントには限られた「コンテキスト」しかありません（つまり、一度に処理できる情報量には限りがあります）。一方、コンピュータのメモリは安価で潤沢です。アドレス帳から連絡先を検索するタスクを考えてみましょう。従来のソフトウェアプログラムは、連絡先のリストを一件ずつ効率的に保存・処理し、一件ずつ確認してから次に進むことができます。

しかし、もし LLM エージェントが、すべての連絡先を返すツールを使い、その後それぞれをトークン単位で読み込まなければならないとしたら、無関係な情報のために限られたコンテキスト空間を浪費していることになります（アドレス帳から連絡先を探すのに、各ページを上から順に読んでいくところを想像してみてください。これはブルートフォース検索です）。エージェントにとっても人間にとっても、より優れていてより自然なアプローチは、まず関連するページに飛ぶこと（おそらくアルファベット順に見つけること）です。

特定の影響度の高いワークフロー（評価タスクに一致するもの）を狙った、よく考えられた少数のツールを構築し、そこからスケールアップしていくことをお勧めします。アドレス帳のケースでは、`list_contacts` ツールではなく、`search_contacts` や `message_contact` といったツールを実装することを選ぶとよいでしょう。

ツールは機能を統合し、内部で *複数の* 個別の操作（または API 呼び出し）を扱うことができます。たとえば、ツールはツールのレスポンスを関連メタデータで充実させたり、頻繁に連鎖する複数ステップのタスクを単一のツール呼び出しで処理したりできます。

いくつか例を挙げます。

* `list_users`、`list_events`、`create_event` の各ツールを実装する代わりに、空き状況を見つけてイベントをスケジュールする `schedule_event` ツールの実装を検討しましょう。
* `read_logs` ツールを実装する代わりに、関連するログ行とその周辺のコンテキストだけを返す `search_logs` ツールの実装を検討しましょう。
* `get_customer_by_id`、`list_transactions`、`list_notes` の各ツールを実装する代わりに、顧客の最近かつ関連性の高い情報をすべて一度にまとめる `get_customer_context` ツールを実装しましょう。

構築する各ツールが、明確で独自の目的を持つようにしましょう。ツールは、同じ基盤リソースへのアクセスが与えられたときに人間が行うのとほぼ同じように、エージェントがタスクを細分化して解決できるようにし、同時に、中間出力によって消費されていたであろうコンテキストを削減するべきです。

ツールが多すぎたり、ツール同士が重複していたりすると、エージェントが効率的な戦略を追求するのを妨げることにもなりかねません。構築する（あるいは構築しない）ツールを慎重かつ選択的に計画することは、本当に大きな見返りをもたらします。

### ツールの namespacing

あなたの AI エージェントは、（他の開発者によるものも含め）数十の MCP サーバーと数百もの異なるツールにアクセスできるようになる可能性があります。ツールが機能的に重複していたり、目的が曖昧だったりすると、エージェントはどれを使うべきか混乱してしまいます。

Namespacing（関連するツールを共通のプレフィックスの下にグループ化すること）は、多数のツール間の境界を明確に区切るのに役立ちます。MCP クライアントは、これをデフォルトで行うこともあります。たとえば、サービス別（例：`asana_search`、`jira_search`）やリソース別（例：`asana_projects_search`、`asana_users_search`）にツールを namespacing すると、エージェントが適切なタイミングで適切なツールを選択するのに役立ちます。

プレフィックスベースの namespacing とサフィックスベースの namespacing のどちらを選ぶかが、私たちのツール使用評価において無視できない影響を持つことがわかっています。影響は LLM によって異なるため、あなた自身の評価に応じて命名スキームを選ぶことをお勧めします。

エージェントは、間違ったツールを呼び出したり、正しいツールを間違ったパラメータで呼び出したり、ツールの呼び出し回数が少なすぎたり、ツールのレスポンスを誤って処理したりすることがあります。タスクの自然な細分化を反映した名前を持つツールを選択的に実装することで、エージェントのコンテキストに読み込まれるツールとツールの説明の数を減らすと同時に、エージェントのコンテキストから本来のエージェント的計算をツール呼び出し自体へとオフロードできます。これにより、エージェントが誤りを犯す全体的なリスクが低減します。

### ツールから意味のあるコンテキストを返す

同じ趣旨で、ツールの実装は、シグナルの高い（high signal）情報だけをエージェントへ返すよう注意するべきです。柔軟性よりもコンテキスト上の関連性を優先し、低レベルの技術的識別子（たとえば `uuid`、`256px_image_url`、`mime_type` など）は避けるべきです。`name`、`image_url`、`file_type` のようなフィールドのほうが、エージェントの下流のアクションやレスポンスに直接情報を与える可能性がはるかに高いです。

また、エージェントは、難解な識別子よりも、自然言語の名前・用語・識別子のほうをはるかにうまく扱える傾向があります。任意の英数字 UUID を、より意味的に意味があり解釈しやすい言語（あるいは 0 始まりの ID スキームでさえ）に解決するだけで、ハルシネーションを減らすことによって、検索（retrieval）タスクにおける Claude の精度が大幅に向上することがわかっています。

場合によっては、下流のツール呼び出しを起動するためだけであっても、エージェントが自然言語と技術的識別子の両方の出力を扱える柔軟性を必要とすることがあります（たとえば `search_user(name='jane')` → `send_message(id=12345)`）。シンプルな `response_format` enum パラメータをツールに公開することで、その両方を可能にできます。これにより、エージェントはツールが `"concise"`（簡潔）または `"detailed"`（詳細）のどちらのレスポンスを返すかを制御できます（下の画像を参照）。

GraphQL のように、受け取りたい情報の各要素を正確に選べるようにして、さらに高い柔軟性を持たせるためにフォーマットを追加することもできます。以下は、ツールレスポンスの冗長性を制御する ResponseFormat enum の例です。

```
enum ResponseFormat {
   DETAILED = "detailed",
   CONCISE = "concise"
}
```

Slack のスレッドとスレッド返信は、一意の `thread_ts` によって識別され、スレッド返信を取得するにはこれが必要です。`thread_ts` やその他の ID（`channel_id`、`user_id`）は `"detailed"` なツールレスポンスから取得でき、それらを必要とするさらなるツール呼び出しを可能にします。`"concise"` なツールレスポンスはスレッドの内容だけを返し、ID は除外します。この例では、`"concise"` なツールレスポンスを使うことでトークンを約 ⅓ に抑えています。

ツールのレスポンス構造（たとえば XML、JSON、Markdown）でさえ、評価のパフォーマンスに影響を与えることがあります。万能の解決策（one-size-fits-all）は存在しません。これは、LLM が次トークン予測（next-token prediction）で訓練されており、トレーニングデータに一致するフォーマットのほうがうまく機能する傾向があるためです。最適なレスポンス構造は、タスクやエージェントによって大きく異なります。あなた自身の評価に基づいて、最適なレスポンス構造を選ぶことをお勧めします。

### トークン効率の観点でツールのレスポンスを最適化する

コンテキストの質を最適化することは重要です。しかし、ツールのレスポンスでエージェントに返されるコンテキストの *量* を最適化することも、同じくらい重要です。

大量のコンテキストを使い切ってしまう可能性のあるツールレスポンスについては、ページネーション・範囲選択（range selection）・フィルタリング・切り捨て（truncation）を、適切なデフォルトパラメータ値とともに何らか組み合わせて実装することをお勧めします。Claude Code では、ツールレスポンスをデフォルトで 25,000 トークンに制限しています。エージェントの実効的なコンテキスト長は時間とともに拡大していくと見込んでいますが、コンテキスト効率の高いツールの必要性は残り続けるでしょう。

レスポンスを切り捨てることにした場合は、必ず役立つ指示によってエージェントを導いてください。ナレッジ検索タスクにおいて、単一の広範な検索の代わりに、小さく的を絞った検索を何度も行うといった、よりトークン効率の高い戦略を追求するよう、エージェントを直接後押しできます。同様に、ツール呼び出しがエラー（たとえば入力検証中）を発生させた場合は、不透明なエラーコードやトレースバックではなく、具体的で行動可能な改善内容を明確に伝えるように、エラーレスポンスをプロンプトエンジニアリングできます。

### ツールの説明をプロンプトエンジニアリングする

さて、ツールを改善するための最も効果的な手法のひとつにたどり着きました。それは、ツールの説明（description）と仕様（spec）をプロンプトエンジニアリングすることです。これらはエージェントのコンテキストに読み込まれるため、それらが総体としてエージェントを効果的なツール呼び出しの挙動へと導くことができます。

ツールの説明や仕様を書くときは、チームの新入社員にそのツールをどう説明するかを考えてみてください。あなたが暗黙のうちに持ち込んでいるかもしれないコンテキスト（特殊なクエリフォーマット、ニッチな用語の定義、基盤リソース間の関係）を考え、それを明示的にしましょう。期待される入力と出力を明確に記述し（厳格なデータモデルで強制し）、曖昧さを避けましょう。とりわけ、入力パラメータは曖昧さのない名前を付けるべきです。`user` という名前のパラメータの代わりに、`user_id` という名前のパラメータを試してみましょう。

評価があれば、プロンプトエンジニアリングの効果をより高い確信を持って測定できます。ツールの説明への小さな改良でさえ、劇的な改善をもたらすことがあります。Claude Sonnet 3.5 は、私たちがツールの説明に精密な改良を加えた後、[SWE-bench Verified](https://www.anthropic.com/engineering/swe-bench-sonnet) 評価で最先端（state-of-the-art）のパフォーマンスを達成し、エラー率を劇的に低減してタスク完了率を向上させました。

ツール定義に関するその他のベストプラクティスは、私たちの [Developer Guide](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#best-practices-for-tool-definitions) でご覧いただけます。Claude 向けのツールを構築している場合は、ツールが Claude の[システムプロンプト](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use#tool-use-system-prompt)へどのように動的に読み込まれるかについても読むことをお勧めします。最後に、MCP サーバー向けのツールを書いている場合は、[tool annotations](https://modelcontextprotocol.io/specification/2025-06-18/server/tools) が、どのツールがオープンワールドのアクセスを必要とするか、あるいは破壊的な変更を行うかを開示するのに役立ちます。

## 今後の展望

エージェント向けの効果的なツールを構築するには、私たちのソフトウェア開発の慣行を、予測可能で決定論的なパターンから、非決定論的なパターンへと方向転換させる必要があります。

本記事で説明した、反復的で評価駆動（evaluation-driven）のプロセスを通じて、私たちはツールを成功させる要因に一貫したパターンを見出しました。効果的なツールは、意図的かつ明確に定義され、エージェントのコンテキストを賢明に使い、多様なワークフローの中で組み合わせることができ、エージェントが現実世界のタスクを直感的に解決できるようにします。

将来的には、エージェントが世界とやり取りするための具体的なメカニズムは進化していくと見込んでいます。MCP プロトコルのアップデートから、基盤となる LLM そのもののアップグレードに至るまでです。エージェント向けのツールを改善するための、体系的で評価駆動なアプローチをとることで、エージェントがより有能になるにつれて、それらが使うツールもそれと並んで進化していくことを確実にできます。
