# Effective context engineering for AI agents（AI エージェントのための効果的なコンテキストエンジニアリング）

**URL:** <https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents>  
**対応ドメイン:** D5  
**優先度:** S

---

応用 AI においてプロンプトエンジニアリングが数年にわたり注目を集めてきたあと、新たな用語が台頭してきた。それが context engineering である。言語モデルを用いた開発は、プロンプトのために適切な単語やフレーズを見つけることよりも、「どのようなコンテキストの構成が、私たちのモデルに望ましい振る舞いを最も生み出しやすいか」というより広い問いに答えることへと、その重点が移りつつある。

コンテキストとは、大規模言語モデル（LLM）からサンプリングを行う際に含まれるトークンの集合を指す。ここでの工学的課題は、望ましい結果を一貫して達成するために、LLM 固有の制約に対してそれらのトークンの有用性を最適化することである。LLM を効果的に扱うには、多くの場合コンテキストの中で考えること、言い換えれば、ある時点で LLM が利用できる全体的な状態と、その状態がどのような潜在的振る舞いを生み出しうるかを考慮することが求められる。

本稿では、新たに生まれつつある context engineering という技芸を探究し、操作可能で効果的なエージェントを構築するための洗練されたメンタルモデルを提示する。

## Context engineering vs. prompt engineering（コンテキストエンジニアリング 対 プロンプトエンジニアリング）

Anthropic では、context engineering をプロンプトエンジニアリングの自然な発展形と捉えている。プロンプトエンジニアリングとは、最適な結果を得るために LLM への指示を記述し整理する手法を指す（概要と有用なプロンプトエンジニアリング戦略については当社のドキュメントを参照）。context engineering とは、LLM の推論中に最適なトークン（情報）の集合を取捨選択し維持するための一連の戦略を指し、プロンプト以外の場所に入り込みうるあらゆる情報も含む。

LLM を用いた開発の初期には、日常的なチャット対話以外のユースケースの大半が、ワンショットの分類やテキスト生成タスク向けに最適化されたプロンプトを必要としたため、プロンプティングが AI エンジニアリング作業の最大の構成要素であった。その名が示すとおり、プロンプトエンジニアリングの主たる焦点は、効果的なプロンプト、とりわけシステムプロンプトをどう書くかにあった。しかし、複数ターンの推論やより長い時間軸にわたって動作する、より高性能なエージェントの構築へと向かうにつれて、私たちはコンテキスト状態全体（システム指示、ツール、Model Context Protocol（MCP）、外部データ、メッセージ履歴など）を管理するための戦略を必要とする。

ループ内で動作するエージェントは、次の推論ターンに関連しうるデータをますます多く生成し、この情報は周期的に取捨選択されなければならない。context engineering とは、絶え間なく変化しうる情報の宇宙の中から、限られたコンテキストウィンドウに何を入れるかを取捨選択する技芸であり科学である。

> プロンプトを書くという離散的なタスクとは対照的に、context engineering は反復的であり、モデルに何を渡すかを決めるたびに取捨選択のフェーズが発生する。

## Why context engineering is important to building capable agents（なぜ context engineering が高性能なエージェント構築に重要なのか）

LLM はその速度と、ますます大量のデータを扱う能力を持つにもかかわらず、人間と同様にある時点で焦点を失ったり混乱を来したりすることを私たちは観察してきた。needle-in-a-haystack 形式のベンチマークに関する研究は、context rot という概念を明らかにした。すなわち、コンテキストウィンドウ内のトークン数が増えるにつれて、そのコンテキストから情報を正確に呼び出すモデルの能力は低下する。

一部のモデルは他よりも緩やかな劣化を示すものの、この特性はすべてのモデルに現れる。したがってコンテキストは、限界的な収穫が逓減する有限の資源として扱われなければならない。限られたワーキングメモリ容量を持つ人間と同様に、LLM は大量のコンテキストを解析する際に引き出す「アテンション予算（attention budget）」を持つ。新たに導入されるトークンはいずれもこの予算をいくらか消費し、LLM が利用できるトークンを慎重に取捨選択する必要性を高める。

このアテンションの希少性は、LLM のアーキテクチャ上の制約に起因する。LLM は transformer アーキテクチャに基づいており、これによってすべてのトークンがコンテキスト全体にわたって他のすべてのトークンに注意を向けることができる。これは n 個のトークンに対して n² 個のペアワイズな関係をもたらす。

コンテキスト長が増えるにつれて、これらのペアワイズな関係を捉えるモデルの能力は薄く引き伸ばされ、コンテキストサイズとアテンションの焦点との間に自然な緊張関係が生じる。さらにモデルは、短いシーケンスが長いものよりも一般的に多く含まれる訓練データの分布から、そのアテンションパターンを発達させる。これは、コンテキスト全体にわたる依存関係について、モデルの経験が少なく、専用のパラメータも少ないことを意味する。

position encoding interpolation のような手法は、モデルがもともと訓練されたより小さなコンテキストに長いシーケンスを適合させることで、より長いシーケンスを扱えるようにするが、トークン位置の理解にいくらかの劣化を伴う。これらの要因は、急峻な崖ではなく、性能の勾配を生み出す。すなわちモデルは、より長いコンテキストでも高い能力を維持するが、より短いコンテキストでの性能と比べると、情報検索や長距離の推論において精度の低下を示す場合がある。

これらの現実は、高性能なエージェントを構築するには、思慮深い context engineering が不可欠であることを意味する。

## The anatomy of effective context（効果的なコンテキストの解剖学）

LLM が有限のアテンション予算によって制約されていることを踏まえると、優れた context engineering とは、ある望ましい結果の見込みを最大化する、可能な限り小さな高シグナルのトークン集合を見つけることを意味する。この実践を実装するのは口で言うほど容易ではないが、以下の節では、この指導原則がコンテキストのさまざまな構成要素にわたって実際に何を意味するのかを概説する。

システムプロンプトは極めて明確であるべきで、エージェントにとって適切な高度（altitude）でアイデアを提示する、単純で直接的な言葉を用いるべきである。適切な高度とは、二つのよくある失敗モードのあいだにある「ゴルディロックス・ゾーン」である。一方の極端では、エンジニアがエージェントの正確な振る舞いを引き出そうとして、複雑で壊れやすいロジックをプロンプトにハードコードする様子が見られる。このアプローチは脆さを生み、時間とともに保守の複雑さを増大させる。もう一方の極端では、エンジニアが曖昧で高レベルなガイダンスを与えてしまい、望ましい出力に対する具体的なシグナルを LLM に与えられなかったり、共有された文脈があると誤って想定してしまったりすることがある。最適な高度はバランスをとる。すなわち、振る舞いを効果的に導けるほど具体的でありながら、振る舞いを導く強力なヒューリスティクスをモデルに与えられるほど柔軟であるべきだ。

> スペクトラムの一方の端には壊れやすい if-else のハードコードされたプロンプトがあり、もう一方の端には過度に一般的だったり、共有された文脈があると誤って想定したりするプロンプトがある。

私たちは、プロンプトを明確に区別された複数のセクション（`<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` など）に整理し、XML タグ付けや Markdown ヘッダーのような技法を用いてこれらのセクションを区切ることを推奨する。ただし、モデルがより高性能になるにつれて、プロンプトの正確なフォーマットの重要性は低下しつつある可能性が高い。

システムプロンプトをどう構成するにせよ、期待される振る舞いを完全に概説する最小限の情報集合を目指すべきである。（最小限とは必ずしも短いことを意味しない点に注意。望ましい振る舞いを遵守させるために、依然として十分な情報をあらかじめエージェントに与える必要がある。）まず、利用可能な最良のモデルで最小限のプロンプトをテストし、自分のタスクでどう機能するかを確認することから始め、初期テストで見つかった失敗モードに基づいて明確な指示や例を追加して性能を改善していくのが最善である。

ツールはエージェントが環境と相互作用し、作業を進めながら新たな追加コンテキストを取り込むことを可能にする。ツールはエージェントとその情報／行動空間との間の契約を定義するため、トークン効率の良い情報を返すことと、効率的なエージェントの振る舞いを促すことの両面で、ツールが効率を促進することが極めて重要である。

Writing tools for AI agents – with AI agents において、私たちは LLM によく理解され、機能の重複が最小限であるツールの構築について論じた。よく設計されたコードベースの関数と同様に、ツールは自己完結的で、エラーに対して堅牢で、意図された用途に関して極めて明確であるべきだ。入力パラメータも同様に、説明的で曖昧さがなく、モデルが本来持つ強みを活かすものであるべきである。

私たちが目にする最もよくある失敗モードの一つは、あまりにも多くの機能を網羅していたり、どのツールを使うべきかについて曖昧な判断点を生んだりする、肥大化したツールセットである。人間のエンジニアが、ある状況でどのツールを使うべきか明確に断言できないのなら、AI エージェントにそれ以上を期待することはできない。後ほど論じるように、エージェントのために実用最小限のツールセットを取捨選択することは、長い対話にわたるコンテキストのより信頼性の高い保守と剪定にもつながる。

例を提示すること、すなわち few-shot prompting は、私たちが引き続き強く推奨する、よく知られたベストプラクティスである。しかしチームはしばしば、特定のタスクで LLM が従うべきあらゆるルールを明確にしようとして、エッジケースの一覧をプロンプトに詰め込みがちである。これは推奨しない。代わりに私たちは、エージェントの期待される振る舞いを効果的に描き出す、多様で正典的（canonical）な例の集合を取捨選択することを推奨する。LLM にとって、例は千語に値する「絵」なのである。

コンテキストのさまざまな構成要素（システムプロンプト、ツール、例、メッセージ履歴など）にわたる私たちの全体的な指針は、思慮深くあること、そしてコンテキストを情報量豊かでありながら引き締まったものに保つことである。それでは、実行時にコンテキストを動的に取得する話に入ろう。

## Context retrieval and agentic search（コンテキストの取得とエージェント的探索）

Building effective AI agents において、私たちは LLM ベースのワークフローとエージェントの違いを強調した。その記事を書いて以来、私たちはエージェントについてのシンプルな定義へと引き寄せられてきた。すなわち、ループ内でツールを自律的に使用する LLM、である。

顧客と協働するなかで、私たちはこの分野がこのシンプルなパラダイムへと収斂していくのを目にしてきた。基盤となるモデルがより高性能になるにつれて、エージェントの自律性のレベルもスケールしうる。より賢いモデルは、エージェントが微妙な問題空間を独力でナビゲートし、エラーから回復することを可能にする。

私たちは今、エンジニアがエージェントのためのコンテキスト設計をどう考えるかに変化が生じているのを目にしている。今日、多くの AI ネイティブなアプリケーションは、エージェントが推論する対象となる重要なコンテキストを浮かび上がらせるために、何らかの形の埋め込みベースの推論前検索（pre-inference time retrieval）を採用している。この分野がよりエージェント的なアプローチへと移行するにつれて、私たちはこれらの検索システムを「just in time（ジャストインタイム）」なコンテキスト戦略で補強するチームをますます多く目にしている。

関連するすべてのデータをあらかじめ前処理するのではなく、「just in time」アプローチで構築されたエージェントは、軽量な識別子（ファイルパス、保存されたクエリ、ウェブリンクなど）を保持し、これらの参照を用いて、ツールを介して実行時に動的にデータをコンテキストへ読み込む。Anthropic のエージェント的コーディングソリューションである Claude Code は、このアプローチを用いて大規模なデータベースに対する複雑なデータ分析を行う。モデルは、狙いを定めたクエリを書き、結果を保存し、`head` や `tail` のような Bash コマンドを活用することで、完全なデータオブジェクトをコンテキストに読み込むことなく大量のデータを分析できる。このアプローチは人間の認知を反映している。私たちは一般に情報のコーパス全体を暗記するのではなく、ファイルシステム、受信箱、ブックマークのような外部の整理・索引付けシステムを導入して、必要に応じて関連情報を取り出すのである。

ストレージ効率を超えて、これらの参照のメタデータは、明示的に与えられるものであれ直観的なものであれ、振る舞いを効率的に洗練する仕組みを提供する。ファイルシステム上で動作するエージェントにとって、tests フォルダ内にある `test_utils.py` という名前のファイルの存在は、`src/core_logic/` に置かれた同名のファイルとは異なる目的を暗示する。フォルダ階層、命名規則、タイムスタンプはいずれも、人間にもエージェントにも、情報をどのように、いつ活用すべきかを理解する助けとなる重要なシグナルを提供する。

エージェントに自律的にデータをナビゲートし取得させることは、progressive disclosure（漸進的開示）も可能にする。言い換えれば、エージェントが探索を通じて関連コンテキストを段階的に発見できるようにする。各相互作用が次の判断に資するコンテキストを生み出す。ファイルサイズは複雑さを示唆し、命名規則は目的を示唆し、タイムスタンプは関連性の代理指標となりうる。エージェントは理解を層ごとに積み上げ、ワーキングメモリには必要なものだけを保持しつつ、追加の永続化のためにノートテイキング戦略を活用できる。この自己管理されたコンテキストウィンドウは、網羅的だが無関係かもしれない情報の中で溺れるのではなく、関連する部分集合にエージェントの焦点を保たせる。

もちろんトレードオフは存在する。実行時の探索は、事前計算されたデータの取得よりも遅い。それだけでなく、LLM が自らの情報環境を効果的にナビゲートするための適切なツールとヒューリスティクスを持てるようにするには、明確な方針を持った思慮深いエンジニアリングが必要となる。適切なガイダンスがなければ、エージェントはツールを誤用したり、行き止まりを追いかけたり、重要な情報を識別できなかったりして、コンテキストを浪費しかねない。

ある種の状況では、最も効果的なエージェントはハイブリッド戦略を採用するかもしれない。すなわち、速度のために一部のデータをあらかじめ取得しつつ、その裁量でさらなる自律的探索を進めるのである。「適切な」自律性のレベルの決定境界は、タスクに依存する。Claude Code はこのハイブリッドモデルを採用するエージェントである。CLAUDE.md ファイルは素朴にあらかじめコンテキストに投入される一方で、glob や grep のようなプリミティブによって、その環境をナビゲートしジャストインタイムでファイルを取得することができ、古くなった索引や複雑な構文木の問題を効果的に回避する。

ハイブリッド戦略は、法務や財務の業務のように、内容の動的さがより少ないコンテキストにより適しているかもしれない。モデルの能力が向上するにつれて、エージェント設計は、人間による取捨選択を徐々に減らしつつ、知的なモデルに知的に振る舞わせる方向へと向かうだろう。この分野の進歩の速さを踏まえると、「機能する最も単純なことをせよ（do the simplest thing that works）」が、Claude の上にエージェントを構築するチームへの私たちの最良の助言であり続ける可能性が高い。

### Context engineering for long-horizon tasks（長期的タスクのための context engineering）

長期的（long-horizon）タスクでは、トークン数が LLM のコンテキストウィンドウを超える一連の行動にわたって、エージェントが一貫性、コンテキスト、目標指向の振る舞いを維持することが求められる。大規模なコードベースの移行や包括的なリサーチプロジェクトのように、数十分から数時間にわたる連続作業に及ぶタスクでは、エージェントはコンテキストウィンドウのサイズ制限を回避するための専門的な技法を必要とする。

より大きなコンテキストウィンドウを待つというのは、一見明白な戦術に思えるかもしれない。しかし、当面のあいだは、あらゆるサイズのコンテキストウィンドウが、少なくとも最も強力なエージェント性能が望まれる状況においては、コンテキスト汚染（context pollution）と情報の関連性の懸念にさらされる可能性が高い。エージェントが拡張された時間軸にわたって効果的に作業できるようにするために、私たちはこれらのコンテキスト汚染の制約に直接対処するいくつかの技法を開発してきた。すなわち、コンパクション（compaction）、構造化されたノートテイキング、そしてマルチエージェントアーキテクチャである。

#### Compaction（コンパクション）

コンパクションとは、コンテキストウィンドウの限界に近づきつつある会話を取り上げ、その内容を要約し、その要約とともに新しいコンテキストウィンドウを再初期化する実践である。コンパクションは通常、より良い長期的一貫性を促すための context engineering における最初のレバーとして機能する。その核心において、コンパクションはコンテキストウィンドウの内容を高い忠実度で蒸留し、エージェントが性能の劣化を最小限に抑えて作業を継続できるようにする。

たとえば Claude Code では、メッセージ履歴をモデルに渡して最も重要な詳細を要約・圧縮させることで、これを実装している。モデルは、冗長なツール出力やメッセージを破棄しつつ、アーキテクチャ上の決定、未解決のバグ、実装の詳細を保持する。エージェントはその後、この圧縮されたコンテキストに、直近でアクセスされた 5 つのファイルを加えて作業を継続できる。ユーザーは、コンテキストウィンドウの制限を気にすることなく継続性を得られる。

コンパクションの技芸は、何を残し何を捨てるかの選択にある。過度に積極的なコンパクションは、その重要性が後になって初めて明らかになるような、微妙だが重要なコンテキストの喪失を招きかねないからだ。コンパクションシステムを実装するエンジニアには、複雑なエージェントのトレースに対してプロンプトを注意深くチューニングすることを推奨する。まず再現率（recall）を最大化して、コンパクションプロンプトがトレースからあらゆる関連情報を捉えるようにし、次に、余分な内容を排除することで適合率（precision）を改善するよう反復する。

容易に取り除ける余分な内容の一例は、ツール呼び出しとその結果をクリアすることである。ツールがメッセージ履歴の奥深くで一度呼び出されたあと、なぜエージェントがその生の結果を再び見る必要があるだろうか。最も安全で最も軽いタッチのコンパクションの形態の一つがツール結果のクリアリングであり、これは Claude Developer Platform の機能として最近ローンチされた。

#### Structured note-taking（構造化されたノートテイキング）

構造化されたノートテイキング、すなわちエージェント的メモリ（agentic memory）とは、エージェントがコンテキストウィンドウの外部のメモリに永続化されるノートを定期的に書き込む技法である。これらのノートは、後の時点でコンテキストウィンドウに引き戻される。

この戦略は、最小限のオーバーヘッドで永続的なメモリを提供する。Claude Code が ToDo リストを作成したり、あなたのカスタムエージェントが NOTES.md ファイルを維持したりするように、このシンプルなパターンは、エージェントが複雑なタスクにわたって進捗を追跡し、さもなくば数十回のツール呼び出しのなかで失われてしまうような重要なコンテキストや依存関係を維持することを可能にする。

Claude playing Pokémon は、コーディング以外の領域でメモリがエージェントの能力をどのように変容させるかを実証している。このエージェントは、数千ものゲームステップにわたって正確な集計を維持し、「直近の 1,234 ステップのあいだ、私は Route 1 で自分のポケモンを訓練しており、Pikachu は目標である 10 レベルに向けて 8 レベル上昇した」といった目標を追跡する。メモリ構造について何のプロンプトを与えられずとも、探索した地域のマップを作り上げ、どの重要な実績を解除したかを覚え、どの攻撃が異なる相手に対して最も有効かを学ぶ助けとなる戦闘戦略の戦略的ノートを維持する。

コンテキストのリセット後、エージェントは自身のノートを読み、数時間に及ぶ訓練のシーケンスやダンジョン探索を継続する。要約ステップをまたぐこの一貫性が、すべての情報を LLM のコンテキストウィンドウだけに保持していたら不可能であろう長期的な戦略を可能にする。

Sonnet 4.5 のローンチの一環として、私たちは Claude Developer Platform 上でメモリツールをパブリックベータとしてリリースした。これは、ファイルベースのシステムを通じて、コンテキストウィンドウの外部に情報を保存し参照することをより容易にする。これによりエージェントは、時間をかけて知識ベースを構築し、セッションをまたいでプロジェクトの状態を維持し、すべてをコンテキストに保持しておくことなく過去の作業を参照できる。

#### Sub-agent architectures（サブエージェントアーキテクチャ）

サブエージェントアーキテクチャは、コンテキストの制約を回避するもう一つの方法を提供する。一つのエージェントがプロジェクト全体にわたって状態を維持しようとするのではなく、専門化されたサブエージェントが、クリーンなコンテキストウィンドウで焦点を絞ったタスクを処理できる。メインエージェントが高レベルの計画とともに調整を行い、その一方でサブエージェントが深い技術的作業を実行したり、ツールを使って関連情報を見つけたりする。各サブエージェントは、数万トークンあるいはそれ以上を使って広範に探索するかもしれないが、返すのは凝縮され蒸留された作業の要約（多くの場合 1,000〜2,000 トークン）だけである。

このアプローチは、関心の明確な分離を達成する。詳細な探索コンテキストはサブエージェント内に隔離されたまま保たれ、その一方でリードエージェントは結果の統合と分析に集中する。How we built our multi-agent research system で論じられたこのパターンは、複雑なリサーチタスクにおいて単一エージェントシステムに対する大幅な改善を示した。

これらのアプローチのあいだの選択は、タスクの特性に依存する。たとえば、

- コンパクションは、広範なやり取りを要するタスクの会話の流れを維持する。
- ノートテイキングは、明確なマイルストーンを伴う反復的な開発に優れる。
- マルチエージェントアーキテクチャは、並列探索が報われる複雑なリサーチや分析を扱う。

モデルが向上し続けても、拡張された相互作用にわたって一貫性を維持するという課題は、より効果的なエージェントを構築するうえで中心的であり続けるだろう。

## Conclusion（結論）

context engineering は、私たちが LLM を用いて構築する方法における根本的な転換を表している。モデルがより高性能になるにつれて、課題は完璧なプロンプトを練り上げることだけではなく、各ステップでモデルの限られたアテンション予算にどの情報を入れるかを思慮深く取捨選択することになる。長期的タスクのためにコンパクションを実装するにせよ、トークン効率の良いツールを設計するにせよ、エージェントがジャストインタイムで環境を探索できるようにするにせよ、指導原則は同じである。すなわち、望ましい結果の見込みを最大化する、最小の高シグナルなトークン集合を見つけることだ。

私たちが概説した技法は、モデルが向上するにつれて進化し続けるだろう。私たちはすでに、より賢いモデルがより指図的でない（less prescriptive）エンジニアリングを必要とし、エージェントがより高い自律性で動作できるようになりつつあることを目にしている。しかし、能力がスケールしても、コンテキストを貴重で有限な資源として扱うことは、信頼性が高く効果的なエージェントを構築するうえで中心的であり続けるだろう。

今すぐ Claude Developer Platform で context engineering を始めよう。また、当社の memory and context management cookbook を通じて、役立つヒントやベストプラクティスにアクセスしよう。
